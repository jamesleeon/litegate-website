Identity / Decision Service

LiteGate IDS Development Guide

IDS is not a heavy framework and it is not your backend service. It is a sidecar-style decision endpoint that returns JSON. LiteGate asks it how to authenticate the caller, where to route the request, which headers to inject, and what rate limit to apply.

AuthHow should LiteGate verify this caller?
RouteWhich physical shard or lane should receive traffic?
HeadWhich trusted headers should be injected downstream?
RateLimitWhich caller or tenant quota should be enforced?
ClientSigned request, app id, token, or session
LiteGate asks IDS for a decision
Backend ServerReceives real traffic plus trusted injected headers
Keep the boundary clear. Real business traffic does not enter IDS. IDS returns decisions; LiteGate executes them; backend services handle business requests.

1. Request and response contract

The contract is the core of IDS development: LiteGate sends a request with gateway credentials and a caller identifier; IDS returns a normalized decision envelope.

Request from LiteGate

GET /api/gate/credentials?client_id=app-c1&type=ids
X-Gateway-ID: litegate-prod
X-Gateway-Secret: shared-secret

X-Gateway-ID and X-Gateway-Secret let your IDS confirm that the query comes from a trusted gateway.

Response from IDS

{
  "code": 1,
  "message": "ok",
  "data": {
    "auth": { "type": "md5", "secret": "secret-c1", "ttl": "300" },
    "route": { "selector": { "sid": "c1" }, "meta": { "lane": "stable" } },
    "head": { "X-LiteGate-Identity-Ref": "ref_7f3a9c" },
    "ratelimit": { "qps": "100", "limit_key": "tenant-c1" }
  }
}

2. Minimal IDS

A working IDS can be a small HTTP handler that looks up the caller and emits the four-part JSON decision.

func credentials(w http.ResponseWriter, r *http.Request) {
    appID := r.URL.Query().Get("client_id")
    merchant, ok := merchants[appID]
    if !ok {
        json.NewEncoder(w).Encode(map[string]any{
            "code": 404001, "message": "appid disabled or not found",
        })
        return
    }
    json.NewEncoder(w).Encode(map[string]any{
        "code": 1,
        "data": map[string]any{
            "auth": map[string]string{"type": "md5", "secret": merchant.Secret, "ttl": "300"},
            "route": map[string]any{"selector": map[string]string{"sid": merchant.SID}},
            "head": map[string]string{"X-Tenant-ID": merchant.TenantID},
            "ratelimit": map[string]string{"qps": merchant.QPS, "limit_key": merchant.TenantID},
        },
    })
}
No special SDK is required. Replace the in-memory table with your merchant, tenant, app, or session store.

3. What the backend looks like

The backend is a separate program. It receives traffic after LiteGate has verified the caller, selected a shard, and injected trusted headers.

func handler(w http.ResponseWriter, r *http.Request) {
    tenant := r.Header.Get("X-Tenant-ID")
    verified := r.Header.Get("X-Lito-Verified")
    fmt.Fprintf(w, "tenant=%s verified=%s", tenant, verified)
}

For shard routing, register backend instances through discovery with tags such as sid=c1 and sid=c2. LiteGate can then match route.selector.sid against the actual instance metadata.

4. Four decision segments

auth

Defines how LiteGate verifies the caller locally. Common values are none, md5, md5-secure, and jwt.

route

selector is hard isolation and is mandatory for openapi, verify, and issue modes (missing selector triggers a fail-closed error). meta is a softer preference for lane, version, or canary metadata.

head

Headers in this section are trusted headers injected by the gateway. LiteGate removes untrusted client-supplied copies before injecting them.

ratelimit

Dynamic per-caller or per-tenant quota. This differs from a static route-level limit, which is shared by the entire route.

5. IDS modes

ModePurposeTypical use
openapi recommended firstExtract app id, fetch credentials, verify signature locally, inject routing and headers.Partner API, merchant API, signed requests.

verifyExtract token, read Redis decision, inject route and headers.Authenticated business APIs.
refresh / revokeMove or delete token mappings during refresh and logout.Session lifecycle.

6. MD5 signing in openapi mode

When auth.type=md5, the client signs the request. IDS only returns the secret; LiteGate verifies the signature locally.

sign = md5(method + "\n" + path + "\n" + body_md5 + "\n" + timestamp + "\n" + secret)

7. Physical shard routing

IDS can control the physical traffic target by returning route.selector.sid.

{
  "route": { "selector": { "sid": "c1" } }
}

If the backend discovery pool contains instances tagged sid=c1 and sid=c2, this caller is pinned to c1. This is the core mechanism for tenant-level physical isolation.

Do not use shard selectors with a single static upstream that has no metadata. Discovery-backed instances must carry the tags that IDS returns.

8. Wire IDS into LiteGate

ids:
  providers:
    openapi-main:
      mode: openapi
      url: "http://127.0.0.1:19091"
      path: /api/gate/credentials
      fail_policy: deny
      negative_ttl: 60
routes:
  - name: openapi-orders
    match:
      path_prefix: /openapi/orders
    action:
      type: proxy
      service_name: orders-service
      ids:
        enabled: true
        provider: openapi-main
        request_extractor: "{header._appid}"

9. Cache and denial policy

  • Positive cache: controlled by auth.ttl.
  • Negative cache: invalid app ids such as code=404001 are cached to reduce IDS pressure.
  • Fail policy: keep fail_policy: deny for authentication paths unless you intentionally accept fail-open behavior.

10. Field reference

FieldMeaning
auth.typeSigner or verifier type used by LiteGate.
auth.secretSecret used for local signature verification.
route.selectorHard isolation metadata, commonly sid (Mandatory for openapi/verify/issue modes).
route.metaSoft metadata used for lane, canary, or version preference.
headTrusted downstream request headers.
ratelimit.qpsDynamic quota for this caller or tenant.
ratelimit.limit_keyBucket key, usually tenant id or app id (fallback to key or client IP if empty).

11. Test checklist

  • Call IDS directly and confirm it returns code: 1 and a four-part data object.
  • Send a correctly signed request through LiteGate and confirm 200.
  • Send a bad signature and confirm 401 or 403.
  • Run two backend instances with different sid tags and verify that each app id lands on the intended shard.
  • Return a small dynamic ratelimit.qps and verify that excess requests return 429.

12. FAQ

Is IDS the backend server?

No. IDS is the side decision service. The backend server receives real traffic.

Does IDS need a database or Redis?

Not by protocol. Production IDS usually reads your app, tenant, merchant, or session store.

Why do I see "processor not found for the specified mode"?

The provider has no valid mode. Set mode: openapi, issue, verify, refresh, or revoke.

Why does a valid app id return no healthy upstream?

Your IDS returned a selector, but the upstream pool has no instance carrying the matching tag. Use discovery-backed instances with matching metadata.
