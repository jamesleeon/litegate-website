gRPC as a First-Class Citizen: one proto: grpc takes the whole chain

LiteGate treats gRPC as a native protocol, not "HTTP/2 we can sort of forward." Put proto: grpc on a route and you get: automatic h2c/grpcs split, automatic mTLS upgrade, trailer pass-through, real grpc.health.v1 health checks, gRPC-Web transcoding, server reflection, and grpc-timeout deadline propagation — all out of the box, no plugins.

1Enable with proto: grpc

Declare the upstream protocol as gRPC on a proxy action; the gateway forwards with gRPC semantics and preserves HTTP/2 trailers:

sites/grpc.yaml
domain: grpc.example.com
routes:
  - name: greeter
    match:
      path_prefix: /
    action:
      type: proxy
      service_name: greeter-svc   # or upstream: host:port
      proto: grpc                 # grpc for cleartext; grpcs for TLS backends

Service-discovery tags enable it just as easily (online on tag):

service meta
litegate.ingress.host   = grpc.example.com
litegate.upstream.proto = grpc

2h2c cleartext & TLS auto-split · mTLS auto-upgrade

gRPC backends come in two shapes: cleartext HTTP/2 (h2c) and TLS (grpcs). LiteGate tells them apart automatically:

  • proto: grpc → cleartext h2c: forwarded through a dedicated http2.Transport (AllowHTTP + raw-TCP DialTLS), never mistakenly using TLS.
  • proto: grpcs → TLS gRPC.
  • When the route has mTLS (Litemesh) enabled, grpc is automatically upgraded to grpcs — no config change needed.
Trailer pass-through

The gateway natively forwards gRPC trailers: grpc-status / grpc-message / grpc-status-details-bin, and preserves TE: trailers. This is a prerequisite for gRPC to work correctly — many "just forward HTTP/2" approaches drop trailers and break clients.

3Native Health Check (grpc.health.v1)

For gRPC backends, a TCP probe only confirms "the port is open," not that the gRPC service is actually Serving. LiteGate's active health check can call the standard grpc.health.v1.Health/Check directly (with both h2c and grpcs clients), recognizing SERVING / NOT_SERVING so load-balancer eviction is accurate.

Real probing, not fake

A gRPC process whose "port is alive but service isn't ready" is correctly marked unhealthy and drained, instead of having requests pushed into it.

4gRPC-Web Transcoding

Browsers can't speak standard gRPC (HTTP/2 binary framing) — only gRPC-Web. LiteGate transcodes application/grpc-web and application/grpc-web-text (base64 framing) into standard gRPC at the edge and converts responses back — so your frontend can call gRPC backends directly without standing up a separate Envoy.

5Server Reflection

gRPC server reflection auto-routing is supported, so tools like grpcurl / Postman can discover backend services and methods through the gateway — a big boost for debugging and integration, with no manual proto import.

6Deadline Propagation & Unified Error Mapping

gRPC carries the call deadline in a grpc-timeout header. The gateway parses and propagates the remaining deadline upstream; when it expires, the client gets a proper grpc-status: 4 (DEADLINE_EXCEEDED), not a bare HTTP 504.

Every error path (upstream unreachable, dial failure, auth rejection, etc.) is uniformly mapped to grpc-status, so clients receive a standard gRPC status code rather than a half-baked HTTP error:
