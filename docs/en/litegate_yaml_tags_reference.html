Scenario 1: Frontend/Backend Split & Microservice Co-location (SPA static hosting + dynamic API proxy — hybrid routing)

Hybrid merge
This showcases one of LiteGate's most powerful core capabilities: Hybrid Merge. In this typical production setup, the static SPA frontend (index.html, etc.) is hosted directly from the local filesystem via static YAML (using the type: serve action), while the dynamic routing rules for backend API microservices (e.g. order-service) are driven entirely by self-registration tags that service instances report at startup. The gateway merges both into a single unified routing tree in memory — preserving edge security and control while giving development teams highly agile self-service publishing.

Static hosting gateway YAML (frontend-only.yaml)

domain: app.example.com
port: 80              # 💡 Tip: global default listening port for the static site
force_https: true

routes:
  # 1. Static hosting route (static-YAML-only; instances must not declare such physical paths)
  - name: frontend-spa
    priority: 10
    match:
      path_prefix: /
    action:
      type: serve
      root: /var/www/dist
      index: index.html
      spa: true
      # 💡 Optional: enable Serve KV automated synchronization mode
      # kv_mode: true
      # kv_provider: litemesh
      # kv_key: litegate/config/front/main
      # keep_versions: 3

  # 💡 Tip: dynamic API proxy routes do NOT need to be declared here!
  # Backend services report their own IP, port, and route Tags to the registry at startup.
  # The gateway auto-detects them and performs a Hybrid Merge with this top-level domain node, enabling hot-pluggable service routes.

Service self-registration tags (order-service node Tags)

# ── 1. Ingress & matching ──
# When the gateway receives a request for app.example.com (triggers Hybrid Merge and in-memory lookup)
litegate.ingress.host=app.example.com
# Declare prefix matching. Requests are dispatched to this instance group only when the path starts with /api
litegate.ingress.path_prefix=/api
# Raise priority to 100 to ensure high-priority matching and avoid being shadowed by the default SPA "/" route
litegate.router.priority=100

# ── 2. Path rewriting ──
# Strip prefix. Removes /api before forwarding so the service receives /orders instead of /api/orders
litegate.router.strip_prefix=/api

# ── 3. Load balancing ──
# Instance weight. This instance carries 80% of the traffic during load balancing
litegate.upstream.weight=80

💡 Hybrid Merge: a core architectural feature of LiteGate. Administrators only maintain infrastructure nodes (domain binding, SSL certs) in global static YAML, while service developers declare API routing characteristics directly in their own node Tags. At runtime the gateway merges both into one unified routing tree — preserving security while eliminating the cumbersome centralized-config bottleneck of traditional gateways.

Scenario 2: Dynamic API Gateway (multi-instance proxy + local rate limiting & circuit breaking)

API Gateway
This is the most common deployment of LiteGate as a pure microservice API gateway. Here the gateway hosts no local static resources and focuses entirely on high-concurrency L7 reverse-proxy traffic. By combining QPS rate limiting (with an adaptive algorithm option), request-timeout control and self-healing retries, plus per-instance circuit-breaker protection, it builds a robust distributed defense layer for downstream services, isolating and intercepting cascading failures before they bring the system down.

Gateway static YAML (pure-api-proxy.yaml)

domain: api.example.com
port: 443

routes:
  - name: api-gateway-route
    priority: 50
    match:
      path_prefix: /v1
    action:
      type: proxy
      service_name: payment-service
      timeout: 5             # 💡 YAML side uses plain integer seconds, no unit
      retry: 3
      rate_limit: 150
      rate_limit_policy: adaptive
      circuit_breaker:
        enabled: true
        timeout: 30          # 💡 plain integer seconds
        max_requests: 10
        trip_threshold: 15

Service self-registration tags (payment-service node Tags)

# ── 1. Ingress & matching ──
litegate.ingress.host=api.example.com
litegate.ingress.path_prefix=/v1
litegate.router.priority=50

# ── 2. L5 Service network timeout & retry ──
# Max time to wait for the upstream response
litegate.service.timeout=5s
# Max retry attempts on transient network failures or 5xx errors
litegate.service.retry=3

# ── 3. L5 Service circuit-breaker config ──
# ⚠️ Tag side uses flat breaker_*, and timeout is plain integer seconds (no s suffix)
litegate.service.breaker=true
litegate.service.breaker_timeout=30
litegate.service.breaker_max_requests=10
litegate.service.breaker_trip_threshold=15

# ── 4. L4 Middleware rate limiting ──
litegate.middleware.ratelimit=150
# 💡 Note: rate_limit_policy (adaptive) is static-YAML-only; dynamic tags don't support it yet and treat the limit as a hard QPS cap

💡 High-availability gateway protection: this scenario shows the gateway as a core shield. By simply declaring litegate.service.breaker=true and rate-limit rules in dynamic tags, a microservice gains elastic distributed rate limiting and automatic circuit breaking without touching any client- or server-side SDK code.

Scenario 3: Business-level SSO & Auth Context Injection (IDS Session)

Unified identity interception
For complex business systems (B2B or B2C) that need high-security session verification and Claims context injection. Here the gateway mounts the IDS unified identity & security engine, automatically extracting session credentials from inbound requests and validating them against a shared Redis cache. The gateway not only provides asynchronous sliding-window session renewal, but also — after a successful check — decrypts the user principal (Claims) and injects them as standard X-Claim-* request headers into the upstream request, freeing backend services from repetitive identity-verification code.

Static gateway config (internal-session-auth.yaml)

domain: app.example.com
port: 443

routes:
  - name: internal-auth-api
    priority: 200
    match:
      path_prefix: /api/v1
    action:
      type: proxy
      service_name: biz-service
      ids:
        enabled: true
        mode: verify          # 💡 Session verification runs in the verify processor; ids_session is the provider type, not a mode
        provider: saas-auth
        extractor: "{header.Authorization}"
        session_ttl: 7200
        renew_threshold: 3600
        inject_headers: true
        inject_claims:
          X-User-Id: sub
          X-User-Email: email

Service self-registration tags (biz-service node Tags)

# ── 1. Ingress & matching ──
litegate.ingress.host=app.example.com
litegate.ingress.path_prefix=/api/v1
litegate.router.priority=200

# ── 2. Middleware interception & provider binding ──
# Enable the unified IDS engine and set the verification mode to verify (session verification)
litegate.middleware.ids.enabled=true
litegate.middleware.ids.mode=verify

# ── 3. Unified IDS engine inline parameter configuration ──
# Extract the session credential from the Authorization header (auto-detects Bearer / UUID format)
litegate.middleware.ids.extractor={header.Authorization}
# Hard session lifetime (seconds)
litegate.middleware.ids.session_ttl=7200
# Trigger async sliding renewal when the session has less than 3600 seconds left, preventing abrupt logout
litegate.middleware.ids.renew_threshold=3600

# ── 4. Identity context injection ──
# 💡 Note: inject_headers / inject_claims are static-YAML-only; dynamic tags don't support them.
# After a successful check, the gateway decrypts Claims and injects X-Claim-* request headers uniformly.

💡 Self-healing session defense: LiteGate performs session validation and on-demand async renewal at the edge. Teams no longer need to wire a Redis client and identity-filter logic into every microservice — simplifying code and avoiding the consistency disasters caused by a single service mishandling session renewal.

Scenario 4: Open-Platform MD5 Offline Signing & Dynamic Credential Fetch

Open platform core
For external merchants or open-API scenarios. External clients carry a SecretKey and perform offline MD5 signature verification. The gateway dynamically resolves the AppId, fetches/aligns keys for high-concurrency signature verification behind an L1 (LRU) / L2 (Redis) two-tier cache, and establishes a 404 negative-cache mechanism to fully de-noise and shed high-frequency malicious probing from the backend identity service (IDS). To secure the origin path, the gateway automatically attaches client_id and client_secret credentials when calling the unified identity center.

Static gateway config (open-api-md5.yaml)

domain: openapi.ai.com
port: 443

routes:
  - name: open-orders
    priority: 150
    match:
      path_prefix: /openapi/v1/orders
    action:
      type: proxy
      service_name: openapi-backend
      ids:
        enabled: true
        mode: openapi
        provider: dynamic-openapi
        extractor: "{header._appid}"
        # Mount the remote dynamic credential-fetch engine
        discovery_service: identityservice
        path: /api/gate/credentials
        client_id: litegate
        client_secret: secret
        negative_ttl: 60
        fail_policy: deny

Service self-registration tags (openapi-backend node Tags)

# ── 1. Dynamic business route & unified IDS engine registration ──
# Declare the business API route's matching path prefix
litegate.http.routers.open-orders.path=/openapi/v1/orders
# Enable the unified IDS engine in openapi (offline MD5 signature) mode
litegate.http.routers.open-orders.ids.enabled=true
litegate.http.routers.open-orders.ids.mode=openapi
# Bind the associated dynamic IDS provider
litegate.http.routers.open-orders.ids.provider=dynamic-openapi

# ── 2. Unified IDS engine inline parameter configuration ──
# Extract the AppId variable from the request header
litegate.http.routers.open-orders.ids.extractor={header._appid}
# Declare the credential provider (the gateway dynamically dispatches the backend identityservice for origin fetch)
litegate.http.routers.open-orders.ids.discovery_service=identityservice
# Mount the custom credential-verification endpoint path
litegate.http.routers.open-orders.ids.path=/api/gate/credentials
# Bind the client credentials for the identity center (ClientID / ClientSecret)
litegate.http.routers.open-orders.ids.client_id=litegate
litegate.http.routers.open-orders.ids.client_secret=secret
# 404 negative-cache block duration for invalid AppId (seconds)
litegate.http.routers.open-orders.ids.negative_ttl=60

💡 Routing-elevation protection: after signature verification passes, the gateway securely extracts sid internally and promotes it to the physical routing key KeyRouteSelector["sid"], working with Litemesh for targeted shard routing while fully blocking sid from being passed downstream — keeping the edge secure.

Scenario 5: Single-node Local Static Reverse Proxy (no service discovery)

Single-node static (legacy)
The simplest single-node, micro, or traditional deployment. In this mode the gateway depends on no dynamic service discovery (no Consul or Litemesh) and acts purely as a front proxy, forwarding all traffic to a hard-coded local or internal IP and port (e.g. 127.0.0.1:8080).

Gateway static YAML (local-static-proxy.yaml)

domain: api.example.com
port: 80

routes:
  - name: local-api-route
    priority: 100
    match:
      path_prefix: /v1
    action:
      type: proxy
      # Force the upstream type to a static physical address list
      upstream_type: static
      # Hard-code the local microservice physical listening port
      upstream:
        - 127.0.0.1:8080

Service self-registration tags

💡 Pure static orchestration: no dynamic tags
Because this scenario never goes through any registry, the gateway relies solely on the static YAML upstream for local dispatch. Instances neither need nor are able to report any dynamic tags — upholding a clean static/dynamic boundary.

💡 Registry-free minimalism: ideal for micro workloads or single-node Docker-Compose rapid deployment, preserving the familiar mental model of a traditional reverse proxy (like a minimal Nginx setup).

Scenario 6: Multi-instance Static Load-Balancing Cluster

Internal static cluster
For traditional internal clusters with fixed physical machines or fixed container IPs. Still without service discovery, the gateway lists multiple fixed upstream IPs directly in static YAML and drives multi-node load balancing at L7 (round-robin, IP hash), backed by network-level timeout and retry for high availability.

Gateway static YAML (multi-static-lb.yaml)

domain: app.example.com
port: 80

routes:
  - name: app-lb-route
    priority: 100
    match:
      path_prefix: /
    action:
      type: proxy
      upstream_type: static
      # Round-robin load balancing (supports rr / ip_hash)
      lb_policy: rr
      # Bind multiple backend static IPs and ports
      upstream:
        - 10.0.0.21:8080
        - 10.0.0.22:8080
        - 10.0.0.23:8080

Service self-registration tags & LB notes

⚖️ Multi-node static high availability:
In static LB mode, the gateway's internal dispatcher establishes active TCP health probes to each statically configured upstream instance. Once a node is detected offline, the gateway removes it from the healthy routing set in memory and seamlessly retries and dispatches the request to other healthy backends.
💡 Pure static orchestration: no dynamic tags
Because this scenario never goes through any registry, the gateway relies solely on the static YAML upstream for local dispatch. Instances neither need nor are able to report any dynamic tags.

💡 Registry-free cluster architecture: ideal for traditional enterprise intranets, isolated private networks, or fixed VM environments where deploying Consul-style discovery is inconvenient — high availability with minimal effort.

Scenario 7: WebDAV Static Hosting + Built-in Simple SSO Auth

Protected static storage
For team intranet file sharing or personal private storage. LiteGate provides out-of-the-box static WebDAV file hosting, with a visual web browsing UI and read/write restrictions. To secure files, mount the built-in Simple SSO auth component directly on the route. No complex OIDC service certificates required — just 4 lines protect your private filesystem, with a fully customizable local login template.

Gateway static YAML (webdav-sso.yaml)

domain: storage.example.com
port: 443
force_https: true

routes:
  - name: secure-webdav
    priority: 100
    match:
      path_prefix: /dav
    action:
      type: webdav
      root: /data/share     # Local physical storage path
      web_ui: true          # Enable the graphical web UI
      read_only: false      # Allow upload, delete, and modify

      # Mount the built-in Simple SSO defense
      auth:
        type: sso
        sso_enabled: true
        secret: "please_change_to_a_random_secret_at_least_32_bytes_long" # Session cookie signing key (production must use a random key of at least 32 bytes)
        template: /var/litegate/login.html # Custom local login page
        users:
          admin: "8c6976e5b5410415bde908bd4dee15dfb167a9c873fc4bb8a81f6f2ab448a918" # ciphertext of admin123

Dynamic tags & CLI helper notes

🛠️ Password-hash generator CLI:
To avoid leaking config and storing plaintext, LiteGate ships a strong salted password hash. Never write plaintext under users. In a terminal run: litegate --hash "your_password" The console prints a strong SHA256 / Bcrypt ciphertext to paste straight into the config file.
💡 Static-only protection note:
WebDAV's root, web_ui, and read_only are site-level physical-storage orchestration. By LiteGate's product boundary, dynamic tags neither provide nor are allowed to declare such local mount paths, eliminating filesystem-state inconsistency across multi-node container deployments at the source.

💡 The leverage of built-in SSO: instead of configuring an external OIDC identity center, LiteGate's built-in SSO needs only a few lines to deliver enterprise-grade interception and cookie sliding renewal — extremely portable for team collaboration and intranet storage projects.

Scenario 8: Legacy Domain 301 Migration + Legacy Root-Path 302 Redirect Fix

Legacy self-healing & migration
In real production it's common to perform seamless old-to-new domain migration (e.g. moving sales.old-example.com to a.example.com). At the same time, because the legacy system's routing was poorly written, users hitting the root path / often can't open the page, so you need to force a 302 redirect to the sub-app /sys entry at the gateway ingress while proxying other traffic unchanged. LiteGate offers readable redirect directives with template-variable substitution, sparing you cryptic Nginx regex rewrites.

File 1: New domain config (a.example.yaml — self-healing proxy)

domain: a.example.com
port: 80              # 💡 Optional. Defaults to 80/443 if omitted
force_https: true

routes:
  # ── 1. Fix the legacy broken root route: exact-match root, 302 to the sub-app ──
  - name: root-to-sys-redirect
    priority: 100     # 💡 High priority to precisely intercept before the fuzzy "/" match
    match:
      path: /         # Exact match for "/"
    action:
      type: redirect
      location: /sys  # 302 force-redirect to the real sub-app entry
      status: 302

  # ── 2. Reverse-proxy all other sub-path traffic to the legacy machine ──
  - name: legacy-system-proxy
    priority: 10      # Lower priority
    match:
      path_prefix: /  # LPM prefix match for all other traffic (e.g. /sys/login)
    action:
      type: proxy
      upstream_type: static
      upstream:
        - "http://10.0.0.10:8080"

File 2: Legacy domain redirect (sales.old-domain.yaml — smooth migration)

domain: sales.old-example.com
port: 80

routes:
  # ── 3. 301 permanent redirect to the new domain, preserving path & query via a template var ──
  - name: global-domain-migration
    match:
      path_prefix: /
    action:
      type: redirect
      # 💡 Uses the ReplaceVars template variable {uri} to pass through path and query unchanged!
      location: "https://a.example.com{uri}"
      status: 301

Legacy self-healing & smooth migration notes

🔄 301 seamless domain migration:
By using the built-in variable {uri} in Location (e.g. https://a.example.com{uri}), the gateway automatically captures and reattaches the original path and query string. For example, visiting http://sales.old-example.com/sys/login?user=admin permanently redirects (301) to https://a.example.com/sys/login?user=admin, protecting old bookmarks and remaining SEO-friendly.
🛠️ Zero-intrusion legacy root (LPM) self-healing:
Because the legacy app routing is poorly written, hitting / directly errors out or shows a blank page. LiteGate declares path: / with priority: 100 high priority to redirect root traffic to /sys first, while the rest (e.g. /sys/login) — having a longer prefix — is captured by the second path_prefix: / route and still proxied to the physical machine 10.0.0.10:8080, avoiding the huge cost and risk of refactoring the business app.

💡 No more Nginx config headaches: in Nginx you'd configure multiple Server blocks and write cryptic Rewrite regex for 301 migration and 302 fixes (error-prone and hard to maintain). LiteGate's YAML is clear and intuitive, and all redirect policies hot-reload and take effect instantly with no restart.
