Tag & go: zero-config from registration to HTTPS

In the LiteGate ecosystem, a microservice only has to "register with tags" to complete the full loop — from DNS resolution and route generation to HTTPS certificate issuance. Whether tags come from Docker labels, Consul tags, Litemesh/LiteDeploy meta, or a Discovery agent, the gateway generates entries from the same set of litegate.* tags.

1Core concept: two tag layers, distinct roles

When registering a service you place tags in two different places. One layer defines "how to identify and route to me"; the other declares "which shard / tenant I belong to".

Layer A · Service-definition tags (on Service Meta)
Tell the gateway how to identify me, how to authenticate, how to route to me. Set once, applies to all instances of the service. The gateway reads these and generates routes.

litegate.ingress.hostlitegate.ingress.path_prefixlitegate.middleware.authlitegate.router.sidlitegate.service.timeout

↓ gateway reads them and generates routes
Layer B · Instance-filter tags (on each instance's Meta)
Tell the gateway which shard / tenant / cluster I belong to. Each instance tags its own group, used for runtime RouteSelector filtering — these generate no routes.

sid=cluster-01env=prodtenant=corp_abc

The key distinction

Layer A's litegate.router.sid = cluster-01 means "what I'm looking for"; Layer B's sid = cluster-01 means "what I am". When the service definition sets router.sid=cluster-01, the gateway routes only to instances with sid=cluster-01, skipping other shards entirely. Both tag kinds coexist in one Meta object and are processed separately. See Backend Tag System.

2Five discovery sources, one tag set

There are five backend discovery sources: litemesh, litedeploy, docker, consul and discovery. Once inside LiteGate they are all normalized to the same ServiceEndpoint, then driven by the same litegate.* tags for routing, upstream pools, auth, IDS, mTLS and observability.

About static / external / hybrid

These are not business discovery sources: static is a fixed upstream (no discovery at all); external and hybrid are internal upstream_type names — the discovery source maps to external at runtime, and Consul + Discovery together map to hybrid.

What "discovery" precisely means

provider: discovery (mapping to upstream_type: external) specifically refers to LiteGate connecting over HTTP blocking query to a camodns_discovery-style aggregation layer compatible with /v1/catalog/services and /v1/health/service/:service_name, unifying heterogeneous registries into a service catalog LiteGate understands. It is not a sixth business source.

3Magic Ingress: Tag & Forget

Traditionally, putting a service online means six steps: deploy, configure routes, buy a domain, set public DNS, request a certificate, set internal DNS. With Magic Ingress you do one thing: deploy a tagged instance, e.g. litegate.ingress.host=order.example.com — the rest is automatic.

The only manual step

Register with tags

When starting via the Litemesh SDK (or Sidecar / Docker label / Consul registration), attach the magic tags such as litegate.ingress.host.

Discovery

CatalogLoader watches in real time

Detects the new service instantly, parses the magic tags, builds a virtual SiteConfig in memory on the fly, and updates the Router to take over that domain's traffic immediately.

Public reachability

DDNS auto-resolution

Detects the gateway's public egress IP and calls your DNS provider API (Aliyun / Cloudflare, etc.) to point the domain's A record at the gateway's public IP.

Certificate

AutoCert auto-issuance

Checks Litemesh KV for an existing valid certificate; if none, acquires a cluster distributed lock, requests one from Let's Encrypt, saves it to cluster KV, and all online nodes sync and load it instantly.

Internal reachability

Split-horizon DNS

Triggers a dns_update webhook to update internal DNS, pointing the domain at the gateway's internal IP. Internal users use the same domain and same valid certificate.

Net result

Zero config maintenance (no nginx.conf / gateway.yaml), zero ops (no manual resolution, no buying/uploading certs, no renewals), and a consistent internal/external experience (same domain, same HTTPS). Certificate details in Certificate Lifecycle.

4Register = online: expected timeline

Taking a payment microservice registering with a litegate.ingress.host tag as an example, the real timeline from SDK call to HTTPS fully ready:

TimeEventState
T + 0sService calls the SDK and registers🚀 start
T + 0.5sLiteGate perceives via SSE, builds the in-memory route🌐 internal HTTP reachable
T + 1sInternal DNS (CamoDNS) resolution takes effect📡 domain ready
T + 60sCertificate issued and broadcast across the cluster🔒 HTTPS fully ready
Why this architecture is solid

Decoupled: app developers don't need to know gateway config, only their own domain. Highly available: if a LiteGate node dies, internal DNS fails over to another via heartbeat. Zero-touch: certificates, resolution and routing are fully automatic — truly "Architecture as Operations".

5Steering & priority

When physical steering (token_steering) is enabled, the rich metadata decoded from a token can change the routing target at runtime, overriding the static shard tagged at registration.

priority
RouteSelector returned dynamically by SaaS / Session   ← highest priority (overridden by identity at runtime)
    ↓ if absent, falls back to
litegate.router.sid / router.* static tags             ← default shard set at registration

This means one set of gateway route rules can support elastic, per-second multi-tenant physical isolation: login state decides which cluster to route to, with no need to hardcode a static sid at registration. Identity decisions are detailed in the IDS Development Guide.

Mind the distinction

Above, litegate.* prefixed keys are service-definition tags (Layer A); the trailing sid / cluster are instance-side tags (Layer B). Both coexist in one Meta and are processed separately. For hands-on Docker provider config, see the Docker Service Discovery Deployment Guide.
