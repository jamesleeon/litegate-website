Let users forget certificate files by making HTTPS a gateway lifecycle capability

LiteGate does not bypass certificate rules. It removes manual certificate file work. Users declare domains, DNS providers, or authorization policies; LiteGate handles lookup, issuance, storage, renewal, and hot loading at runtime.

Automatic HTTPSOn-Demand TLSShared storageACME

1Why this deserves its own document

With a conventional reverse proxy, certificate management is usually an external step: obtain files with certbot, acme.sh, or an ops script, copy .pem/.key to the right directory, then reload the proxy.

LiteGate pulls that work into the entry layer. The TLS handshake looks up certificates by SNI. Missing certificates can trigger authorized on-demand issuance. Results are saved to authoritative storage, mirrored to local disk, renewed periodically, and reloaded without a manual proxy reload script.

Precise positioning

The promise is not “users do not need domain ownership.” The promise is “users do not manage certificate files.” DNS reachability, DNS API credentials, HTTP-01 ports, and SaaS authorization remain part of the security boundary.

2Three certificate paths

Load existing certificates

TLSManager scans the certificate directory, reads .pem/.crt plus .key, and indexes certificates by CommonName and SAN. Wildcard certificates are supported.

Ensure certificates at startup

When static dns_providers are configured, AutoCertManager ensures certificates exist, SANs match, chains verify, and renewal happens before expiry.

Issue on first access

When on_demand is enabled and no SNI certificate is found, LiteGate authorizes the domain first, then obtains a certificate through ACME.

3What happens during one HTTPS handshake

  1. Read SNI
    The client starts TLS and LiteGate reads ServerName from ClientHello.
  2. Normalize the host
    The host is lowercased, IDNA-normalized, and rejected if it is not a valid hostname.
  3. Find an in-memory cert
    LiteGate tries exact match first, then one-level wildcard matching such as *.example.com.
  4. Fallback to on-demand
    If nothing matches, the on-demand handler can authorize, obtain, save, and cache a new certificate.
flow
ClientHello(SNI)
  -> TLSManager.GetCertificate
  -> exact / wildcard memory match
  -> OnDemandHandler
  -> ACME obtain
  -> storage + local disk
  -> memory cache
  -> handshake continues

4How ACME validation is handled

HTTP-01

LiteGate intercepts /.well-known/acme-challenge/ before normal routing and returns the ACME token directly. Public port 80 must reach LiteGate for this path.

DNS-01

With a DNS provider configured, LiteGate can write validation records through the DNS API. This is the right path for wildcard certificates and environments where port 80 is closed.

If DNS update is enabled, LiteGate can also synchronize records before issuance, avoiding the gap where a service is registered but DNS does not yet point to the gateway.

5Multi-node deployments do not each invent their own certificates

Certificates can live on local disk, or in Consul or Litemesh as authoritative storage. After the authoritative write succeeds, LiteGate mirrors the certificate to local disk so TLSManager can load it quickly.

CapabilityEffect
Distributed lockWhen several LiteGate nodes see the same new domain, only one node obtains from the CA while others wait and read the result.
Pending stateLitemesh storage can mark domains as pending, reducing duplicate issuance during lock contention or split-brain windows.
Deterministic jitterIssuance and renewal add delay jitter to avoid a cluster hitting the CA or DNS provider at the same instant.
Local mirrorShared storage provides consistency; local files provide fast loading and watcher-based reloads.

6Renewal and hot loading

LiteGate periodically checks static certificates and dynamic domain certificates. It verifies the certificate chain, expiry window, expected SANs, and whether on-demand domains are still authorized.

After issuance or renewal, the certificate is saved to authoritative storage and synchronized to the local certificate directory. LiteGate then reloads TLS certificates. External certificate file changes are also detected through a debounced directory watcher.

What users stop doing manually

No PEM/KEY copying, no reload script, no repeated issuance on every node, and no separate certificate expiry patrol for the gateway layer.

7Security boundaries stay explicit

Domain authorization

On-demand domains must come from a defined site, Litemesh KV, or a SaaS Ask URL. Global catch-all sites do not authorize arbitrary domains.

Ask URL

Ask uses HTTPS by default. Internal HTTP requires explicit allow_insecure_ask; private CAs can be pinned with ask_ca.

Account key

storage_secret can encrypt the ACME account key. It is recommended for shared storage and must be at least 32 bytes.

9Minimal user-facing configuration

Fixed public domain

auto_cert:
  enabled: true
  email: ops@example.com
  renew_days: 30
  dns_providers:
    - name: cloudflare
      enabled: true
      domains: ["app.example.com"]
      provider_config:
        api_token: "${CF_API_TOKEN}"

SaaS on-demand domain

auto_cert:
  enabled: true
  email: ops@example.com
  on_demand:
    enabled: true
    ask: "https://control.example.com/tls/ask"
    saas_template_site: "tenant-template"
  storage_secret: "${AUTOCERT_STORAGE_SECRET}"
Production checklist

DNS API credentials, ports 80/443, CA rate limits, Ask service availability, shared-storage permissions, and storage_secret handling still belong in the release checklist.
