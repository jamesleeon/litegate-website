Unified Authentication: bring both "human" and "machine" traffic into the gateway

LiteGate pushes authentication down to the gateway edge so backend services don't each reimplement login and token validation. It covers both interactive human traffic (OIDC/OAuth2 login, SSO) and machine / API traffic (JWT, API-Key + quota), and can inject the verified identity into downstream request headers.

Human vs machine

Rule of thumb: a real person in a browser → OIDC; a program/service calling an API → JWT or API-Key. All three coexist in one gateway, bound per route.

2OIDC / OAuth2 (human login)

For interactive login: unauthenticated requests are redirected to the IdP, and a session is established on callback. Natively supports token introspection, multiple providers, and a custom login page template.

sites/app.yaml
routes:
  - name: admin
    match:
      path_prefix: /admin
    action:
      type: proxy
      service_name: admin-svc
      auth:
        type: sso
        secret: "your-long-sso-cookie-signing-secret"
        template: "./templates/auth/login.html"   # optional custom login page

For identity-driven decisions and tenant steering, see the IDS Guide.

3JWT validation (machine / API traffic)

For non-interactive service-to-service calls: validate the incoming Bearer JWT. The gateway verifies the signature against JWKS, checks issuer / audience, and projects claims into downstream request headers for the backend.

middleware
middlewares:
  jwt-auth:
    type: jwt_auth
    config:
      jwks_url: "https://idp.example.com/.well-known/jwks.json"
      issuer:   "https://idp.example.com/"
      audience: "api.example.com"
      inject_claims: "sub:X-Auth-Subject,email:X-Auth-Email"
Secure by default: algorithm pinning

JWT validation explicitly pins RS/ES asymmetric algorithms (`WithValidMethods`), eliminating the classic RS256→HS256 algorithm-confusion attack and `alg: none`.

4API Key + per-key quota

For public-facing APIs: validate the API Key and apply a separate QPS quota per key. Keys can be extracted from header / query / Bearer; quotas use a cluster-shared Redis token bucket (falling back to bounded local limiting if Redis is down), and keys are masked in logs.

middleware
middlewares:
  api-key:
    type: api_key
    config:
      extract_from: "header.X-API-Key"   # or query.api_key / bearer
      keys: "key_live_aaa,key_live_bbb"
      quotas: "key_live_aaa:50,key_live_bbb:10"   # per-key QPS
      default_quota: "5"

5Forward auth (remote authorization)

If you already have an in-house auth service, the gateway can subrequest the authorization decision to it (like Nginx auth_request / Caddy forward_auth): the external endpoint returns 2xx to allow, 401/403 to deny, and can return identity headers. Great for reusing existing RBAC / session centers.

6Identity injection (claims → header)

Whether identity comes from OIDC, JWT, or forward auth, the gateway can inject the parsed identity (user ID, email, tenant, roles, etc.) into downstream request headers. Backends just read the headers — no token parsing of their own. This is the key to truly pushing auth down to the gateway, and a common pattern for multi-tenant SaaS ingress.

Anti-spoofing

Before injecting, the gateway strips any client-forged authoritative headers of the same name, so the identity headers a backend sees can only come from the gateway itself.

7mTLS / SPIFFE & IDS

Beyond the HTTP-layer auth above, LiteGate also offers transport-layer and intrusion-detection identity:

  • mTLS / SPIFFE: mutual certificate authentication with SPIFFE ID verification for zero-trust meshes. See the Litemesh mTLS guide.
  • IDS intrusion-detection chain: a pluggable request-signing / identity-verification engine — a LiteGate differentiator. See the IDS Guide.
