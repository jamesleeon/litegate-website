Global config.yaml Reference

The global config.yaml configures gateway-level capabilities: listener ports, TLS and auto-certs, service discovery, caching, the Dashboard, logging and metrics. It is optional — without it the gateway starts on built-in defaults; you only need it to enable those features. Site routes live elsewhere, in the sites/ directory (see the YAML & Tag Reference).

Gateway-levelOptionalGenerate with -init
Generate a template

Running litegate -init produces a fully commented config.example.yaml; this page explains each block of it. After editing, validate with litegate -t.

1Listeners: http / real_ip

yaml
mode: observability        # performance (high throughput, less logging) / observability (default, full monitoring)

http:
    port: 80               # HTTP listening port (also needed for the ACME HTTP challenge)
    https_port: 443        # HTTPS listening port
    response_header_timeout: 30   # seconds to wait for the upstream response header

real_ip:
    trusted_proxies: ["127.0.0.1", "10.0.0.5"]   # only trust X-Forwarded-For from these proxies
    selector_trusted_clients: ["127.0.0.1", "10.0.0.0/8"]  # trusted internal clients allowed to do east-west X-Selector-Sid routing
FieldDefaultNotes
modeobservabilityperformance trades logging for throughput; observability keeps full monitoring and access logs.
http.port80HTTP listening port. The HTTP-01 challenge for auto-certs depends on it.
http.https_port443HTTPS listening port. Enabled automatically once certs are ready.
http.response_header_timeout30Max seconds to wait for the upstream response header.
real_ip.trusted_proxies["127.0.0.1"]Only requests from these IPs/CIDRs have their X-Forwarded-For trusted.
real_ip.selector_trusted_clients—Trusted internal clients allowed to carry X-Selector-Sid for physical shard routing.

2Global IP restriction

A server-level firewall with the highest priority, applied before site matching.

yaml
ip_restriction:
    # Whitelist: if non-empty, ONLY these IPs are allowed (use with caution)
    # allow_ips: ["127.0.0.1", "192.168.0.0/16"]
    # Blacklist: deny specific IPs access to any site
    deny_ips:
        # - "1.2.3.4"
Mind the priority

Once allow_ips is non-empty it becomes "allow-list only" and blocks everything else. Prefer deny_ips in production. Routes also have their own ip_restriction, which stacks on top.

3TLS & auto certs

Choose one of two paths: static certificates (tls) or automatic ACME certificates (auto_cert, recommended). How it works: see the Certificate Lifecycle.

yaml
tls:
    enabled: false
    certs_dir: ./certs

auto_cert:
    enabled: true
    email: admin@example.com      # email for expiry notifications
    certs_dir: ./certs
    renew_days: 10                # renew 10 days before expiry
    check_interval: 24            # check every 24 hours
    use_staging: false            # set true while testing to avoid rate limits
    propagation_dns: ["223.5.5.5", "114.114.114.114"]

    # DNS providers (Aliyun/Cloudflare/Tencent/Huawei) for DNS-01 and wildcards
    dns_providers:
        # - name: "cf-global"
        #   type: "cloudflare"
        #   domains: ["*.example.com"]
        #   config: { api_token: "..." }

    # On-Demand TLS (SaaS mode): issue certs for unknown domains on the fly
    # on_demand:
    #    enabled: false
    #    ask: "http://your-backend/check-domain"
FieldDefaultNotes
auto_cert.enabledtrueEnable automatic HTTPS. Any domain not matched in dns_providers defaults to HTTP-01.
auto_cert.emailadmin@litegate.comACME account email, used for expiry notifications.
auto_cert.renew_days10How many days before expiry to renew.
auto_cert.use_stagingfalseUse the Let's Encrypt staging environment while testing to avoid rate limits.
tls.enabledfalseUse local static certificates (alternative to auto_cert).

4Registry & service discovery

consul / litemesh handle the gateway's own config watch, cert storage and self-registration; service_discovery is a separate module defining where backend services are discovered. Discovery details: see Service Discovery.

yaml
litemesh:
    enabled: true
    address: "192.168.1.100:8787"
    namespace: "default"
    roles:
        service_register: true     # register LiteGate itself as a service

service_discovery:
    catalogs:
        - enabled: false
          provider: litemesh        # litemesh / litedeploy / docker / consul / discovery
          namespace: "default"
          default_domain: ""        # default suffix: service -> service.example.com
        - enabled: false
          provider: docker
          socket_path: "/var/run/docker.sock"
          network_mode: "internal"  # internal (container IP) / external (host IP:port)
FieldNotes
litemesh.enabled / consul.enabledWhether to use that registry for config watch / cert storage / self-registration.
service_discovery.catalogs[].providerDiscovery source: litemesh / litedeploy / docker / consul / discovery.
...default_domainAuto-append a domain suffix to discovered services (orders → orders.example.com).
...network_mode (docker)internal uses container IPs, external uses host-mapped ports.
Remote bootstrap

You can also skip the local config.yaml and bootstrap the main config from KV with litegate -consul-addr ... or -litemesh-addr ...; the env vars LITEGATE_CONSUL_ADDR/TOKEN/KEY are also supported.

5Cache & Redis

Shared state such as IDS sessions, rate limiting and negative caching relies on Redis; without it, it falls back to local in-memory (fine for single-node).

yaml
cache:
    local_ttl: "60s"
    redis:
        enabled: false
        mode: "standalone"        # standalone / cluster
        address: "localhost:6379"
        db: 1
        pool_size: 20

redis:                            # global metadata store
    enabled: false
    mode: "standalone"
    address: "localhost:6379"
    db: 0
FieldDefaultNotes
cache.redis.enabledfalseEnable for multi-node shared sessions/rate limiting; optional on a single node.
*.modestandalonestandalone or cluster (cluster uses an addrs list).
*.address / addrlocalhost:6379Redis address; addr is an alias for address.

6Identity & auth

The global identity center, OIDC security parameters, and reusable OIDC/OAuth2 and IDS providers. IDS development: see the IDS Development Guide.

yaml
identity_service:
    endpoint: "http://localhost:8080"
    client_id: "gateway-001"
    client_secret: "your-secret"
    timeout: "5s"

oidc_security:
    cookie_secret: ""             # AES-256 cookie encryption key (must be >= 32 bytes if set)
    cookie_same_site: lax         # lax / strict / none
    max_concurrent_sessions: 10

# OIDC / OAuth2 SSO providers (referenced by a route's remote_auth.provider)
auth_providers:
    # google:
    #     type: oidc
    #     issuer: "https://accounts.google.com"
    #     client_id: "..."

# IDS provider connection settings, keyed by name (referenced by a route's ids.provider).
# Freeform per-provider map passed verbatim to a custom IDS plugin factory (DB/Redis/HTTP,
# with secrets). Route IDS behavior itself is self-contained under action.ids; nothing here
# is a routing template. The built-in "http" provider needs no entry here.
ids_providers:
    # myids:
    #     driver: postgres
    #     dsn: "postgres://user:pass@10.0.0.5:5432/tenants"
    #     redis_addr: "10.0.0.6:6379"
cookie_secret length

Once set, oidc_security.cookie_secret must be >= 32 bytes, or startup validation fails. ids_providers entries are not merged into auth_providers; they are separate per-provider connection settings handed to the IDS plugin factory at instantiation.

7Logging & observability

Logs, access logs, metrics, Dashboard. Operational and troubleshooting details: see Observability & Deployment.

yaml
log:
    level: info                   # debug / info / warn / error
    format: console               # console / json
    access_log:
        enabled: true
        stdout: true              # print access logs to the console
        routing_detail: false     # log detailed routing decisions

metrics:
    enabled: false                # Prometheus metrics (also via -enable-metrics)
    port: 9090

pprof:
    enabled: false
    port: 6060

dashboard:
    enabled: false                # disabled by default for security
    port: 9999
    username: "admin"
    password: ""                  # generate a Bcrypt hash with litegate -hash
    token: "..."                  # management API token (X-API-Key)
    allowed_origins: []           # empty = same-origin only
    require_secure_cookie: false  # set true when the Dashboard is served over HTTPS
FieldDefaultNotes
log.level / formatinfo / consoleLevel and output format (json recommended in production).
log.access_log.enabledtrueStructured access-audit logging switch.
metrics.enabled / portfalse / 9090Expose the Prometheus /metrics endpoint.
dashboard.enabled / portfalse / 9999Web admin panel, off by default.
dashboard.password—Must be a Bcrypt hash; generate with litegate -hash "pw".

9Full example

The fully commented standard template is generated by litegate -init. A minimal working global config can be just a few lines:

config.yaml
mode: observability
http:
    port: 80
    https_port: 443
auto_cert:
    enabled: true
    email: admin@example.com
log:
    level: info
    format: json
sites_dir: ./sites
Validate before deploy

After editing, run litegate -t to validate before going live; changes to sites/*.yaml require no restart and hot-reload automatically.
