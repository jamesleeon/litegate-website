A built-in MCP server: let AI assistants run gateway operations directly

LiteGate ships a built-in Model Context Protocol (MCP) server, making the gateway natively integrable with AI agents. When you maintain the gateway with assistants like Cline, Cursor or Roo-Code, the AI can call the built-in toolchain directly: read the environment, validate config, hot-reload, and read live errors and certificate status — "autopilot"-grade operations. Once MCP is connected and you state your intent, you usually no longer need to supply LiteGate docs.

1Enable & connect

Enable the mcp block in the gateway's config.yaml, then point your AI client's mcp_settings.json at the SSE endpoint.

config.yaml
mcp:
  enabled: true
  port: 8080   # MCP SSE listen port
mcp_settings.json
{
  "mcpServers": {
    "litegate-autopilot": {
      "type": "sse",
      "url": "http://127.0.0.1:8080/mcp/sse"
    }
  }
}
Client config location

For Cline / Roo-Code, this is typically at %APPDATA%\Code\User\globalStorage\saoudrizwan.claude-dev\settings\mcp_settings.json. Once connected, the AI can create sites, hot-load routes or troubleshoot via the toolset.

3Typical closed loops

The AI usually runs a class of request end to end along these chains, forming a closed loop from inventory to verification.

① Create a reverse-proxy site

detect_environment→test_upstream→create_proxy_site→preview_config_change→save_site_config→verify_site→get_recent_errors

② Bind a domain to a discovered service

list_services→get_service_detail→bind_domain_to_service→preview_config_change→save_site_config→verify_site

③ Create a TCP/UDP stream

test_upstream→create_stream_proxy→preview_config_change→save_stream_config→verify_stream→get_recent_errors

④ Dynamic frontend ZIP sync & upgrade (Serve KV)

detect_environment→get_gateway_config→set_kv→verify_site→get_recent_errors

Write the latest frontend ZIP version, URL and SHA256 to KV (e.g. litegate/config/front/main); the gateway hot-swaps the deployment, then a real request verifies it.

Why "no docs" works

MCP already exposes schema and config guides, current runtime status and existing config, and provides validation, save, rollback, verification and troubleshooting. So external docs are no longer required input for common site building — they're more of a supplement.

5Still worth strengthening

  • preview_config_change impact analysis could be finer, naming affected sites, ports and listeners.
  • verify_site could return stronger assertions like the matched route and final upstream.
  • verify_stream could further confirm the listener is the LiteGate process itself.
  • The rollback path could harden error handling and status reporting when a backup fails.
