LiteGate Plugin Development Guide

LiteGate uses a compile-time registry plugin model: a plugin is just a plain Go package that registers itself with the public contract package in init(). You write your own main, blank-import the plugin package, call litegate.Run(), and go build produces a gateway binary with your plugin baked in. No fork, no core changes, no dynamic .so — the plugin lives in your own repo. This guide walks you through Action / Middleware / LoadBalancer and IDS identity-governance plugins from scratch.

Compile-time registry Zero core changes Pure Go — no RPC / WASM cost 5 seams
Why not dynamic loading?

LiteGate deliberately rules out runtime .so loading (Go's plugin package is unavailable on Windows and brittle on Linux — it requires exact dependency-version parity), and avoids out-of-process go-plugin / gRPC (per-call RPC overhead) and WASM / interpreters (the perf and isolation cost isn't worth it). The payoff is zero runtime overhead — plugins and built-ins compile into the same binary and travel the same call path.

1The compile-time registry model

Early LiteGate dispatched actions and picked load-balancing algorithms with hardcoded switch statements — adding a capability meant editing the core. That is now unified into a registry model: built-ins and third-party plugins follow exactly the same registration path — the core mounts its own built-in capabilities through this very registry, so the plugin path and the built-in path stay on par and never diverge.

  • A new capability = one Go package + one init(), zero core changes.
  • Enterprises keep private plugins in private repos, compile custom binaries, never conflict with upstream.
  • Third parties only touch the public contract layer pkg/plugin (narrow interfaces / narrow DTOs); internal core types never leak, so the core can evolve freely without breaking your plugin.
Two-layer contract

Built-in / first-party code uses the internal contract (referencing heavy types like loader.Action directly); third-party repos use the public facade pkg/plugin (exposing only Context / ActionConfig / SiteConfig / Endpoint / DiscoveryClient). Both are dispatched through the same registry; the core bridges internal types to the narrow interfaces via adapters.

2Five extension seams

All defined in the public package github.com/jamesleeon/LiteGate/pkg/plugin. The first three are general plugins; the last two (IDS) are identity-governance specific.

Every register function returns error: empty name, nil factory, duplicate, or collision with a built-in type is rejected (e.g. RegisterHandler("proxy", …) always fails).

3Quick start: an Action plugin

Three steps: write the plugin package → write your own main that blank-imports it → go build.

① Plugin package (minimal skeleton)

go
package greeting

import (
    "net/http"

    "github.com/jamesleeon/LiteGate/pkg/plugin"
    "github.com/rs/zerolog/log"
)

// actionType is the name users put in YAML action.type; must not clash with built-ins.
const actionType = "greeting"

type handler struct {
    deps *plugin.Dependencies // only global, read-only, thread-safe things
}

type config struct {
    Message string `json:"message"`
}

// Execute is the only method you must implement. Called per matching request; may be concurrent.
func (h *handler) Execute(ctx *plugin.Context, cfg plugin.ActionConfig) error {
    // Read THIS route's config fresh from cfg per request; never cache it on struct fields.
    var c config
    if err := cfg.Unmarshal(&c); err != nil {
        return err
    }
    if name := ctx.Req.URL.Query().Get("name"); name != "" {
        c.Message += ", " + name
    }
    ctx.W.Header().Set("Content-Type", "text/plain; charset=utf-8")
    ctx.W.WriteHeader(http.StatusOK)
    _, err := ctx.W.Write([]byte(c.Message))
    return err
}

// init auto-registers the plugin when the package is imported.
func init() {
    if err := plugin.RegisterHandler(actionType, func(deps *plugin.Dependencies) plugin.Handler {
        return &handler{deps: deps}
    }); err != nil {
        log.Error().Err(err).Str("plugin", actionType).Msg("register failed")
    }
}

② Your own main

go
package main

import (
    "log"

    "github.com/jamesleeon/LiteGate/pkg/litegate"

    _ "my-litegate/greeting" // ← blank import triggers init() registration
)

func main() {
    if err := litegate.Run(); err != nil {
        log.Fatalf("litegate exited: %v", err)
    }
}

litegate.Run() reuses all of LiteGate's CLI flags and startup logic; you can also pass options like litegate.Run(litegate.WithConfigPath("custom.yaml")).

③ Config: write custom keys directly in YAML

Any key under action that doesn't match a named field is collected into the plugin config, readable via ActionConfig.

yaml
routes:
  - name: "demo"
    match: { path_prefix: "/demo" }
    action:
      type: "greeting"         # ← the name you registered
      message: "hi"            # ← readable via cfg.Get("message") / cfg.Unmarshal
      threshold: 100           # ← numbers arrive as int/float

④ Build

bash
cd my-litegate
go mod tidy
go build -o litegate .
./litegate -config config.yaml
Local development

When hacking on LiteGate source, add replace github.com/jamesleeon/LiteGate => ../LiteGate to your go.mod to point at the local tree.

4Public contract at a glance (pkg/plugin)

These are narrow interfaces / narrow DTOs, deliberately hiding internal heavy types — so the core can evolve without breaking your plugin.

go
// Three behavior interfaces
type Handler interface {
    Execute(ctx *Context, cfg ActionConfig) error
}
type MiddlewareHandler interface {
    Wrap(next http.Handler, site SiteConfig, name string, config map[string]string) http.Handler
    Stage() MiddlewareStage
    Priority() int
}
type LoadBalancer interface {
    Next(endpoints []Endpoint, key string) Endpoint
}

// Request context (the only fields a plugin sees)
type Context struct {
    W             http.ResponseWriter
    Req           *http.Request
    Site          string
    MatchedPrefix string
    ClientIP      string
}

// Read-only config view
type ActionConfig interface {
    Type() string                 // action.type
    Get(key string) (any, bool)   // fetch one raw value
    Unmarshal(target any) error   // decode into your struct (by json tag)
}

// Host dependencies handed to the factory (all narrow interfaces)
type Dependencies struct {
    Logger    *zerolog.Logger
    Discovery DiscoveryClient // GetEndpoints(ctx, name, ns, tag) ([]string, error)
}

// Optional lifecycle interfaces (Action supports all; Middleware/LB support Provisioner+CleanerUpper)
type Provisioner  interface { Provision(deps *Dependencies) error } // first instantiation, global init
type Validator    interface { Validate(cfg ActionConfig) error }    // config-load validation, fail-closed
type CleanerUpper interface { Cleanup() error }                     // release resources gracefully
Reading configNotes
cfg.Type()returns action.type
cfg.Get(key)fetch one raw value — always type-assert and handle failure
cfg.Unmarshal(&struct)recommended: decode by json tag, no per-field assertions

5Middleware plugin

Middleware processes on the request chain, then calls next to hand off. Put initialization in the factory (Wrap runs once at build time; the returned http.Handler is what runs per request).

go
type myMW struct{ deps *plugin.Dependencies }

func (h *myMW) Wrap(next http.Handler, site plugin.SiteConfig, name string, config map[string]string) http.Handler {
    threshold := config["threshold"] // config is map[string]string
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // ...pre-processing (site.Domain() etc. available)...
        next.ServeHTTP(w, r) // don't forget next, or the chain breaks
    })
}
func (h *myMW) Stage() plugin.MiddlewareStage { return plugin.StageAuth }
func (h *myMW) Priority() int                 { return 500 }

func init() {
    _ = plugin.RegisterMiddleware("my_mw", func(deps *plugin.Dependencies) plugin.MiddlewareHandler {
        return &myMW{deps: deps}
    })
}

Stage (coarse ordering) + Priority (fine ordering within a stage) together decide execution order, regardless of declaration order in config. Within a stage, higher Priority runs further out / earlier.

StageOrderTypical useBuilt-in reference
StagePreSecurityfirstthings that must precede security (CORS preflight)cors = 1000
StageAuthnextauth, IP restrictionjwt_auth = 800 · api_key = 700
StageRateLimitthenrate limit, WAF, circuit breakerratelimit = 300
StageRoutelast (closest to business)rewrite, compression, cachestrip_prefix = 100

Enable it by declaring my_mw in the site/route middleware list. Middleware may optionally implement Provisioner / CleanerUpper to join lifecycle management.

6LoadBalancer plugin

Custom upstream selection. Next is called on every upstream pick (hot path) — it must be thread-safe and non-blocking.

go
type gpuLB struct{}

// Next picks one endpoint to return; it MUST be an element of the input slice.
func (l *gpuLB) Next(endpoints []plugin.Endpoint, key string) plugin.Endpoint {
    var best plugin.Endpoint
    for _, ep := range endpoints {
        if !ep.Healthy() {
            continue
        }
        if best == nil || ep.ActiveRequests() < best.ActiveRequests() {
            best = ep
        }
    }
    return best // nil when none available; host falls back to round_robin
}

func init() {
    _ = plugin.RegisterLoadBalancer("gpu_aware", func() plugin.LoadBalancer {
        return &gpuLB{} // note: LB factory takes no Dependencies
    })
}

Config: action.lb_policy: "gpu_aware".

Three hard rules

① Return an element of the input endpoints (the host maps it back to a real endpoint); nil = give up, host falls back to round_robin. ② Never cache an Endpoint reference on a struct field — it's valid only for this call. ③ Next is on the hot path; no blocking I/O.

7IDS identity-governance plugins

When the IDS engine handles a request it needs a credential + routing payload (CredentialResponse): identity, headers to inject, rate-limit hints, plus the selector / meta used for dynamic routing. By default that payload comes from the gateway making an HTTP call to a remote auth service; a plugin lets you produce the same structure with in-process Go code — the engine then proceeds as usual (caching, route injection, rate limiting unchanged).

Three pluggable seams (read / write / control)

Credential Provider (read — verify/fetch) · Session Issuer (write — issue) · Mode Processor (control — intercept lifecycle). The first two are selected independently, each default to built-in "http", and apply per route. For using IDS itself, see the IDS Development Guide.

Credential Provider (read / verify)

Implement CredentialProvider, choose your own data source (local rules / DB / Redis / even your own HTTP), return *plugin.CredentialResponse.

go
func (p *myProvider) GetCredentials(ctx context.Context, req *plugin.CredentialRequestContext) (*plugin.CredentialResponse, error) {
    if req.Token == "" {
        // no such credential → negative cache (core caches briefly, blocks repeated misses)
        return &plugin.CredentialResponse{NotFound: true}, nil
    }
    return &plugin.CredentialResponse{
        Auth: map[string]string{"app_id": req.AppID, "ttl": "300"}, // ttl controls cache seconds
        Route: plugin.RouteNode{
            Selector: map[string]string{"tenant": "corp_abc"}, // ← dynamic routing
            Meta:     map[string]string{"gray": "true"},
        },
        Head: map[string]string{"X-Tenant-Id": "corp_abc"}, // inject downstream
    }, nil
}

func init() {
    _ = plugin.RegisterProvider("myids", func(deps *plugin.Dependencies) (plugin.CredentialProvider, error) {
        return &myProvider{deps: deps}, nil
    })
}

Config (per route): set ids.ids_provider: "myids"; leave empty = default http. Different routes can use different providers.

Session Issuer (write / issue)

Issuing (login → token) is a separate path: implement SessionIssuer, configure via ids.ids_issuer (independent of the provider).

go
func (i *myIssuer) Authenticate(ctx context.Context, r *http.Request, cfg plugin.SessionIssuerConfig) (*plugin.CredentialResponse, *http.Response, error) {
    // In-process issuing: compute and return; *http.Response is nil (core nil-guards it)
    return &plugin.CredentialResponse{
        Auth: map[string]string{"session": "...", "ttl": "7200"},
    }, nil, nil
}

func init() {
    _ = plugin.RegisterSessionIssuer("myids", func(deps *plugin.Dependencies) (plugin.SessionIssuer, error) {
        return &myIssuer{}, nil
    })
}
Read/write decoupled

A verify-only plugin registers just CredentialProvider and configures only ids_provider; leave ids_issuer empty to fall back to built-in http — it won't fail startup just because you didn't implement issuing.

Mode Processor (control / intercept)

To fully control request-lifecycle hijacking (pre-security checks, hard cut-off, transforming the downstream response), implement IDSModeProcessor.

go
func (p *MyMode) ModeName() string { return "my-mode" }

func (p *MyMode) ProcessRequest(w http.ResponseWriter, r *http.Request, ctx context.Context, config *plugin.IDSConfig) (context.Context, bool) {
    if r.Header.Get("X-My-Secure-Token") == "" {
        w.WriteHeader(http.StatusForbidden)
        _, _ = w.Write([]byte("Forbidden"))
        return ctx, false // cut off the request
    }
    return ctx, true // proceed
}

func (p *MyMode) WrapResponse(w http.ResponseWriter, r *http.Request, rw http.ResponseWriter, config *plugin.IDSConfig) http.ResponseWriter {
    return rw // don't hijack the response by default
}

func init() {
    _ = plugin.RegisterIDSModeProcessor(&MyMode{})
    // Overriding a built-in mode (verify/issue…) must be explicit: plugin.ReplaceIDSModeProcessor(&MyVerify{})
}

Config: route ids.mode: "my-mode" activates it.

Why this is clean

CredentialResponse / CredentialRequestContext are pure data + stdlib http.Request, referencing no internal types — so the internal code adopted these public types directly, and the built-in "http" origin registers into the same public registry. Third-party plugins and the built-in origin share the exact same contract and dispatch path.

8Build & distribute

Option 1: litegate build --with (built-in one-shot build, recommended)

LiteGate ships a one-shot build CLI that generates main.go + go.mod, fetches deps, pins the version, and compiles.

bash
litegate build --with github.com/acme/litegate-llm -o my-litegate

# Local dev: replace the LiteGate dependency with a local source tree
litegate build --with ./greeting --replace=../LiteGate -o my-litegate
# or set the env var LITEGATE_LOCAL_PATH=../LiteGate
  • Requires a local Go toolchain (auto-detects and adapts to your Go version) + network access to the Go module proxy.
  • Repeat --with to stack multiple plugins; -o/--output sets the artifact path.
Dev versions require --replace

When LiteGate runs a development version (version string containing -dev) and you're not inside the LiteGate module directory, that version doesn't exist on the module proxy, so litegate build requires an explicit --replace=<local path> or LITEGATE_LOCAL_PATH.

Option 2: you own main (native go build)

As in section 3, create your own Go module, blank-import the plugin in main, call litegate.Run(), and compile with native go build. Lock the version via go.mod; use replace for local dev. A template lives in the repo at examples/plugin-template.

No dynamic loading

There is no .so hot-swap. Plugins must be compiled into the binary — a deliberate trade-off for zero runtime overhead.

9Lifecycle at a glance

text
Program start
  └─ each plugin's init() → Register{Handler,Middleware,LoadBalancer,Provider,...}
Config load / hot reload
  └─ loader validates per route → Validator.Validate(cfg)  [Action only; fail-closed: bad config = no start]
First request hits the type (lazy load, singleflight-deduped)
  └─ factory(deps) → Provisioner.Provision(deps)          [once, global init, no route config here]
Every request
  └─ Handler.Execute() / middleware chain / LB.Next()     [high concurrency, type-level shared instance]
Shutdown / reload unload
  └─ CleanerUpper.Cleanup()                               [once, graceful release]

Every boundary is guarded by recover: panics in the factory / Provision / Validate / plugin Execute are turned into an error or a 500 and never crash the process. Don't rely on it as normal flow, though.

10Pitfalls checklist

  1. Handler / LB are cross-route, type-level singletons. All routes using that type share one instance, called concurrently. Per-route config must be read fresh from cfg inside Execute, never cached on struct fields, or routes overwrite each other.
  2. You can't override built-in types. RegisterHandler("proxy", …) / RegisterMiddleware("cors", …) / RegisterLoadBalancer("round_robin", …) all return errors. Pick a non-clashing name. (To override the built-in IDS "http", use ReplaceProvider / ReplaceSessionIssuer.)
  3. Validate is fail-closed (Action only). Any invalid route config blocks gateway start/reload — that's correct. Push config validation into Validate.
  4. Provision has no route config. It does route-agnostic global init only; route-dependent logic belongs in Execute.
  5. Mind value types. YAML numbers arrive as int/float64, not strings. Prefer cfg.Unmarshal(&struct).
  6. Minimal dependencies. A plugin only gets Logger and Discovery (narrow interfaces). Need more? Push to add a narrow interface in pkg/plugin rather than expecting internal concrete types.
  7. IDS: don't cache yourself. The core already does L1 (LRU) + L2 (Redis) + negative cache + singleflight outside your provider call. Just return raw data. Cache duration via Auth["ttl"] (seconds).
  8. IDS: fail-closed by default. On error the core applies fail_policy (default deny); auth failure should return an error or NotFound, never silently pass.
  9. Don't leak goroutines across reloads. Background goroutines started in Provision must watch the host cancel signal and stop in Cleanup, or repeated reloads accumulate leaks.

11Testing your plugin

In your own repo you can unit-test against the pkg/plugin interfaces directly (construct a plugin.Context and a stub implementing ActionConfig) without booting the whole gateway.

  1. Unit test: build a *plugin.Context (with httptest.NewRecorder() + httptest.NewRequest()) and an ActionConfig stub, call Execute, assert the response.
  2. Registration check: assert RegisterHandler returns nil; duplicate / built-in override returns an error.
  3. End-to-end: see the repo's internal/action/plugin_external_test.go — one E2E case per plugin type using the public API.
  4. Config test: litegate -t -config config.yaml checks config syntax and triggers Action Validate.

12FAQ

Can I load .so plugins dynamically?

No. LiteGate uses a compile-time registry; plugins must be compiled into the binary. This is a deliberate trade-off for zero runtime overhead and cross-platform stability (Go's plugin is unavailable on Windows).

What host capabilities does a plugin get?

Third parties get Dependencies{ Logger, Discovery } — a logger plus a narrow service-discovery view. Bring your own Redis / storage; to get more host capability, add a narrow interface in pkg/plugin.

Any difference between built-ins and my plugin?

They dispatch through the same registry, but capabilities are asymmetric: built-in Actions take the "host internal shortcut" (direct access to Executor private state), while third parties go through the public Dependencies contract. A deliberate perf/encapsulation trade-off.

Will a plugin panic take down the gateway?

No. The factory / Provision / Validate / plugin Execute are all recovered into an error or a 500. Still, handle your own errors — don't treat recover as normal flow.

Can one name be both an Action and a Middleware?

Yes — they're independent registries with separate namespaces. Internal Register* does cross-check the external registry to prevent duplicates within the same table.
