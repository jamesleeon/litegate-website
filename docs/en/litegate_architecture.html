A reversed execution pipeline, on a lock-free data plane that hot-reloads in nanoseconds

LiteGate targets private and hybrid clouds with heterogeneous infrastructure, running natively on bare metal, VMs and Docker without being bound to Kubernetes. Its engineering core distills to three ideas: a fully decoupled control and data plane, a physically reversed execution pipeline, and unified multi-source service discovery.

0System Diagram

LiteGate features a clean three-column architecture: incoming traffic from the internet on the left resolves through custom hosts and protocols into the central LiteGate routing engine. After hitting rules and middleware chains, besides proxying to the backend infrastructure on the right, LiteGate's unique built-in Web Server handles SPA hosting, Markdown files, and WebDAV requests directly in-place for local response loops without forwarding to external nginx instances. All service discovery providers support seamless hot reloads, with Kubernetes integration currently progressing under generic protocols.

1Three core design philosophies

Control / data plane split

Decoupling config flow from traffic flow lets the data plane run lock-free and allocation-free, trading for extreme throughput.

Reversed execution pipeline

Physical execution isn't a mechanical L1→L7 stack — network defense and quotas go first, security and identity injection in the middle, load balancing and link governance last.

Multi-source hub & in-house consensus

Compatible with Consul and Discovery protocols, while natively integrating the minimal in-house Litemesh Gossip protocol for decentralized, efficient clustering.

2Control plane and data plane

Two highly decoupled planes collaborate to deliver "route-transparent, zero-overhead hot updates" under high concurrency.

Control plane responsibilities

CatalogLoader is an open adapter: it merges endpoints from static files, Consul Catalog, Litemesh container instances and external API registries indiscriminately, and can even carry the same service's Consul physical instances alongside Litemesh container instances.

Data plane responsibilities

The router holds a single read-only route-tree pointer with no global mutex, matching against a read-only in-memory copy; all upstreams are compiled and cached in PoolManager, so requests in the hot path never reflect or reallocate memory.

3The reversed pipeline: a request's full journey

Conceptually L1–L7, but with a deeply reversed physical order. Security and rate limiting are front-loaded so the gateway never collapses just because a backend auth service fails.

L1 · Edge defense · router.go

Router.ServeHTTP (pre-processing)

Before any config context applies: global IP filtering (fast-fail); stripping client-forged sensitive headers like X-Tenant-ID/X-SID; URL path normalization; zero-overhead interception of ACME challenges and /_litegate/ system endpoints (a hit returns immediately, never entering the business layer).

L2 · Route match

GetSiteMatch

An internal Trie tree matches Host/Path to a site [id] and route, injecting the matched MatchedPrefix into the Action Context. No Site/Route match returns 404 here.

L3 · Lifecycle container · core.go

CoreHandler (TraceID & lifecycle)

Only after a route hit is the request wrapped in CoreHandler: a W3C-compliant X-Trace-Id (if the client didn't supply one), panic isolation via recover() (a single middleware panic still only returns 500, the process never dies), and logMeta initialization.

L4 · Middleware chain · BuildRouteHandler

Core → plugins → CORS → IP → WAF → RateLimit → Auth → Compress

RateLimit runs before Auth: it caps the route's total throughput, both protecting backends before they crash and shielding downstream IDS auth services and Redis from invalid floods. After Auth (OIDC / IDSSession / RemoteAuth) succeeds, decrypted tenant metadata is mapped into standard headers (X-Tenant-ID, X-SID, X-DB-Name) for zero-intrusion context propagation.

L3/L5/L6 · Proxy execution · proxy.go

Proxy Executor (routing & rewriting)

Path rewriting (StripPrefix dynamically reads MatchedPrefix / PrependPrefix); canary draw (per-request dice roll on router.gray_weight, hits route to upstream.gray nodes, empty canary pool falls back to the full pool); two-phase FilterEndpoints (hard selector + soft meta + auto fallback); LB node selection (RoundRobin / IpHash / Weighted / P2C) with timeout/retry/breaker.

Backend

Outbound dispatch → write back

Physically dispatches to the chosen healthy instance, receives the response and writes it back to the client.

L7 · Observability · core.go (defer)

Observability

On completion, aggregates the whole pipeline: route ID, chosen backend address, auth result, per-stage latency — emitted as standardized JSON logs.

4Zero-downtime hot reload: atomic pointer swap

In production, any route reset (adding a domain, changing canary weights) must not break TCP connections in transit. LiteGate avoids jitter with an atomic pointer swap.

  1. Detect change
    Watcher or KV detects a config edit / discovery change.
  2. Build offline
    Copy and parse the full config, silently build a brand-new Trie tree in memory, run validation and port-conflict checks — without touching the live data plane.
  3. Atomic swap
    Once ready, run Val.Store(NewTree); the pointer switches in nanoseconds.
  4. Reclaim old tree
    The old tree has no new active references and is reclaimed by GC after draining in-flight requests.
Fail-Safe

If building NewTree fails (bad format, cert conflict, etc.), the reload aborts with an error and the pointer never swaps — the old tree keeps serving stably. Ports 80/443 stay bound throughout, with no process fork or graceful port shutdown and zero system-level I/O cost.

5Litemesh cluster consensus

For multi-node clusters, LiteGate ditches heavy components like K8s / ZooKeeper, reaching eventual consistency with an improved Gossip protocol.

Notify-Pull mode

Traditional Gossip broadcasts member detail frequently in large clusters, risking bandwidth storms. Litemesh instead broadcasts only a tiny "change ID" when a node changes; receivers decide locally, then asynchronously unicast a pull for details from the source — cutting bandwidth by about 85%.

Anti-entropy recovery

Packet loss can cause temporary state drift. Every 5 minutes Litemesh runs a lightweight full-state check against a random node, reconciling differences to guarantee eventual consistency even after severe network partitions.

6L4 Stream and elastic HA

Beyond 7-layer web serving, LiteGate natively integrates a high-performance L4 Stream Server for non-HTTP traffic — MySQL connection pools, DNS protocol proxying, and low-level IoT TCP/UDP data. The forwarding layer embeds a closed-loop traffic governance engine for microsecond-level self-healing.

Canary soft routing

A per-request dice roll on gray_weight routes hits into the canary pool; if the whole canary pool is down, traffic auto-falls back to the full healthy pool for HA.

Two-phase instance filter

Phase one hard-matches router.selector and tenant tags — an empty isolation pool returns 502 directly, so cross-tenant traffic is never physically mixed; phase two soft-matches router.meta within the isolation pool, falling back if nothing matches.

Breaker & retries

Each outbound failure, when the retry factor allows, auto-resends to a different physical node; an instance that keeps erroring past the breaker threshold is marked "subhealthy" and removed from the active pool.

7Performance & connection-pool tuning

LiteGate hits ~55k RPS at ~50MB resident memory on laptop-class hardware thanks to aggressive PoolManager tuning. Parameters live in DefaultProxyConfig() in internal/proxy/types.go.

go
func DefaultProxyConfig() *ProxyConfig {
    return &ProxyConfig{
        MaxIdleConns:        4096,             // global max idle conns
        MaxIdleConnsPerHost: 4096,             // max idle conns per backend host
        MaxConnsPerHost:     8192,             // max total conns per backend host
        IdleConnTimeout:     90 * time.Second,

        DefaultDialTimeout:           10 * time.Second,
        DefaultTLSHandshakeTimeout:   10 * time.Second,
        DefaultResponseHeaderTimeout: 30 * time.Second,

        CleanupInterval:        5 * time.Minute,  // idle conn reaper
        TransportIdleTimeout:   10 * time.Minute,
        EnableTransportCleanup: true,
    }
}

Very high idle keep-alives per host

Caddy or standard Go proxies default to 2 or 100 idle conns per host; under load the idle pool fills instantly, forcing endless "connect → transfer → close" cycles with huge handshake cost and easy ephemeral-port exhaustion (Address already in use). LiteGate raises this to 4096, so concurrent requests reuse long-lived connections and TLS handshake cost is amortized to zero.

Dynamic transport cleanup & self-healing

The CleanupInterval: 5m background reaper sweeps the idle pool; once a backend's traffic drops or it is scaled out, those idle conns are physically closed and memory is returned smoothly to the OS, keeping resident usage at a low ~50MB.
