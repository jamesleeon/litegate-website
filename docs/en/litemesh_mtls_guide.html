1. How mTLS works

LiteGate mTLS is built around Litemesh workload identity. LiteGate obtains its own client certificate from Litemesh CA, then verifies backend certificates against the Litemesh root CA and the expected SPIFFE identity.

  • Certificate lifecycle: certificates are issued and rotated dynamically by Litemesh. LiteGate does not need static client certificate files for downstream mTLS.
  • Identity verification: backend URI SAN must match the expected SPIFFE ID.
  • Fail-closed: if the certificate provider is unavailable or identity verification fails, mTLS traffic is rejected.
In mTLS mode, insecure_skip_verify does not bypass Litemesh root CA and SPIFFE identity verification.

2. Backend dual mode: plain HTTP + mTLS

The recommended backend deployment listens on two ports: a plain HTTP port such as :8080 and an mTLS HTTPS port such as :8443. The service can first run without the mtls tag, then add the mTLS listener and tag when ready.

tlsConfig := &tls.Config{
    MinVersion: tls.VersionTLS12,
    GetCertificate: certManager.GetCertificate,
    GetClientCertificate: certManager.GetClientCertificate,
    ClientAuth: tls.RequireAndVerifyClientCert,
    ClientCAs: rootPool,
    RootCAs: rootPool,
    VerifyConnection: allowClientSPIFFE(
        "spiffe://litemesh.local/ns/gateway/sa/litegate",
    ),
}

server := &http.Server{
    Addr: ":8443",
    Handler: mux,
    TLSConfig: tlsConfig,
}
log.Fatal(server.ListenAndServeTLS("", ""))

You can remove the SPIFFE allow-list during early testing, but production backends should usually restrict which workload identities may call them.

3. Register with Litemesh

Register the normal port, mTLS port, tags, and metadata together. The bare mtls tag is the shared switch used by both LiteGate and the Litemesh SDK.

err := client.Register(&litemesh.ServiceInstance{
    ID: "user-service-1",
    Name: "user-service",
    Namespace: "default",
    Addr: "10.0.0.12",
    Port: 8080,
    Protocol: "http",
    Tags: []string{"mtls"},
    Meta: map[string]string{
        "mtls_port": "8443",
        "spiffe_id": "spiffe://litemesh.local/ns/default/sa/user-service",
        "tls_server_name": "user-service",
    },
})
One declaration, two directions. LiteGate discovery reads the same mtls tag for north-south proxying, and the Litemesh SDK reads it for east-west CallService traffic.

4. North-south LiteGate configuration

Global Litemesh identity

litemesh:
  enabled: true
  address: "http://127.0.0.1:8787"
  mtls: true
  spiffe_id: "spiffe://litemesh.local/ns/gateway/sa/litegate"

mtls: true and a non-empty spiffe_id are required for LiteGate to initialize its certificate manager.

Route configuration

domain: api.example.com
routes:
  - name: user-service
    match:
      path_prefix: /users
    action:
      type: proxy
      upstream_type: litemesh
      service_name: user-service
      namespace: default

In most cases, the route does not need to mention mTLS. The backend instance declares its security mode through Litemesh tags.

Force mTLS on a route

action:
  type: proxy
  upstream_type: litemesh
  service_name: user-service
  namespace: default
  mtls: true
  expected_spiffe_ids:
    - spiffe://litemesh.local/ns/default/sa/user-service

5. East-west SDK calls

East-west traffic is backend-to-backend and does not pass through LiteGate. The SDK discovers the target service, sees the bare mtls tag, switches protocol to HTTPS, uses mtls_port, and sends the request using the injected mTLS client.

cm, _ := client.WatchCertificate(
    "spiffe://litemesh.local/ns/default/sa/order-service",
    privateKey,
)
root, _ := client.GetRootCA()
tlsConfig := litemesh.BuildDynamicTLSConfig(cm, root)
mtlsClient := &http.Client{Transport: &http.Transport{TLSClientConfig: tlsConfig}}
client.SetMTLSClient(mtlsClient)

resp, err := client.CallService(ctx, "user-service", "/api/users", nil)
For production SDK calls, inject a real mTLS client with SetMTLSClient. The SDK fallback for self-signed HTTPS is for local testing only.

8. Compared with Istio sidecars

This approach is an in-process gateway plus SDK model. Istio uses sidecar proxies such as Envoy and iptables interception. Both solve service identity and mTLS, but with different tradeoffs.

DimensionLiteGate + Litemesh SDKIstio sidecar
Runtime modelmTLS logic is inside LiteGate and service processes.Transparent traffic interception through sidecar proxies.
Operational weightLightweight, explicit, easy to debug.More powerful but heavier control plane and proxy fleet.
CoverageSDK paths are protected; direct bypasses are not automatically captured.Can enforce broad traffic capture at the pod level.
Feature breadthFocused on mTLS, discovery, and certificate rotation.Full mesh traffic management, telemetry, policy, and resilience.

9. FAQ

Must I set spiffe_id?

No, if your service follows the default format spiffe://litemesh.local/ns/<namespace>/sa/<service-name>. Set it explicitly when your identity format differs.

What is the difference between port and mtls_port?

port is the plain service port. mtls_port is the TLS port used after mTLS is enabled.

Can insecure_skip_verify skip mTLS verification?

No. LiteGate still enforces Litemesh root CA and SPIFFE identity checks in mTLS mode.

Do north-south and east-west need separate backend declarations?

No. The backend declares Tags: []string{"mtls"} and Meta["mtls_port"] once. LiteGate and the SDK both consume that declaration.
