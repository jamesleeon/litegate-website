One entry model for sites, APIs, service discovery, certificates, identity and mTLS

LiteGate is more than a reverse proxy. It folds Caddy-style site onboarding and Traefik-style dynamic backends into a single gateway: host static sites, proxy APIs, discover dynamic services, auto-issue certificates, run IDS identity decisions, and reach backends securely over Litemesh mTLS — all on a lock-free data plane that hot-reloads in nanoseconds.

Who it's for

Teams where static sites and backend APIs must coexist, who want auto HTTPS, who prefer starting with fixed upstreams and later sliding into service discovery, who deploy as a single binary or Docker rather than reaching for Kubernetes first, or who need SaaS entry features (remote auth, tenant context injection, sid-based steering).

1Start from the minimal viable entry

LiteGate can run from static YAML first, then progressively switch to service discovery, auto certificates, IDS and mTLS. On the left is Lite mode (.lite.yaml); on the right, the equivalent standard mode — same routing model.

lite
app.example.com:
  /api:
    proxy: 127.0.0.1:8080
  /:
    root: ./dist
    spa: true
yaml
domain: app.example.com
routes:
  - name: api
    match: { path_prefix: /api }
    action:
      type: proxy
      upstream: ["127.0.0.1:8080"]
  - name: frontend
    match: { path_prefix: / }
    action:
      type: serve
      root: ./dist
      spa: true
Same routing model throughout

You can later switch seamlessly to Litemesh, LiteDeploy, Docker, Consul or Discovery sources, and add IDS and mTLS — without rewriting your mental model of configuration.

2How a request flows through the gateway

LiteGate is conceptually L1–L7, but its physical execution order is reversed and optimized: network defense and rate limiting first, security and identity injection in the middle, load balancing and link governance last. So even if a backend auth service misbehaves, invalid traffic can't take the gateway down.

L1 · Edge defense

Router.ServeHTTP

Global IP filtering (fast-fail), stripping of client-forged identity headers like X-Tenant-ID/X-SID, path normalization, and interception of ACME challenges and /_litegate/ system paths.

L2 · Route match

Site Match

A read-only Trie tree matches Host/Path to a site [id] and a specific route, injecting MatchedPrefix into the execution context. No match returns 404.

L3 · Lifecycle container

CoreHandler

Generates a W3C-compliant X-Trace-Id, mounts panic recovery, and initializes the structured log container. A single middleware panic returns 500 — the process never crashes.

L4 · Middleware chain

CORS → IP → WAF → RateLimit → Auth → Compress

Rate limiting runs before auth, protecting downstream IDS/Redis from high-frequency invalid requests. On success, Auth injects tenant metadata (X-Tenant-ID, X-SID, X-DB-Name) into headers and context.

L5/L6 · Proxy execution

Proxy Executor

Path rewriting (Strip/Prepend), canary draw, two-phase instance filtering (hard selector + soft meta), LB scheduling (P2C/RR/Weighted/IpHash), and timeout/retry/breaker governance.

Backend

Dispatch & write back

Sends to a healthy matched instance; if mTLS is declared, the hop is upgraded to Litemesh mTLS.

L7 · Observability

Observability (defer)

On completion, aggregates the whole pipeline: route ID, final upstream address, auth result, per-stage latency — emitted as structured JSON logs.

3Architecture at a glance

Control plane

The loader normalizes local files (fsnotify), Consul Catalog, Litemesh membership, external API registries and MCP pushes into ActiveSite / ActiveStream. Config flow and traffic flow are fully decoupled.

Data plane

The router holds a single read-only route-tree pointer (atomic.Value), no global mutex. All upstreams are compiled and cached into PoolManager, so the data plane runs lock-free and allocation-free for maximum throughput.

Atomic hot reload

Adding a domain or changing canary weights builds a brand-new Trie tree offline, then swaps it in via an atomic pointer swap. If the build fails, the pointer never swaps and the old tree keeps serving — enterprise-grade fail-safe.

No jitter

No process fork, no graceful port shutdown. Ports 80/443 stay bound throughout; the old tree is reclaimed by GC after draining in-flight requests, so connections in transit are never disrupted.

5Why it's fast and frugal

LiteGate reaches ~55k RPS at ~50MB resident memory on laptop-class hardware, thanks to aggressive tuning of the proxy connection pool (PoolManager) and a lock-free, allocation-free data plane.

Idle keep-alives per hostMaxIdleConnsPerHost: 4096 (typical gateways use 2–100)Reuses long-lived connections under load, amortizing TLS/TCP handshakes to zero and avoiding ephemeral-port exhaustion
