Up and running in five minutes: from download to your first site

LiteGate is a single-binary, zero-dependency unified ingress gateway: no database, no Kubernetes required up front — copy one executable and run. This page walks you through "get the binary → initialize → start → first site".

Single binaryRuns with zero configHot-reloaded sites~50MB RSS

1Get & install

Pick any of three options. If you just want a quick look, skip to Run in 3 steps and use the -hello mode.

Option 1: Prebuilt binary (simplest)

Drop the litegate executable for your platform onto the machine and make it executable:

bash
# Linux / macOS
chmod +x litegate
./litegate -version

# Windows (PowerShell)
.\litegate.exe -version

Option 2: Build from source

Prerequisites: Go 1.22+ and Git.

bash
git clone https://github.com/jamesleeon/LiteGate.git
cd LiteGate
go mod download
go build -ldflags="-s -w" -o bin/litegate ./cmd/litegate/main.go

Cross-compile for other platforms:

bash
# Linux AMD64
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -ldflags="-s -w" -o bin/litegate-linux-amd64 ./cmd/litegate/main.go
# Linux ARM64 (Raspberry Pi, etc.)
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 go build -ldflags="-s -w" -o bin/litegate-linux-arm64 ./cmd/litegate/main.go
# Windows
CGO_ENABLED=0 GOOS=windows GOARCH=amd64 go build -ldflags="-s -w" -o bin/litegate.exe ./cmd/litegate/main.go

Option 3: Docker

bash
docker build -t litegate:latest .
docker run -d --name litegate -p 80:80 -p 443:443 \
  -v $PWD/config.yaml:/app/config.yaml \
  -v $PWD/sites:/app/sites \
  -v $PWD/certs:/app/certs \
  litegate:latest
Tiny image

The Dockerfile is a multi-stage build; the final Alpine-based image contains only the binary and CA certificates. Production details: see Observability & Deployment.

2CLI flags

All of LiteGate's command-line switches. Day to day you'll only use the first few; the rest are for remote bootstrap and debugging.

FlagPurpose
-config <path>Path to the config file, default config.yaml (overridden by the LITEGATE_CONFIG env var).
-initGenerate a default config.example.yaml and create the sites/ streams/ certs/ www/ directories.
-exampleGenerate a few example site configs in sites/ (hello, static site, frontend+proxy).
-helloStart a temporary hello site at http://localhost:8080 without writing any files. The fastest way to try it.
-hash <password>Generate a Bcrypt hash for a password (for Dashboard, WebDAV SSO users / password, etc.).
-tValidate the configuration and exit (pre-deploy self-check).
-versionPrint the version and exit.
-log-level <level>Override log level: debug / info / warn / error.
-enable-metricsEnable Prometheus metrics (same as setting metrics.enabled in config).
-export-lite <file>Compile a lite .lite.yaml into standard YAML and print to stdout.
-consul-addr / -litemesh-addr ...Bootstrap the main config remotely from Consul / Litemesh KV (see the config reference).

3Run in 3 steps

  1. Initialize
    Generate default config and directory skeleton.
  2. Add a site
    Drop a site YAML into sites/.
  3. Start
    Run the binary — listening takes effect immediately.
bash
# 1) Initialize: generate config.example.yaml and sites/ streams/ certs/ www/
./litegate -init
cp config.example.yaml config.yaml      # tweak as needed; works as-is with zero edits

# 2) Add a site (or generate a few examples first with -example)
./litegate -example

# 3) Start
./litegate -config config.yaml
Want the fastest look?

Skip all of the above and just run ./litegate -hello, then open http://localhost:8080 — it writes no files.

Runs with zero config

When no config.yaml is found, LiteGate enters zero-config mode, starts with built-in defaults, and loads only the sites under sites/. In other words, the global config.yaml is optional — you only need it to enable auto-certs, the Dashboard, service discovery, and similar features.

4Your first site

Site configs live in sites/ — one file per site — and changes are auto-detected and hot-reloaded with no restart. Here's the most common "static frontend + backend API proxy" site:

sites/app.yaml
domain: app.example.com
port: 80                 # optional, defaults to 80 / 443
routes:
  # Backend API: reverse-proxy /api to local 8080, stripping the prefix
  - name: api-backend
    match:
      path_prefix: /api
    action:
      type: proxy
      upstream_type: static
      upstream: ["127.0.0.1:8080"]
      strip_prefix: /api

  # Frontend SPA: everything else served from local files, falling back to index.html
  - name: frontend
    match:
      path_prefix: /
    action:
      type: serve
      root: ./dist
      index: index.html
      spa: true
Action types at a glance

type supports proxy (reverse proxy), serve (static hosting), respond (direct response), redirect, webdav (file service), and markdown (Markdown rendering). Full field semantics: see the YAML & Tag Reference.

Hot-reload check

Save sites/app.yaml and the console immediately logs the site load; change a weight or path and it takes effect instantly, no restart. Under the hood it builds a new routing tree off-line and swaps an atomic pointer, so in-flight connections aren't disturbed.

5Lite mode .lite.yaml

If even standard YAML nesting feels verbose, use the .lite.yaml minimal syntax — the left form is equivalent to the standard YAML on the right; it's the same routing model.

lite
app.example.com:
  /api:
    proxy: 127.0.0.1:8080
  /:
    root: ./dist
    spa: true
yaml
domain: app.example.com
routes:
  - name: api
    match: { path_prefix: /api }
    action:
      type: proxy
      upstream: ["127.0.0.1:8080"]
  - name: frontend
    match: { path_prefix: / }
    action:
      type: serve
      root: ./dist
      spa: true

To see what a .lite.yaml compiles to, verify with -export-lite:

bash
./litegate -export-lite app.lite.yaml
