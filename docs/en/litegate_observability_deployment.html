Observability & Deployment

Put LiteGate into production safely: watch state in real time with the Web Dashboard, audit traffic with structured access logs, wire monitoring and alerts with Prometheus metrics, then use systemd / Docker for process supervision and auto-start.

DashboardAccess logsPrometheussystemd / Docker

1Web Dashboard

A built-in visual control panel: see gateway state in the browser, edit site configs online, inspect discovery backends, and tail live logs. It is off by default for security and must be enabled explicitly in config.yaml.

config.yaml
dashboard:
    enabled: true
    port: 9999
    username: "admin"
    password: "$2a$..."          # must be a Bcrypt hash; generate with litegate -hash
    token: "tk_xxx"             # X-API-Key for the management API
    allowed_origins: []          # empty = same-origin only; set explicit origins for cross-origin
    require_secure_cookie: false # set true when the Dashboard is served over HTTPS

Generate the password hash:

bash
./litegate -hash "your_strong_password"

Live metrics

CPU, memory, goroutines; RPS, total requests, live connections; expiry countdown for every domain's certificate.

Site management

Edit a site's .yaml in the browser with instant effect; per-site error rate and latency.

Service discovery

View online Litemesh/Consul services and confirm the real backend IPs a route binds to.

Log analysis

Live access-log stream, with key-error lookup and search.

Three production-security rules

① Change the default admin username; ② set a strong Bcrypt password; ③ restrict access to the Dashboard port by IP (add an outer ip_restriction, or bind it to an internal network only).

2Access logs

Structured audit logs recording each request's route ID, final upstream address, auth result, and per-stage latency. They can go to stdout or be batched to an external collector.

config.yaml
log:
    level: info
    format: json                 # json recommended in production for ingestion
    access_log:
        enabled: true
        stdout: true             # print to console
        routing_detail: false    # log detailed routing decisions (enable when debugging)
        webhook:
            enabled: false
            url: "http://mq-gateway:8080/access-log"
            batch_size: 100
            interval: "1s"
Pairs with alert logging

log.webhook (distinct from access_log.webhook) can push error/fatal/panic-level logs to Slack / Feishu / DingTalk. One is for full access auditing, the other for key alerts.

3Prometheus metrics

Exposes a standard Prometheus /metrics endpoint for Grafana dashboards.

config.yaml
metrics:
    enabled: true
    port: 9090

Or enable temporarily at startup (without editing config):

bash
./litegate -enable-metrics
prometheus.yml
scrape_configs:
  - job_name: litegate
    static_configs:
      - targets: ["127.0.0.1:9090"]

4pprof profiling

When you need to diagnose CPU / memory / goroutine issues, enable the Go pprof endpoint.

config.yaml
pprof:
    enabled: true
    port: 6060
bash
go tool pprof http://127.0.0.1:6060/debug/pprof/profile?seconds=30
Internal only

Neither the pprof nor the metrics port should be exposed to the public internet — bind them internally or firewall them.

5systemd deployment (Linux)

Install LiteGate as a systemd service for auto-start and process supervision.

1) Install the binary and directories

bash
sudo cp litegate /usr/local/bin/
sudo chmod +x /usr/local/bin/litegate
sudo mkdir -p /opt/litegate/{sites,streams,certs}
sudo cp config.yaml /opt/litegate/

2) Create the service unit

/etc/systemd/system/litegate.service
[Unit]
Description=LiteGate - Cloud-Native API Gateway
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
ExecStart=/usr/local/bin/litegate -config /opt/litegate/config.yaml
WorkingDirectory=/opt/litegate
Restart=always
RestartSec=5
LimitNOFILE=65535

# Security hardening
NoNewPrivileges=true
ProtectSystem=strict
ReadWritePaths=/opt/litegate/certs /opt/litegate/sites

# Logging
StandardOutput=journal
StandardError=journal
SyslogIdentifier=litegate

[Install]
WantedBy=multi-user.target

3) Enable and start

bash
sudo systemctl daemon-reload
sudo systemctl enable litegate
sudo systemctl start litegate
sudo systemctl status litegate
sudo journalctl -u litegate -f          # tail logs
Binding 80/443

Don't run as root. Grant the binary the privileged-port capability instead: sudo setcap 'cap_net_bind_service=+ep' /usr/local/bin/litegate

Updates don't touch the unit

To upgrade, stop the service, replace the binary, and start again. Changes to sites/*.yaml need no restart — LiteGate hot-reloads them automatically.

6Docker deployment

bash
docker build -t litegate:latest .

docker run -d --name litegate \
  -p 80:80 -p 443:443 \
  -v $PWD/config.yaml:/app/config.yaml \
  -v $PWD/sites:/app/sites \
  -v $PWD/certs:/app/certs \
  --restart unless-stopped \
  litegate:latest
Persist certificates

Always mount certs/ as a volume, otherwise rebuilding the container loses issued certs and triggers re-issuance. For multiple replicas, switch to shared storage (Litemesh/Consul KV) for unified certs — see the Certificate Lifecycle.

7Production go-live checklist

Before go-live

  • litegate -t passes config validation
  • log.format: json, level set to info
  • Dashboard: changed username + strong password + IP-restricted
  • auto_cert.use_staging: false (real certs)
  • real_ip.trusted_proxies configured correctly
  • LimitNOFILE / setcap in place

After go-live

  • Prometheus scrapes /metrics
  • Cert expiry countdown healthy, auto-renew working
  • Access logs land / push correctly
  • metrics / pprof ports not exposed publicly
  • Certs consistent across replicas
  • Key-log alert webhook connected
Jitter-free hot reload

Adding domains, changing canary weights, etc. all take effect via "build a new routing tree off-line + atomic pointer swap". 80/443 stay listening the whole time and in-flight connections are unaffected — no graceful restart needed.
