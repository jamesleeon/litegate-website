Backend services use tags to hand entry, governance, and instance selection to LiteGate

The latest tag system centers on litegate.[id].<layer>.<key> and litegate.<layer>.<key>. Tags can come from Docker labels, Consul tags/meta, Litemesh/LiteDeploy meta, or tags/metadata returned by a Discovery agent.

LatestL1-L7No legacy tagsEndpoint filtering

1Where tags live

Global route tags

Use litegate.<layer>.<key>. This is best for one main route or defaults shared by named routes.

Named route tags

Use litegate.<id>.<layer>.<key>. One backend can expose multiple entries such as orders, admin, and metrics.

Instance matching tags

litegate.route.selector.* and litegate.route.meta.* belong to instance tags or metadata. They filter backend instances at request time and do not create routes.

Feasibility

This is already aligned with the code model. The parser merges global and named tags into ServiceSpec values; the proxy passes selector, meta, and tags into discovery filtering, then calls only matched healthy instances.

3Instance-side tags filter at request time

Strict selector isolation

# Route definition
litegate.router.selector.sid=c1
litegate.router.selector.region=hangzhou

# Backend instance metadata or tags
litegate.route.selector.sid=c1
litegate.route.selector.region=hangzhou

Every selector condition must match. Use this for tenant, shard, region, and other hard boundaries.

Soft meta preference

# Route definition
litegate.router.meta.env=gray

# Backend instance metadata or tags
litegate.route.meta.env=gray

Meta matches are preferred. If no instance matches meta, LiteGate falls back to the selector-isolated pool.

Keep meanings separate

litegate.route.selector.* and litegate.route.meta.* are instance identity, not route definition tags. The code intentionally skips them during route generation.

6Operating guidance

Separate route definition from instance identity

Use litegate.<layer>.* for entry, governance, and connection policy. Use litegate.route.selector.* or litegate.route.meta.* for shard, tenant, or gray instance identity.

Use selector for safety boundaries

Use router.selector for tenant, shard, or region isolation. Use router.meta for gray, version, or environment preference.

Keep dimensions stable

Low-cardinality keys such as sid, version, and region are good route dimensions.
