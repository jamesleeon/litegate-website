五分钟跑起来：从下载到第一个站点

LiteGate 是单二进制、零依赖的统一入口网关：不需要数据库、不需要先装 Kubernetes，拷贝一个可执行文件就能跑。本页带你完成「拿到二进制 → 初始化 → 启动 → 第一个站点」的全过程。

单二进制零配置可启动站点热加载~50MB 常驻内存

1获取与安装

三种方式任选其一。如果你只是想先看看效果，直接跳到 三步跑起来 用 -hello 模式即可。

方式一：使用预编译二进制（最简单）

把对应平台的 litegate 可执行文件放到机器上，赋予执行权限即可：

bash
# Linux / macOS
chmod +x litegate
./litegate -version

# Windows (PowerShell)
.\litegate.exe -version

方式二：从源码编译

前置要求：Go 1.22+、Git。

bash
git clone https://github.com/jamesleeon/LiteGate.git
cd LiteGate
go mod download
go build -ldflags="-s -w" -o bin/litegate ./cmd/litegate/main.go

交叉编译到其它平台：

bash
# Linux AMD64
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -ldflags="-s -w" -o bin/litegate-linux-amd64 ./cmd/litegate/main.go
# Linux ARM64（树莓派等）
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 go build -ldflags="-s -w" -o bin/litegate-linux-arm64 ./cmd/litegate/main.go
# Windows
CGO_ENABLED=0 GOOS=windows GOARCH=amd64 go build -ldflags="-s -w" -o bin/litegate.exe ./cmd/litegate/main.go

方式三：Docker

bash
docker build -t litegate:latest .
docker run -d --name litegate -p 80:80 -p 443:443 \
  -v $PWD/config.yaml:/app/config.yaml \
  -v $PWD/sites:/app/sites \
  -v $PWD/certs:/app/certs \
  litegate:latest
镜像很小

Dockerfile 采用多阶段构建，最终镜像基于 Alpine，只含二进制和 CA 证书。生产部署细节见 可观测性与部署。

2命令行参数

LiteGate 的全部命令行开关。日常只会用到前几个，其余用于远程引导与排障。

参数作用
-config <path>指定配置文件路径，默认 config.yaml（可被环境变量 LITEGATE_CONFIG 覆盖）。
-init生成默认 config.example.yaml，并创建 sites/ streams/ certs/ www/ 四个目录。
-example在 sites/ 目录生成几个示例站点配置（hello、静态站、前后端代理）。
-hello不写任何配置文件，直接在 http://localhost:8080 起一个临时 hello 站点。最快的体验方式。
-hash <password>为密码生成 Bcrypt 哈希（用于 Dashboard、WebDAV SSO 等的 users / password）。
-t校验配置是否正确后退出（部署前自检）。
-version打印版本号后退出。
-log-level <level>覆盖日志级别：debug / info / warn / error。
-enable-metrics启用 Prometheus 指标（等价于在 config 里开 metrics.enabled）。
-export-lite <file>把极简 .lite.yaml 编译成标准 YAML 并输出到 stdout。
-consul-addr / -litemesh-addr ...从 Consul / Litemesh KV 远程引导主配置（详见配置参考）。

3三步跑起来

  1. 初始化
    生成默认配置与目录骨架。
  2. 放站点
    把站点 YAML 丢进 sites/。
  3. 启动
    运行二进制，监听即生效。
bash
# 1) 初始化：生成 config.example.yaml 和 sites/ streams/ certs/ www/
./litegate -init
cp config.example.yaml config.yaml      # 按需修改，零改动也能直接用

# 2) 放一个站点（也可以先用 -example 生成几个示例）
./litegate -example

# 3) 启动
./litegate -config config.yaml
想最快看到效果？

跳过上面所有步骤，直接 ./litegate -hello，然后浏览器打开 http://localhost:8080 —— 不会写任何文件。

零配置也能跑

找不到 config.yaml 时，LiteGate 进入零配置模式，使用内置默认值启动并只加载 sites/ 下的站点。也就是说，全局 config.yaml 是可选的，只有要开启自动证书、Dashboard、服务发现等能力时才需要它。

4第一个站点

站点配置放在 sites/ 目录，一个文件一个站点，改动会被自动检测并热加载，无需重启。下面是一个最常见的「前端静态托管 + 后端 API 反代」站点：

sites/app.yaml
domain: app.example.com
port: 80                 # 可选，默认 80 / 443
routes:
  # 后端 API：把 /api 反代到本地 8080，并剥离前缀
  - name: api-backend
    match:
      path_prefix: /api
    action:
      type: proxy
      upstream_type: static
      upstream: ["127.0.0.1:8080"]
      strip_prefix: /api

  # 前端 SPA：其余路径走本地静态文件，找不到时回退 index.html
  - name: frontend
    match:
      path_prefix: /
    action:
      type: serve
      root: ./dist
      index: index.html
      spa: true
动作类型一览

type 支持 proxy（反向代理）、serve（静态托管）、respond（直接响应）、redirect（重定向）、webdav（文件服务）、markdown（Markdown 渲染）。每个字段的完整语义见 YAML 与标签参考。

热加载验证

保存 sites/app.yaml 后，控制台会立刻打印站点加载日志；改一行权重或路径，无需重启即时生效。底层用的是「离线构建新路由树 + 原子指针交换」，不会抖动正在传输的连接。

5极简模式 .lite.yaml

如果连标准 YAML 的层级都嫌啰嗦，可以用 .lite.yaml 极简语法——左边的写法等价于右边的标准 YAML，是同一套路由模型。

lite
app.example.com:
  /api:
    proxy: 127.0.0.1:8080
  /:
    root: ./dist
    spa: true
yaml
domain: app.example.com
routes:
  - name: api
    match: { path_prefix: /api }
    action:
      type: proxy
      upstream: ["127.0.0.1:8080"]
  - name: frontend
    match: { path_prefix: / }
    action:
      type: serve
      root: ./dist
      spa: true

想看 .lite.yaml 编译成什么标准 YAML，用 -export-lite 验证：

bash
./litegate -export-lite app.lite.yaml
