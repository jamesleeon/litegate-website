LiteGate IDS 开发指南

IDS（Identity / Decision Service）不是什么重型框架，也不是你的后端——它本质上就是一个返回特定 JSON 的中间决策服务。LiteGate 在每个被保护的请求上侧路调用它，由它回答四件事：① 这个调用方是谁（怎么验）、② 流量该去哪个物理分片、③ 给下游注入什么头部、④ 对这个调用方/租户限多少流。然后网关据此把真实流量打到你的后端服务。本指南用能跑的示例，让你一眼看懂如何开发自己的 IDS。

协议：HTTP + JSON 最小实现 ~40 行 语言无关 四段决策
先分清两个角色（最容易误解的点）

IDS = 决策服务（中间）：网关问它"这个人怎么处理"，它只回 JSON，流量不进它。
Backend Server = 真后端：你的实际业务服务，网关把流量真正打到这里，并带上 IDS 让注入的头。
换句话说：调用方 → 网关 →（侧路问 IDS 拿决策）→ 打到后端服务。IDS 和后端是两个独立的程序。

网关发给你的请求

http
GET /api/gate/credentials?client_id=app-c1&type=ids HTTP/1.1
X-Gateway-ID: litegate          # 网关身份（来自 provider.client_id）
X-Gateway-Secret: secret        # 网关密钥（来自 provider.client_secret）
  • client_id = 网关从请求里提取出的凭证（由 provider 的 lookup/extractor 决定从哪取，如 {header._appid}）。
  • X-Gateway-ID / X-Gateway-Secret = 网关自报家门，让你的 IDS 能确认"这是我信任的网关在问"。
  • 路径 /api/gate/credentials 可在 provider 的 path 里自定义。

你要返回的四段式 JSON

HTTP 状态码始终 200，业务结果放在 body 的 code 里。code: 1 表示成功，其它（如 404001）表示拒绝。

json
{
  "code": 1,
  "message": "ok",
  "data": {
    "auth":      { "type": "md5", "secret": "secret-c1", "ttl": "300" },        # ① 验签策略
    "route":     { "selector": { "sid": "c1" }, "meta": { "lane": "stable" } },  # ② 物理选路
    "head":      { "X-LiteGate-Identity-Ref": "ref_7f3a9c" },              # ③ 下游注入头（opaque 引用）
    "ratelimit": { "qps": "10", "limit_key": "tenant-c1" }            # ④ 租户限流
  }
}
记住这四段

auth（怎么验调用方）+ route（流量去哪个分片）+ head（给下游塞什么）+ ratelimit（对这个人/租户限多少）。一个 IDS 的全部职责就在这四段里。注意：为确保选路，在 openapi、verify 和 issue 模式下，route.selector 是强约束必填项，如果不返回网关会直接返回 401/403/500 阻断请求。

2最小可用 IDS（40 行）

下面是一个能直接跑通的 IDS（Go），就是本指南实测用的版本。注意它只返回 JSON 决策，不处理业务流量。

go
package main

import ("encoding/json"; "log"; "net/http")

// 商户表：appid → 密钥 + 物理分片 + 配额
var merchant = map[string]struct{ secret, sid, qps string }{
    "app-c1": {"secret-c1", "c1", "10"},   // 免费版 10 QPS
    "app-c2": {"secret-c2", "c2", "100"},  // 付费版 100 QPS
}

func main() {
    http.HandleFunc("/api/gate/credentials", func(w http.ResponseWriter, r *http.Request) {
        // （建议）校验网关身份：r.Header.Get("X-Gateway-ID") == "litegate" ...
        m, ok := merchant[r.URL.Query().Get("client_id")]
        w.Header().Set("Content-Type", "application/json")
        if !ok {
            json.NewEncoder(w).Encode(map[string]any{"code": 404001, "message": "merchant not found"})
            return
        }
        json.NewEncoder(w).Encode(map[string]any{
            "code": 1, "message": "ok",
            "data": map[string]any{
                "auth":      map[string]string{"type": "md5", "secret": m.secret, "ttl": "300"},
                "route":     map[string]any{"selector": map[string]string{"sid": m.sid}},
                "head":      map[string]string{"X-Tenant-ID": "tenant-" + m.sid},
                "ratelimit": map[string]string{"qps": m.qps, "limit_key": "tenant-" + m.sid},
            },
        })
    })
    log.Fatal(http.ListenAndServe(":19091", nil))
}
就这样

没有 SDK、没有框架。查个表，返回四段 JSON，IDS 就完成了。真实系统里把 merchant 换成你的商户库/会话库即可。

3后端服务长什么样（它不是 IDS）

这是真正接流量的后端，跟 IDS 是两个独立程序。它不用懂任何鉴权——身份已经被网关在边缘解析好、以 head 形式注入进来了，后端直接读头即可。下面这个 echo 后端打印它收到的注入头，跑两份（c1 / c2）就能配合分片选路。

go
package main

import ("fmt"; "log"; "net/http"; "os")

func main() {
    sid := os.Args[1]   // "c1" 或 "c2"，标识这是哪个分片实例
    port := os.Args[2]  // 如 18101 / 18102
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        // 后端无需鉴权，直接读网关注入的身份头
        fmt.Fprintf(w, "我是后端 %s ｜ 租户=%s ｜ 路径=%s",
            sid, r.Header.Get("X-Tenant-ID"), r.URL.Path)
    })
    log.Printf("backend %s on :%s", sid, port)
    log.Fatal(http.ListenAndServe("0.0.0.0:"+port, nil))
    // 真实部署时，这个后端会带 Meta{sid: "c1"} 注册到服务发现（litemesh/consul），
    // 网关才能按 IDS 返回的 selector.sid 选中它。见第 7 节。
}
完整拓扑（三个独立程序）

md5client（调用方，签名）→ litegate（网关）→ 侧路问 idsservice（IDS 决策）→ 打到 backend c1 / backend c2（真后端）。
仓库 examples/ids-md5-lab/ 就是这套：md5client / idsservice / apiserver（=后端，可起 c1、c2 两份）。

4四段详解

auth · 验签策略

告诉网关"拿到这个调用方的凭证后，怎么验它是真的"。

字段含义
type验签方式：md5（标准签名）/ md5-secure（加 nonce 防重放）/ none（不验签，仅选路+注入+限流）
secret该调用方密钥。网关用它本地复算签名并比对。type=md5 时必填，否则报 "empty secret"。
ttl本次凭证结果的缓存秒数。网关在 TTL 内不再问你，降低 IDS 压力。

route · 物理选路（selector 硬隔离 + meta 软回退）

告诉网关"这个调用方的流量该打到哪个后端实例"——IDS 最有价值的能力。route 下有两个维度，语义完全不同，别用反：

维度语义找不到实例时用途
selector硬隔离：必须命中返回空 → 拒绝（no healthy upstream），绝不跨界租户物理隔离（如 sid）
meta软偏好：优先命中回退到 selector 池，照常服务灰度 / 版本引导（如 version）

匹配顺序：先用 selector 把候选锁死在硬边界内（命中不了直接淘汰）；再在这个池子里用 meta 做软筛选——meta 有命中就只走命中的；meta 一个都没命中，就回退用整个 selector 池。

用它做灰度（金丝雀）发布

A 集群 10 个实例（都注册 sid=A），其中 8 个跑稳定版、2 个跑新版（额外带 version=v2）：

  • IDS 对灰度租户返回 selector:{sid:A}（硬锁 A 集群）+ meta:{version:v2}（软偏好新版）→ 优先打到那 2 个新版实例；
  • 万一新版实例挂了/下线，meta 没匹配 → 自动回退到 8 个稳定实例，灰度租户照常被服务，不会因为新版没了就 503；
  • 对其它租户不返回 meta → 走全集群稳定版。

放量只需 IDS 改一句 JSON（多放几个租户走 v2），不碰网关、不重启。

selector 与 meta 别用反

需要绝不能泄漏的边界（租户数据隔离）→ 用 selector（宁可空也不跨界）。需要优雅降级的偏好（灰度/版本）→ 用 meta（软，回退保命）。用反了：拿 meta 做隔离会在实例缺失时漏到别的池；拿 selector 做灰度会在新版没起来时直接拒绝。

不需要选路就省略 route，流量走默认负载均衡。后端如何带 sid/version 标签注册见 第 7 节。

head · 下游注入

告诉网关"把这些头注入到转发给后端的请求里"。推荐只注入 opaque 引用（如 X-LiteGate-Identity-Ref），后端凭引用再去自己的租户中心换取库名/租户等敏感信息；不要把租户 ID、库名等明文当默认下发头（会被 Fail-Closed 拦截，除非显式 allowed_headers 白名单，属用户自担风险）。

json
"head": { "X-LiteGate-Identity-Ref": "ref_7f3a9c" }

后端收到该 opaque 引用后，向内部服务解析出真实身份/库映射——这是"身份在网关层解析、下游零负担且不泄漏敏感明文"的核心机制。

ratelimit · 租户/调用方限流（第四段，按选路结果再限流）

选路之后，网关还能针对这个调用方/租户单独限流——而且配额由 IDS 动态返回（免费版 10 QPS、付费版 100 QPS，IDS 说了算）。这跟路由上写死的静态 rate_limit（整条路由全局共享一个桶）是两回事。

字段含义
qps / rate / limit限流值（每秒请求数）。三个 key 任取其一（按此顺序优先）。≤0 或不返回 = 不限流。
limit_key限流维度。显式给则用它——同一个 key 的调用方共享一个配额（比如填租户 ID，则该租户下所有用户合用一份 QPS）。

若不给 limit_key，网关会首先尝试从 ratelimit.key 读取。若仍未提供，则直接**回退**以客户端 IP 隔离。限流桶实际按 站点:路由:dyn:<limit_key> 隔离，超限返回 429 Too Many Requests (Quota Exceeded)。

典型用法

SaaS 多租户：limit_key 填租户 ID，qps 按套餐返回。于是免费租户被限 10 QPS、付费租户 100 QPS，且各租户配额互不影响——全由 IDS 一句 JSON 决定，不用改任何网关配置。

5鉴权模式

IDS 引擎按 mode 分派处理器。provider 配置里必须指定 mode，否则报 "processor not found for the specified mode"。

mode用途典型场景
openapi ★ 最常用开放平台调用：提取 appid → 问 IDS 拿密钥 → 本地 MD5 验签 → 选路+注入+限流对外 OpenAPI、电商签名调用
verify会话/令牌校验：用已有 token 换取身份与选路登录后业务接口、Session 鉴权
issue签发：登录成功后注册新的 token/session 选路映射登录入口（login steering）
refresh刷新：续期已有凭证长会话续租
本指南以 openapi 为主线

openapi 是经过完整端到端实测的模式（验签→放行→head 注入→分片选路→租户限流全通）。verify/issue/refresh 是会话/令牌生命周期的不同阶段，四段式契约一致，区别在网关侧对 token 的处理时机。先把 openapi 跑通，其它同构理解。

6MD5 签名（openapi 模式）

当 auth.type=md5 时，调用方必须对请求签名，网关用 IDS 返回的 secret 本地复算并比对。你的 IDS 不负责验签——只负责把 secret 给网关，验签是网关做的。

签名公式

text
sign = md5( secret + appid + method + timestamp + body + secret )

调用方请求头里带：_appid、_timestamp（Unix 秒）、_method、_sign（md5 十六进制）；md5-secure 另加 _nonce 随机串防重放。

实测结论

正确签名 → 200 + head 注入；错误 secret/签名 → 401（"Invalid signature"）；非法 appid → 401（负缓存）；空 appid → 401。

现成客户端

仓库 examples/ids-md5-lab/md5client/md5apiclient 就是可复用的签名客户端，照它实现调用方即可。

7物理分片选路（IDS 的杀手锏）

IDS 返回 route.selector.sid=c1 后，网关只把流量路由到带 sid=c1 标签的后端实例——实现租户级物理隔离。

网关按以下优先级匹配后端实例（任一命中即可）：

实例侧匹配键
Metadatasid ／ litegate.route.sid ／ litegate.route.selector.sid
Tagssid=c1 ／ litegate.route.sid=c1 ／ 字面量 c1

所以后端服务注册到发现源时带上 Meta: {sid: "c1"} 即可被选中（见第 3 节后端示例尾部注释）。

实测结论

起两个后端(sid=c1 / sid=c2)，IDS 对 app-c1 返回 sid=c1、对 app-c2 返回 sid=c2 —— 各请求 6 次，6/6 精确命中对应分片后端，零泄漏。这就是"IDS 决策真正控制了物理流量去向"的证据。

前提

静态单实例 upstream 配 selector 会被过滤光（报 no healthy upstream）。分片选路要配合服务发现（litemesh/consul/docker）+ 后端实例带 sid 标签使用。

8接入 LiteGate

两步：① 全局声明一个 ids 类型 auth provider；② 路由上引用它。

① 全局 config.yaml

yaml
auth_providers:
  my-ids:
    type: ids
    mode: openapi                      # 必填：选模式
    lookup: "{header._appid}"          # 从哪提取凭证（=client_id）
    url: "http://127.0.0.1:19091"     # 你的 IDS 地址
    path: /api/gate/credentials        # IDS 端点路径
    fail_policy: deny                  # IDS 不可达/拒绝时：deny（默认）或 allow
    client_id: litegate                # → X-Gateway-ID
    client_secret: secret              # → X-Gateway-Secret

② 路由 action 上引用

yaml
routes:
  - name: open-api
    match: { path_prefix: /openapi }
    action:
      type: proxy
      upstream_type: litemesh          # 分片选路需要发现源
      service_name: order-service      # 你的后端服务名（c1/c2 都注册成这个名）
      namespace: default
      ids:
        enabled: true
        provider: my-ids
提取别名说明

lookup 是 extractor 的兼容别名，二者等价（运行时统一投射到 request_extractor）。新配置建议直接用 extractor。

9缓存与拒绝策略

  • 正向缓存：auth.ttl 秒内，网关缓存你的返回，不重复调用 IDS。
  • 负缓存：返回 code=404001（appid 无效）时，网关按 negative_ttl（默认 60s）缓存"此 appid 无效"，避免无效请求反复打 IDS。
  • 失败策略：fail_policy: deny（默认）= IDS 不可达就拒绝（fail-closed，安全）；allow = 放行（fail-open，慎用）。
安全默认

鉴权组件请保持 fail_policy: deny。IDS 挂了就 fail-open 放行，等于鉴权形同虚设。

12常见问题

IDS 就是后端服务吗？

不是。IDS 是侧路的决策服务——网关问它"这个人怎么处理"，它只回 JSON，业务流量不进 IDS。真正接流量的是你的后端服务（backend server），网关把流量打到那里并带上 IDS 让注入的头。两者是两个独立程序。

IDS 需要连数据库 / Redis 吗？

不强制。最小实现就是查内存表的 HTTP 端点。生产里你通常连自己的商户/会话库，但那是你的业务存储，跟 IDS 协议无关。网关侧的缓存/限流由网关自己处理。

静态 rate_limit 和 IDS 的 ratelimit 有什么区别？

路由上的静态 rate_limit 是整条路由一个全局桶；IDS 返回的 ratelimit 是按调用方/租户动态限流，配额由 IDS 实时决定、各租户互相隔离。两者可叠加。

报 "processor not found for the specified mode"？

provider 漏了 mode，补上 mode: openapi。

报 "empty secret returned from identity service"？

auth.type=md5 但没返回 secret。要么返回 secret，要么改 type: none。

合法 appid 却 502 / no healthy upstream？

返回了 route.selector.sid，但 upstream 是静态单实例、没 sid 标签，被过滤光了。分片选路要配合服务发现 + 带 sid 标签的后端（见第 7 节）。

IDS 用什么语言写？

任何能起 HTTP 服务、返回 JSON 的语言都行——协议无关。用 Go 只因它最短。
