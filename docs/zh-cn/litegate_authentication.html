统一鉴权：把"人"和"机器"两类流量都收进网关

LiteGate 把鉴权下沉到网关边缘，后端服务不必各自实现登录与令牌校验。它同时覆盖人类交互流量（OIDC/OAuth2 登录、SSO）与机器/API 流量（JWT、API-Key + 配额），并能把校验出的身份注入下游请求头。

1选型速查：什么场景用什么

方式适合关键点
OIDC / OAuth2人类用户登录、SaaS 后台、需要交互式跳转原生，含 introspection、多 Provider、自定义登录页
JWT 校验机器/服务间 API 调用，已签发 Bearer JWT对 JWKS 校验签名 + issuer/audience，锁定 RS/ES 算法
API-Key + 配额对外开放 API、按调用方限额多源提取 + 按 key 配额（集群分布式限流）
远程鉴权已有自研鉴权服务，想复用子请求转发到外部鉴权端点（类 forward_auth）
Basic Auth内部工具、最简单口令保护开箱即用
mTLS / SPIFFE零信任服务网格、双向证书见 Litemesh mTLS
人 vs 机器

简单记法：浏览器里的真人 → OIDC;程序/服务调用 API → JWT 或 API-Key。三者可在同一网关共存，按路由分别绑定。

2OIDC / OAuth2（人类登录）

面向交互式登录：未登录请求重定向到 IdP，回调后建立会话。原生支持 token introspection、多 Provider，并可挂自定义登录页模板。

sites/app.yaml
routes:
  - name: admin
    match:
      path_prefix: /admin
    action:
      type: proxy
      service_name: admin-svc
      auth:
        type: sso
        secret: "your-long-sso-cookie-signing-secret"
        template: "./templates/auth/login.html"   # 可选自定义登录页

身份决策与租户选路的高级用法见 IDS 开发指南。

3JWT 校验（机器 / API 流量）

面向无交互的服务间调用：校验传入的 Bearer JWT。网关对 JWKS 校验签名，校验 issuer / audience，并把 claim 投影成下游请求头供后端使用。

middleware
middlewares:
  jwt-auth:
    type: jwt_auth
    config:
      jwks_url: "https://idp.example.com/.well-known/jwks.json"
      issuer:   "https://idp.example.com/"
      audience: "api.example.com"
      inject_claims: "sub:X-Auth-Subject,email:X-Auth-Email"
默认安全：锁定签名算法

JWT 校验显式锁定 RS/ES 非对称算法（`WithValidMethods`），杜绝经典的 RS256→HS256 算法混淆攻击与 `alg: none`。

4API Key + 按 key 配额

面向对外开放的 API：校验 API Key，并对每个 key施加独立的 QPS 配额。Key 可从 header / query / Bearer 多种位置提取;配额基于集群共享的 Redis 令牌桶（Redis 不可用时退回本地有界限流），日志中 key 自动脱敏。

middleware
middlewares:
  api-key:
    type: api_key
    config:
      extract_from: "header.X-API-Key"   # 或 query.api_key / bearer
      keys: "key_live_aaa,key_live_bbb"
      quotas: "key_live_aaa:50,key_live_bbb:10"   # 每 key QPS
      default_quota: "5"

5远程鉴权（forward auth）

如果你已有自研鉴权服务，网关可以把鉴权决策子请求转发给它（对标 Nginx auth_request / Caddy forward_auth）：外部端点返回 2xx 放行、401/403 拒绝，并可回传身份头。适合复用既有 RBAC / 会话中心。

6身份注入（claims → header）

无论身份来自 OIDC、JWT 还是远程鉴权，网关都能把解析出的身份（用户 ID、邮箱、租户、角色等）注入下游请求头，后端直接读头即可，无需自己再解析令牌。这是把鉴权真正"下沉到网关"的关键一环，也是多租户 SaaS 入口的常用模式。

防伪造

注入前网关会先剥离客户端伪造的同名权威头，确保后端读到的身份头只可能来自网关本身。

7mTLS / SPIFFE 与 IDS

除上述 HTTP 层鉴权外，LiteGate 还支持传输层与入侵检测层的身份能力：

  • mTLS / SPIFFE：双向证书认证，支持 SPIFFE ID 校验，用于零信任服务网格。见 Litemesh mTLS 指南。
  • IDS 入侵检测链路：插件化的请求签名 / 身份校验引擎，是 LiteGate 的差异化能力。见 IDS 开发指南。
