后端服务用标签把入口、治理和实例选择交给 LiteGate

最新版标签体系以 litegate.[id].<layer>.<key> 和 litegate.<layer>.<key> 为主。标签可以来自 Docker labels、Consul tags/meta、Litemesh/LiteDeploy meta，或 Discovery agent 返回的 tags/metadata。

最新版L1-L7不含兼容标签实例过滤

1标签写在哪里

全局路由标签

格式是 litegate.<layer>.<key>。适合一个服务只暴露一条主路由，或给多个命名路由提供默认值。

命名路由标签

格式是 litegate.<id>.<layer>.<key>。同一个后端可以声明多条入口，例如 orders、admin、metrics。

实例侧匹配标签

litegate.route.selector.* 和 litegate.route.meta.* 放在实例 tags 或 metadata 中，用于请求期过滤后端实例，不会生成路由。

可行性判断

可行，而且已经和代码模型一致。解析器先合并全局标签和命名组标签，再生成 ServiceSpec；运行期代理把 selector、meta、tags 传给 discovery 过滤器，最后只访问命中的健康实例。

2七层标签和配置效果

L1

Entrypoint 物理入口

litegate.entrypoint.bind

声明入口绑定地址。未设置时走网关全局入口。

L2

Ingress 入口匹配

litegate.ingress.hostlitegate.ingress.pathlitegate.ingress.path_prefix

决定外部请求用哪个域名和路径进入后端。path 是精确匹配，path_prefix 是前缀匹配。

L3

Router 路由和实例选择

控制路由优先级、匹配规则、路径重写、负载均衡和实例过滤。selector 是强过滤，meta 是带回退的偏好过滤。

3实例侧标签：请求期才参与过滤

强隔离 selector

# 路由定义
litegate.router.selector.sid=c1
litegate.router.selector.region=hangzhou

# 后端实例 metadata 或 tags
litegate.route.selector.sid=c1
litegate.route.selector.region=hangzhou

所有 selector 条件都要匹配。适合租户、分片、地域等安全边界。

软偏好 meta

# 路由定义
litegate.router.meta.env=gray

# 后端实例 metadata 或 tags
litegate.route.meta.env=gray

meta 命中时优先使用；如果没有任何实例命中，会回退到 selector 已隔离的实例池。

不要混用

litegate.route.selector.* 和 litegate.route.meta.* 是实例身份，不是路由定义标签。代码会刻意跳过它们，保留给运行期实例过滤。

4发现源怎么承载标签

发现源标签位置效果
Docker / Swarmcontainer 或 service labels任意 litegate.* label 会被收集为 metadata，并以 key=value 形式进入 tags。
ConsulService Tags 和 MetaTags/Meta 进入 CatalogLoader，解析成路由定义；实例侧 selector/meta 保留给后端过滤。
Litemesh / LiteDeploy实例 Tags 和 Meta支持路由标签、实例标签和 mTLS 约定。服务变更通过 watcher 推送给 LiteGate。
Discovery agent返回的 tags 和 metadata适合聚合异构注册中心。LiteGate 使用同一套标签解析和 endpoint 过滤逻辑。

5常见配置示例

单入口服务

litegate.ingress.host=api.example.com
litegate.ingress.path_prefix=/orders
litegate.service.name=order-service
litegate.service.timeout=8s
litegate.service.retry=2
litegate.upstream.health_path=/healthz

多路由同服务

litegate.public.ingress.host=api.example.com
litegate.public.ingress.path_prefix=/orders
litegate.public.middleware.ratelimit=500

litegate.admin.ingress.host=admin.example.com
litegate.admin.ingress.path_prefix=/orders
litegate.admin.middleware.auth=oidc
litegate.admin.middleware.auth_provider=corp

租户强隔离

litegate.ingress.host=openapi.example.com
litegate.ingress.path_prefix=/v1
litegate.router.selector.sid=c1
litegate.router.selector.region=hangzhou

litegate.route.selector.sid=c1
litegate.route.selector.region=hangzhou

  spiffe_id: "spiffe://litemesh.local/ns/default/sa/order-service"

6落地建议

路由定义和实例属性分开

入口、治理、连接策略写 litegate.<layer>.*；分片、租户、灰度实例身份写 litegate.route.selector.* 或 litegate.route.meta.*。

先用 selector 做安全边界

多租户、分片、地域隔离优先使用 router.selector。灰度、版本偏好、环境偏好再用 router.meta。

保留少量稳定维度

sid、version、region 这类低基数字段适合放入路由标签。
