LiteGate 插件开发指南

LiteGate 采用 编译期注册表（compile-time registry）插件模型：插件就是一个普通 Go 包，在 init() 里向公共契约包注册自己；你写自己的 main，空导入插件包并调 litegate.Run()，go build 即得带你插件的网关二进制。不用 fork、不用改核心代码、不用动态 .so——插件住在你自己的仓库里。本指南带你从零写出 Action / Middleware / LoadBalancer、DNS Provider（证书 + DDNS）以及 IDS 身份治理插件。

编译期注册表 零核心改动 纯 Go，无 RPC / WASM 开销 6 个接缝
为什么不是动态加载？

LiteGate 明确排除运行时 .so 动态加载（Go plugin 包在 Windows 不可用、Linux 要求依赖版本完全一致，脆弱），也不做进程外 go-plugin / gRPC（每调用一次 RPC 开销）或 WASM/解释器（性能与隔离开销不划算）。换来的是零运行时性能损失——插件和内置功能编译进同一个二进制，走同一条调用路径。

1编译期注册表模型

LiteGate 早期用硬编码 switch 分发 action、选负载均衡算法，加一种能力就得改核心。现已统一为注册表模型：内置功能与第三方插件走完全相同的注册路径——核心自己也通过这张注册表挂载内置能力，从而保证插件通道与内置通道始终对等、不会退化。

  • 新增能力 = 写一个 Go 包 + 一个 init()，核心代码零改动。
  • 企业可在私有仓库维护私有插件，编译定制二进制，不与上游冲突。
  • 第三方只接触公共契约层 pkg/plugin（窄接口 / 窄 DTO），核心内部类型不外泄，因此核心可以自由演进而不破坏你的插件。
两层契约

内置 / 第一方代码走内部契约（直接引用 loader.Action 等重型类型）；第三方独立仓库走公共门面 pkg/plugin（只暴露 Context / ActionConfig / SiteConfig / Endpoint / DiscoveryClient）。两者经由同一张注册表分发，核心用适配器把内部类型桥接到窄接口。

2六个可扩展接缝

全部定义在公共包 github.com/jamesleeon/LiteGate/pkg/plugin。前三类是通用插件，中间两类（IDS）是身份治理专用，最后一类（DNS Provider）接管证书签发与 DDNS。

3快速开始：写一个 Action 插件

三步：写插件包 → 写自己的 main 空导入 → go build。

① 插件包（最小骨架）

go
package greeting

import (
    "net/http"

    "github.com/jamesleeon/LiteGate/pkg/plugin"
    "github.com/rs/zerolog/log"
)

// actionType 是用户写在 YAML action.type 里的名字，不能与内置类型重名。
const actionType = "greeting"

type handler struct {
    deps *plugin.Dependencies // 仅放全局、只读、线程安全的东西
}

type config struct {
    Message string `json:"message"`
}

// Execute 是唯一必须实现的方法。每个命中本路由的请求都会调用它，可能并发。
func (h *handler) Execute(ctx *plugin.Context, cfg plugin.ActionConfig) error {
    // 每请求从 cfg 现读“本条路由”的配置，不要缓存到结构体字段上。
    var c config
    if err := cfg.Unmarshal(&c); err != nil {
        return err
    }
    if name := ctx.Req.URL.Query().Get("name"); name != "" {
        c.Message += ", " + name
    }
    ctx.W.Header().Set("Content-Type", "text/plain; charset=utf-8")
    ctx.W.WriteHeader(http.StatusOK)
    _, err := ctx.W.Write([]byte(c.Message))
    return err
}

// init 在包被导入时自动注册本插件。
func init() {
    if err := plugin.RegisterHandler(actionType, func(deps *plugin.Dependencies) plugin.Handler {
        return &handler{deps: deps}
    }); err != nil {
        log.Error().Err(err).Str("plugin", actionType).Msg("register failed")
    }
}

② 你自己的 main

go
package main

import (
    "log"

    "github.com/jamesleeon/LiteGate/pkg/litegate"

    _ "my-litegate/greeting" // ← 空导入即触发 init() 注册
)

func main() {
    if err := litegate.Run(); err != nil {
        log.Fatalf("litegate exited: %v", err)
    }
}

litegate.Run() 复用 LiteGate 全部 CLI 旗标与启动逻辑，也可传选项 litegate.Run(litegate.WithConfigPath("custom.yaml"))。

③ 配置：YAML 里直接写自定义键

action 下任何未匹配命名字段的键会被自动收进插件配置，由 ActionConfig 读取。

yaml
routes:
  - name: "demo"

    action:

      message: "hi"            # ← cfg.Get("message") / cfg.Unmarshal 可读

4公共契约速览（pkg/plugin）

这些都是窄接口 / 窄 DTO，刻意不暴露内部重型类型——核心因此能自由演进而不破坏你的插件。

go
// 三类行为接口
type Handler interface {
    Execute(ctx *Context, cfg ActionConfig) error
}
type MiddlewareHandler interface {
    Wrap(next http.Handler, site SiteConfig, name string, config map[string]string) http.Handler
    Stage() MiddlewareStage
    Priority() int
}
type LoadBalancer interface {
    Next(endpoints []Endpoint, key string) Endpoint
}

// 请求上下文（插件能看到的全部字段）
type Context struct {
    W             http.ResponseWriter
    Req           *http.Request
    Site          string
    MatchedPrefix string
    ClientIP      string
}

// 只读配置视图
type ActionConfig interface {
    Type() string                 // action.type
    Get(key string) (any, bool)   // 取单个原始值
    Unmarshal(target any) error   // 解码到你的 struct（按 json tag）
}

// 工厂收到的宿主依赖（全是窄接口）
type Dependencies struct {
    Logger    *zerolog.Logger
    Discovery DiscoveryClient // GetEndpoints(ctx, name, ns, tag) ([]string, error)
}

// 可选生命周期接口（仅 Action 全支持；Middleware/LB 支持 Provisioner+CleanerUpper）
type Provisioner  interface { Provision(deps *Dependencies) error } // 首次实例化，全局初始化
type Validator    interface { Validate(cfg ActionConfig) error }    // 配置加载期校验，fail-closed
type CleanerUpper interface { Cleanup() error }                     // 优雅释放资源
读配置方式说明
cfg.Type()返回 action.type
cfg.Get(key)取单个原始值，务必类型断言并处理失败
cfg.Unmarshal(&struct)推荐：按 json tag 解码，省去逐字段断言

5Middleware 插件

中间件在请求链上加工，再调 next 交给下一环。初始化放在工厂函数里（Wrap 只在构建期调用一次，返回的 http.Handler 才是每请求执行）。

go
type myMW struct{ deps *plugin.Dependencies }

func (h *myMW) Wrap(next http.Handler, site plugin.SiteConfig, name string, config map[string]string) http.Handler {
    threshold := config["threshold"] // 配置是 map[string]string
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // ...前置处理（可用 site.Domain() 等）...
        next.ServeHTTP(w, r) // 别忘了调 next，否则链断了
    })
}
func (h *myMW) Stage() plugin.MiddlewareStage { return plugin.StageAuth }
func (h *myMW) Priority() int                 { return 500 }

func init() {
    _ = plugin.RegisterMiddleware("my_mw", func(deps *plugin.Dependencies) plugin.MiddlewareHandler {
        return &myMW{deps: deps}
    })
}

Stage（粗排序）+ Priority（同档细排序）共同决定执行顺序，与配置里的声明顺序无关。同一 Stage 内 Priority 越大越靠外、越先执行。

Stage次序典型用途内置参照
StagePreSecurity最先CORS 预检等需先于安全的cors = 1000
StageAuth次之鉴权、IP 限制jwt_auth = 800 · api_key = 700
StageRateLimit再次限流、WAF、熔断ratelimit = 300
StageRoute最后（最贴近业务）改写、压缩、缓存strip_prefix = 100

配置里启用：在站点 / 路由的中间件列表中声明 my_mw 即可。Middleware 可选实现 Provisioner / CleanerUpper 参与生命周期托管。

6LoadBalancer 插件

自定义上游选择策略。Next 在每次选上游时被调用（热路径），必须线程安全、不可阻塞。

go
type gpuLB struct{}

// Next 从给定端点里选一个返回；必须返回入参切片中的某个元素。
func (l *gpuLB) Next(endpoints []plugin.Endpoint, key string) plugin.Endpoint {
    var best plugin.Endpoint
    for _, ep := range endpoints {
        if !ep.Healthy() {
            continue
        }
        if best == nil || ep.ActiveRequests() < best.ActiveRequests() {
            best = ep
        }
    }
    return best // 无可用时返回 nil，宿主会回退到 round_robin
}

func init() {
    _ = plugin.RegisterLoadBalancer("gpu_aware", func() plugin.LoadBalancer {
        return &gpuLB{} // 注意：LB 工厂不收 Dependencies
    })
}

配置：action.lb_policy: "gpu_aware"。

三条铁律

① 必须返回入参 endpoints 中的某个元素（宿主据此映射回真实端点），返回 nil = 放弃，宿主回退到 round_robin；② 别把 Endpoint 引用缓存到结构体字段——它只在本次调用有效；③ Next 在热路径，别做阻塞 I/O。

7IDS 身份治理插件

IDS 引擎处理一个请求时，需要一份凭证 + 分流数据（CredentialResponse）：身份、要注入的头、限流提示，以及动态路由用的 selector / meta。默认这份数据是网关发 HTTP 去远端鉴权服务回源得到的；插件让你用进程内 Go 代码直接产出同一个结构——引擎拿到后照常走（缓存、选路注入、限流不变）。

三个可插拔接缝（读 / 写 / 控制）

凭证 Provider（读，校验/获取）· 会话 Issuer（写，签发）· 模式处理器（控制，拦截生命周期）。前两者各自独立选择、各自默认内置 "http"、按路由生效。IDS 本身的使用见 IDS 开发指南。

凭证 Provider（读 / 校验）

实现 CredentialProvider，自行决定数据来源（本地规则 / DB / Redis / 甚至自己发 HTTP），返回 *plugin.CredentialResponse。在 openapi / verify 模式下，Auth 返回的是网关做本地验签所需的「密钥 + 签名方法 + 算法」，而不是身份字段——网关据此在进程内对请求做 HMAC/MD5 验签，密钥永不下发到下游。

go
func (p *myProvider) GetCredentials(ctx context.Context, req *plugin.CredentialRequestContext) (*plugin.CredentialResponse, error) {
    if req.Token == "" {
        // 查无此凭证 → 负缓存（核心短期缓存，挡住重复打穿）
        return &plugin.CredentialResponse{NotFound: true}, nil
    }
    // 按 req.AppID 取出该应用的验签密钥（来自你的 DB / 配置 / Redis）
    appSecret := p.lookupSecret(req.AppID)
    return &plugin.CredentialResponse{
        // Auth = 本地验签凭据：密钥 + 方法 + 算法（openapi/verify 模式核心据此验签）
        Auth: map[string]string{
            "secret":    appSecret,  // 验签密钥（也认 "secret_key" / "key"）
            "type":      "hmac",     // 签名方法；留空回退 config.sign_method
            "algorithm": "sha256",   // 签名算法
            "ttl":       "300",      // 本条凭证缓存秒数
        },
        Route: plugin.RouteNode{
            Selector: map[string]string{"tenant": "corp_abc"}, // ← 动态路由
            Meta:     map[string]string{"gray": "true"},
        },
        Head: map[string]string{"X-Tenant-Id": "corp_abc"}, // 注入下游
    }, nil
}

func init() {
    _ = plugin.RegisterProvider("myids", func(deps *plugin.Dependencies) (plugin.CredentialProvider, error) {
        return &myProvider{deps: deps}, nil
    })
}
Auth 字段的真正用途

在 openapi / verify 模式下，Auth 返回的是网关做本地签名校验所需的凭据，而非身份本身：secret（验签密钥，核心依次回退认 secret_key / key）、type（签名方法，如 hmac / md5，留空回退 config.sign_method）、algorithm（算法，如 sha256）。ttl（秒）控制本条凭证缓存时长。网关据此对请求做验签，密钥只在网关内使用、永不注入下游；要传给下游的身份信息请放 Head。

配置（按路由选）：路由 ids.ids_provider: "myids"，留空 = 默认 http。不同路由可以用不同 provider。

会话 Issuer（写 / 签发）

签发（登录换 token）是另一条路，实现 SessionIssuer，配置用 ids.ids_issuer（与 provider 独立）。

go
func (i *myIssuer) Authenticate(ctx context.Context, r *http.Request, cfg plugin.SessionIssuerConfig) (*plugin.CredentialResponse, *http.Response, error) {
    // 进程内签发：算出凭证直接返回，*http.Response 给 nil（核心已 nil 守卫）
    return &plugin.CredentialResponse{
        Auth: map[string]string{"session": "...", "ttl": "7200"},
    }, nil, nil
}

func init() {
    _ = plugin.RegisterSessionIssuer("myids", func(deps *plugin.Dependencies) (plugin.SessionIssuer, error) {
        return &myIssuer{}, nil
    })
}
读写解耦

只做校验的插件，只注册 CredentialProvider、只配 ids_provider 即可，ids_issuer 留空自动用内置 http——不会因为没实现签发而启动失败。

模式处理器（控制 / 拦截）

想完全控制请求生命周期劫持（前置安全校验、直接截断、转换下游返回），实现 IDSModeProcessor。

go
func (p *MyMode) ModeName() string { return "my-mode" }

func (p *MyMode) ProcessRequest(w http.ResponseWriter, r *http.Request, ctx context.Context, config *plugin.IDSConfig) (context.Context, bool) {
    if r.Header.Get("X-My-Secure-Token") == "" {
        w.WriteHeader(http.StatusForbidden)
        _, _ = w.Write([]byte("Forbidden"))
        return ctx, false // 截断请求
    }
    return ctx, true // 放行
}

func (p *MyMode) WrapResponse(w http.ResponseWriter, r *http.Request, rw http.ResponseWriter, config *plugin.IDSConfig) http.ResponseWriter {
    return rw // 默认不劫持响应
}

func init() {
    _ = plugin.RegisterIDSModeProcessor(&MyMode{})
    // 覆盖内置模式（verify/issue…）须显式：plugin.ReplaceIDSModeProcessor(&MyVerify{})
}

配置：路由 ids.mode: "my-mode" 即激活。

为什么这套很干净

CredentialResponse / CredentialRequestContext 是纯数据 + stdlib http.Request，不引用任何内部类型——所以内置代码直接采用了这套公共类型，内置 "http" 回源也注册进同一张公共注册表。第三方插件与内置回源走完全相同的契约与分发路径。

8DNS Provider 插件（证书 DNS-01 + DDNS）

接管两件事：ACME 证书签发的 DNS-01 质询（在 _acme-challenge 上增删 TXT 记录）与 动态 DNS（DDNS）同步（把本机公网 IP 写到域名的 A/AAAA 记录）。用于在内置服务商（阿里云 / Cloudflare / 腾讯云 / 华为云）之外接入任意 DNS。

一个名字、两种能力（按需各自实现）

都建立在 marker 接口 DNSProvider（只有 GetName()）之上：证书 DNS-01 → 实现 CertDNSProvider（AddTXTRecord / DeleteTXTRecord）；DDNS 同步 → 实现 DDNSProvider（Sync）。只做证书就只实现前者，只做 DDNS 就只实现后者，两个都要就都实现。核心在用到时按能力做断言——缺哪个能力就用哪个能力时报明确 error，而内置服务商与第三方插件走同一张公共注册表与同一套断言路径。

接口契约（pkg/plugin/dns.go）

go
// marker：所有 DNS 插件的基接口
type DNSProvider interface { GetName() string }

// 能力一：ACME DNS-01 质询（证书签发/续期）
type CertDNSProvider interface {
    DNSProvider
    AddTXTRecord(ctx context.Context, domain, token, keyAuth string) error
    DeleteTXTRecord(ctx context.Context, domain, token, keyAuth string) error
}

// 能力二：动态 DNS（A/AAAA 同步）
type DDNSProvider interface {
    DNSProvider
    Sync(ctx context.Context, domain string, ip string) error
}

// 可选接口（实现才生效）：自定义传播等待 / 超时 / 记录同步路由
type DNSPropagationChecker interface { WaitForPropagation(ctx context.Context, domain, value string) error }
type DNSOptionalTimeout   interface { GetTimeout() (totalTime, checkInterval time.Duration) }
type DNSSyncRecordRouter  interface { SyncRecord(ctx context.Context, domain, recordType, value string) error }

① 实现 + 注册

下面是同时实现证书 DNS-01 与 DDNS 的最小骨架；只需要一种能力时删掉另一组方法即可。

go
const providerName = "mydns" // 写在 YAML dns_providers[].type，不能与内置名重名

type provider struct{ apiKey, apiSecret string }

func (p *provider) GetName() string { return providerName }

// 能力一：证书 DNS-01（在 _acme-challenge.<domain> 上写值为 keyAuth 的 TXT）
func (p *provider) AddTXTRecord(ctx context.Context, domain, token, keyAuth string) error {
    return p.upsertTXT(ctx, "_acme-challenge."+domain, keyAuth)
}
func (p *provider) DeleteTXTRecord(ctx context.Context, domain, token, keyAuth string) error {
    return p.removeTXT(ctx, "_acme-challenge."+domain, keyAuth)
}

// 能力二：DDNS（记录类型由 ip 是 IPv4/IPv6 决定）
func (p *provider) Sync(ctx context.Context, domain, ip string) error {
    return p.upsertAddr(ctx, domain, ip)
}

func init() {
    // 注意：DNS 工厂只收 config map[string]string，不收 *plugin.Dependencies
    err := plugin.RegisterDNSProvider(providerName, func(config map[string]string) (plugin.DNSProvider, error) {
        if config["api_key"] == "" {
            return nil, fmt.Errorf("%s: api_key required", providerName)
        }
        return &provider{apiKey: config["api_key"], apiSecret: config["api_secret"]}, nil
    })
    if err != nil {
        log.Error().Err(err).Str("plugin", providerName).Msg("register DNS provider failed")
    }
}

② 配置：在 auto_cert.dns_providers 里引用

yaml
auto_cert:
  enabled: true
  email: admin@example.com
  dns_providers:
    - name: my-cloud              # 逻辑名（被 domains 路由引用，可与 type 不同）
      type: mydns                 # ← 你注册的插件名
      enabled: true
      domains: ["*.example.com"]  # 该 provider 负责签发证书的域名（DNS-01）
      config:                     # ← 整个 map 原样传给工厂的 config 参数（值均为 string）
        api_key: "AKxxxx"
        api_secret: "yyyy"
        propagation_wait: "60s"   # 可选：DNS-01 传播等待
      ddns:                       # 可选：开启 DDNS 同步
        enabled: true
        sync_a: true
        domains: ["home.example.com"]
几个关键约束

① 工厂不收 Dependencies——没有 Logger / Discovery 注入，只拿 config map[string]string（值都是 string，时长写 "60s" 自己解析）；② 内置名不可覆盖：aliyun / cloudflare / tencent / tencentcloud / huawei / huaweicloud；③ 通配符证书（*.example.com）必须用 DNS Provider，HTTP-01 不支持；④ 必须尊重 ctx 取消（关机/超时），别用 context.Background()。

两条写记录路径

DDNS 周期任务（dns_providers[].ddns）直接调 DDNSProvider.Sync(ctx, domain, ip)，记录类型由 IP 版本决定；而证书管理器的记录同步会优先 DNSSyncRecordRouter.SyncRecord（拿到现成 recordType）再回退 Sync。需要精细控制记录类型时实现 DNSSyncRecordRouter。

9编译与分发

方式 1：litegate build --with（内置一键构建，推荐）

LiteGate 内置一键构建命令行工具，自动生成 main.go + go.mod、拉取依赖、pin 版本、编译。

bash
litegate build --with github.com/acme/litegate-llm -o my-litegate

# 本地联调：把 LiteGate 依赖替换为本地源码树
litegate build --with ./greeting --replace=../LiteGate -o my-litegate
# 或设置环境变量 LITEGATE_LOCAL_PATH=../LiteGate
  • 依赖本地 Go 工具链（自动推导适配当前 Go 版本）+ 网络能访问 Go module 代理。
  • 可重复 --with 叠加多个插件；-o/--output 指定产物路径。
dev 版本必须配 --replace

当 LiteGate 跑的是开发版本（版本号含 -dev）且不在 LiteGate 模块目录内时，module 代理上不存在该版本，litegate build 会要求你显式 --replace=<本地路径> 或设 LITEGATE_LOCAL_PATH。

方式 2：用户拥有 main（原生 go build）

如第 3 节，自建 Go 模块，主函数空导入插件包并调 litegate.Run()，用原生 go build 编译。版本通过 go.mod 锁定，本地开发用 replace 替换。模板见仓库 examples/plugin-template。

不支持动态加载

没有 .so 热插拔。插件必须编译进二进制——这是为零运行时开销做的有意取舍。

10生命周期全景

text
程序启动
  └─ 各插件包 init() → Register{Handler,Middleware,LoadBalancer,Provider,DNSProvider,...}
配置加载 / 热重载
  └─ loader 逐路由校验 → Validator.Validate(cfg)        [仅 Action；fail-closed：错则起不来]
首个请求命中该类型（懒加载，singleflight 去重）
  └─ 工厂 factory(deps) → Provisioner.Provision(deps)   [一次，全局初始化，拿不到路由配置]
每个请求
  └─ Handler.Execute() / 中间件链 / LB.Next()           [高并发，类型级共享实例]
关机 / 热重载卸载
  └─ CleanerUpper.Cleanup()                             [一次，优雅释放]

全链路 panic 都有 recover 兜底：工厂 / Provision / Validate / 插件 Execute 的 panic 都被转成 error 或 500，不会拖垮进程。但别依赖它当正常流程。

11踩坑清单

  1. Handler / LB 是跨路由的类型级单例。所有用该类型的路由共享一个实例并发调用。每路由不同的配置必须在 Execute 里从 cfg 现读，不能缓存到结构体字段，否则路由间互相覆盖。
  2. 不能覆盖内置类型。RegisterHandler("proxy", …) / RegisterMiddleware("cors", …) / RegisterLoadBalancer("round_robin", …) 都返回 error。取个不冲突的名字。（IDS 内置 "http" 要覆盖须用 ReplaceProvider / ReplaceSessionIssuer。）
  3. Validate 是 fail-closed（仅 Action）。任一路由配置非法都会让网关启动/重载失败——这是对的。把配置校验尽量放进 Validate。
  4. Provision 拿不到路由配置。只做与路由无关的全局初始化；需要路由配置的逻辑放 Execute。
  5. 配置值类型要小心。YAML 数字进来是 int/float64，不是 string。优先 cfg.Unmarshal(&struct)。
  6. 依赖最小集。插件只拿到 Logger 和 Discovery（窄接口）。需要更多能力应推动在 pkg/plugin 追加窄接口，不暴露内部具体类型。
  7. IDS：别自己做缓存。核心已在 provider 调用外层做了 L1(LRU)+L2(Redis)+负缓存+singleflight。你只管返回裸数据。缓存时长用 Auth["ttl"]（秒）。
  8. IDS：fail-closed 默认。返回 error 时核心按 fail_policy（默认 deny）处理；鉴权失败应返回 error 或 NotFound，别静默放行。
  9. 热重载下别泄露 goroutine。在 Provision 起的后台 goroutine 必须监听宿主取消信号，并在 Cleanup 里终止，否则多次重载累积泄露。

12测试你的插件

你自己的仓库里可直接对 pkg/plugin 的接口做单元测试（构造 plugin.Context 与一个实现 ActionConfig 的桩），无需启动整个网关。

  1. 单元测试：构造 *plugin.Context（用 httptest.NewRecorder() + httptest.NewRequest()）与一个 ActionConfig 桩，直接调 Execute 断言响应。
  2. 注册校验：断言 RegisterHandler 返回 nil，重名 / 覆盖内置返回 error。
  3. 端到端：参考仓库 internal/action/plugin_external_test.go，三类插件各有一个公共 API 的端到端用例。
  4. 配置测试：litegate -t -config config.yaml 验证配置语法 + 触发 Action 的 Validate。

13常见问题

能动态加载 .so 插件吗？

不能。LiteGate 用编译期注册表，插件必须编译进二进制。这是为零运行时开销与跨平台稳定性（Go plugin 在 Windows 不可用）做的有意取舍。

插件能拿到哪些宿主能力？

第三方只拿到 Dependencies{ Logger, Discovery }——日志 + 服务发现窄视图。需要 Redis / 其它存储请插件自带；需要更多宿主能力，应在 pkg/plugin 追加窄接口。

内置功能和我的插件有区别吗？

走同一张注册表分发，但能力不对等：内置 Action 走“宿主内部捷径”（直接访问 Executor 私有状态），第三方走公共 Dependencies 契约。这是性能与封装的有意权衡。

插件 panic 会不会搞垮网关？

不会。工厂 / Provision / Validate / 插件 Execute 都被宿主 recover 成 error 或 500。但请自己处理错误，别把 recover 当正常流程。

同一个名字能同时是 Action 和 Middleware 吗？

可以——它们是各自独立的注册表，名字空间不冲突。但内部 Register* 会反查外部注册表防止内外同表重名。
