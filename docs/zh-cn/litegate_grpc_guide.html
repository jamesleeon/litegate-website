1一键启用：proto: grpc

在 proxy action 上声明上游协议为 gRPC，网关即按 gRPC 语义转发，并保留 HTTP/2 trailer：

sites/grpc.yaml
domain: grpc.example.com
routes:
  - name: greeter
    match:
      path_prefix: /
    action:
      type: proxy
      service_name: greeter-svc   # 也可用 upstream: host:port
      proto: grpc                 # 明文后端用 grpc；TLS 后端用 grpcs

用服务发现标签同样可以一键开启（打标即上线）：

service meta
litegate.ingress.host   = grpc.example.com
litegate.upstream.proto = grpc

2h2c 明文与 TLS 自动二分 · mTLS 自动升级

gRPC 后端有两种形态：明文 HTTP/2（h2c）和 TLS（grpcs）。LiteGate 自动区分：

  • proto: grpc → 明文 h2c：通过独立的 http2.Transport（AllowHTTP + 裸 TCP DialTLS）转发，不会误用 TLS。
  • proto: grpcs → TLS gRPC。
  • 当该路由启用了 mTLS（Litemesh），grpc 会自动升级为 grpcs，无需手改配置。
trailer 透传

网关原生透传 gRPC trailer：grpc-status / grpc-message / grpc-status-details-bin，并保留 TE: trailers。这是 gRPC 正确工作的前提，很多"凑合转 HTTP/2"的方案会丢 trailer 导致客户端报错。

3原生健康检查（grpc.health.v1）

对 gRPC 后端，TCP 探活只能确认"端口开着"，确认不了 gRPC 服务真的在 Serving。LiteGate 的主动健康检查支持直接调用标准 grpc.health.v1.Health/Check（h2c 与 grpcs 双客户端），识别 SERVING / NOT_SERVING，让负载均衡的摘除更准确。

真探活，不是假探活

这意味着一个 gRPC 进程"端口活着但服务未就绪"时会被正确判为不健康并摘流，而不是继续把请求打进去。

4gRPC-Web 转码

浏览器不能直接说标准 gRPC（HTTP/2 二进制帧），只能用 gRPC-Web。LiteGate 在边缘把 application/grpc-web 与 application/grpc-web-text（base64 分帧）转码成标准 gRPC 转发给后端，响应再反向转回——前端因此可以直接调用 gRPC 后端，无需另起 Envoy。

5服务反射

支持 gRPC 服务反射自动路由，grpcurl / Postman 这类工具可以透过网关发现后端的服务与方法，排障和联调体验大幅提升，无需手工导入 proto。

6超时传播与统一错误映射

gRPC 用 grpc-timeout 头传递调用截止时间。网关会解析并向上游传播剩余 deadline；当超时触发时，返回的是规范的 grpc-status: 4（DEADLINE_EXCEEDED），而不是裸的 HTTP 504。

所有错误路径（上游不可达、拨号失败、鉴权拒绝等）都统一映射为 grpc-status，客户端拿到的是标准 gRPC 状态码，而非半截 HTTP 错误：

场景grpc-status含义
上游不可达 / 拨号失败14UNAVAILABLE
网关超时4DEADLINE_EXCEEDED
鉴权拒绝16UNAUTHENTICATED
内部错误13INTERNAL
