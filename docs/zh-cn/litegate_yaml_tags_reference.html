场景一：前后端分离/微服务混部场景 (SPA 静态托管 + API 动态代理 - 混合路由架构)

混合路由合并场景
这是 LiteGate 最强大的核心能力之一：混合路由合并 (Hybrid Merge)。在这一典型生产场景中，前端静态 SPA 页面（如 index.html 等）完全由网关的静态 YAML 配置（使用 type: serve 动作）进行高效的本地文件系统托管；而后端业务 API 微服务（如 order-service）的动态选路规则，则完全由微服务开发实例在启动时自动上报的动态自注册标签（Tags）驱动。网关会在内存中自动将两者合成为一套统一的选路树，既保障了网络边缘的安全与管控，又赋予了开发团队极度敏捷的微服务自注册发布能力。

静态托管网关 YAML 配置 (frontend-only.yaml)

domain: app.example.com
port: 80              # 💡 提示：静态站点的全局默认监听端口
force_https: true

routes:
  # 1. 静态托管路由 (静态 YAML 专属，动态实例禁止声明此类物理路径)
  - name: frontend-spa
    priority: 10
    match:
      path_prefix: /
    action:
      type: serve
      root: /var/www/dist
      index: index.html
      spa: true
      # 💡 可选：启用 Serve KV 自动化同步模式
      # kv_mode: true
      # kv_provider: litemesh
      # kv_key: litegate/config/front/main
      # keep_versions: 3

  # 💡 提示：微服务 API 动态代理路由无需在此声明！
  # 后端业务微服务启动时，会自动将自身的 IP、端口及注册路由 Tags 上报到注册中心。
  # 网关会自动感知并与这里的顶级 domain 节点进行 Hybrid Merge，实现热插拔 of 微服务路由。

微服务自注册动态标签写法 (order-service 节点 Tags)

# ── 1. 路由 Ingress 与匹配 ──
# 网关接收到包含 app.example.com 的请求时 (触发 Hybrid Merge 并开始在内存中检索)
litegate.ingress.host=app.example.com
# 声明路由前缀匹配。只有当请求路径前缀为 /api 时，才会将请求分发给该实例组
litegate.ingress.path_prefix=/api
# 提升优先级至 100，确保高优先匹配，防止被默认的前端 SPA 静态路由 "/" 覆盖
litegate.router.priority=100

# ── 2. 报文改写与路径重写 ──
# 剥离前缀。在转发至后端前剥离 /api 路径，使微服务内接到的实际路径为 /orders 而非 /api/orders
litegate.router.strip_prefix=/api

# ── 3. 负载均衡与健康调度 ──
# 实例权重。在负载均衡调度时，该物理实例承担 80% 的流量分发
litegate.upstream.weight=80

💡 混合路由合并机制 (Hybrid Merge)：这是 LiteGate 极为核心的架构特点。网关管理员仅需在全局静态 YAML 中维护域名绑定、SSL 证书等基础设施节点，微服务开发者则在其自身的物理节点 Tags 中直接声明 API 选路特征。网关运行时会自动将两者合成为一套统一的路由树，既保障了安全，又排除了传统网关繁琐 of 集中配置变更漏斗。

场景二：微服务 API 动态网关场景 (多实例动态代理 + 局部限流与熔断保护)

API Gateway 场景
这是 LiteGate 作为 纯粹的微服务 API 动态网关 最常见的部署形态。网关在这一场景下不涉及任何本地静态 resource 托管，而是专注于承载 high 并发的七层反向代理流量。它通过整合 QPS 速率限制（支持自适应 Adaptive 算法）、请求超时控制与重试自愈机制，以及 后端物理实例级的断路熔断保护，为下游微服务构建起一套坚固的分布式主动防御层，能有效隔离并拦截底层服务雪崩引发的 system 连锁瘫痪。

网关静态 YAML 配置 (pure-api-proxy.yaml)

domain: api.example.com
port: 443

routes:
  - name: api-gateway-route
    priority: 50
    match:
      path_prefix: /v1
    action:
      type: proxy
      service_name: payment-service
      timeout: 5             # 💡 YAML 侧为纯整数秒，不带单位
      retry: 3
      rate_limit: 150
      rate_limit_policy: adaptive
      circuit_breaker:
        enabled: true
        timeout: 30          # 💡 纯整数秒
        max_requests: 10
        trip_threshold: 15

微服务自注册动态标签写法 (payment-service 节点 Tags)

# ── 1. 路由 Ingress 与匹配 ──
litegate.ingress.host=api.example.com
litegate.ingress.path_prefix=/v1
litegate.router.priority=50

# ── 2. L5 Service 网络级超时与重试 ──
# 等待上游服务响应的最大超时时间
litegate.service.timeout=5s
# 遇到网络闪退或 5xx 错误时的最大重试尝试次数
litegate.service.retry=3

# ── 3. L5 Service 熔断保护配置 ──
# ⚠️ 标签端为平铺式 breaker_*，且 timeout 为纯整数秒（无 s 单位）
litegate.service.breaker=true
litegate.service.breaker_timeout=30
litegate.service.breaker_max_requests=10
litegate.service.breaker_trip_threshold=15

# ── 4. L4 Middleware 限流配置 ──
litegate.middleware.ratelimit=150
# 💡 注: rate_limit_policy (adaptive) 为静态 YAML 专属，动态标签暂不支持；标签端仅按硬限流 QPS 处理

💡 高可用网关防护：本场景展示了网关作为核心屏障的威力。通过在动态标签中简单声明 litegate.service.breaker=true 和限流规则，微服务便能在不需要侵入任何客户端或服务端 SDK 代码的前提下，获得高弹性的分布式限流与自动断路切流能力。

场景三：业务级单点登录与鉴权上下文注入场景 (IDS Session)

统一身份拦截场景
面向 B 端或 C 端需要高安全 Session 验证与 Claims 上下文注入的复杂业务系统。网关在这一场景下挂载 IDS 统一身份与安全认证引擎，从入站请求中自动提取 Session 会话凭证，与后置的 Redis 共享缓存进行高效校验。网关不仅提供异步滑动窗口式自动续期（Session Renewal）机制，还能在校验通过后，将用户主体信息（Claims）解密并自动作为标准的 X-Claim-* 请求头注入到下游请求中，使后端微服务免去繁琐的重复身份验证代码。

静态网关配置 (internal-session-auth.yaml)

domain: app.example.com
port: 443

routes:
  - name: internal-auth-api
    priority: 200
    match:
      path_prefix: /api/v1
    action:
      type: proxy
      service_name: biz-service
      ids:
        enabled: true
        mode: verify          # 💡 会话校验走 verify 处理器；ids_session 是 Provider 的 type，不是 mode
        provider: saas-auth
        extractor: "{header.Authorization}"
        session_ttl: 7200
        renew_threshold: 3600
        inject_headers: true
        inject_claims:
          X-User-Id: sub
          X-User-Email: email

微服务自注册动态标签写法 (biz-service 节点 Tags)

# ── 1. 路由 Ingress 与匹配 ──
litegate.ingress.host=app.example.com
litegate.ingress.path_prefix=/api/v1
litegate.router.priority=200

# ── 2. 中间件安全拦截与提供商绑定 ──
# 启用大一统安全认证控制引擎并指定校验模式为 verify (会话校验)
litegate.middleware.ids.enabled=true
litegate.middleware.ids.mode=verify
# 关联统一的认证授权提供商 saas-auth
litegate.middleware.ids.provider=saas-auth

# ── 3. 动态身份校验提供者 (IDSProvider) 参数配置 ──
# 指定提供商的底层校验模型为 ids_session
litegate.http.ids_providers.saas-auth.type=ids_session
# 从 Authorization 请求头中提取会话凭证（支持 Bearer / UUID 格式自动识别）
litegate.http.ids_providers.saas-auth.extractor={header.Authorization}
# 会话的强制生存时间（秒）
litegate.http.ids_providers.saas-auth.session_ttl=7200
# 当会话剩余时间小于 3600 秒时，触发异步滑动续签，防止用户登录态突然中断
litegate.http.ids_providers.saas-auth.renew_threshold=3600

# ── 4. 身份信息上下文注入 ──
# 💡 注: inject_headers / inject_claims 为静态 YAML 专属，动态标签不支持；
# 会话校验通过后由网关统一完成 Claims 解密并注入 X-Claim-* 请求头

💡 全自愈会话防线：LiteGate 会在边缘层完成 Session 自动校验与按需异步续期。开发团队无需在每个底层 microservice 中单独接入 Redis 客户端并开发身份校验过滤逻辑，不仅简化了代码结构，还规避了由于单个微服务未正确维护 Session 续签所产生的一致性灾难。

场景四：开放平台 MD5 离线签名与动态回源拉取

开放平台核心场景
面向外部商户或开放 API 场景。外部客户端携带密钥（SecretKey）执行离线 MD5 签名校验。网关动态解析 AppId，在 L1 (LRU) / L2 (Redis) 双层缓存防守下，异步对齐或拉取密钥进行高并发验签，同时建立404 负缓存 (Negative Cache) 机制彻底降噪，卸载后端拉取微服务（IDS）高频恶意穿透探针。为了保障回源链路安全，网关在回源时自动挂载 client_id 与 client_secret 双向凭证对齐统一身份中心。

静态网关配置 (open-api-md5.yaml)

domain: openapi.ai.com
port: 443

routes:
  - name: open-orders
    priority: 150
    match:
      path_prefix: /openapi/v1/orders
    action:
      type: proxy
      service_name: openapi-backend
      ids:
        enabled: true
        mode: openapi
        provider: dynamic-openapi
        extractor: "{header._appid}"
        # 挂载远程动态拉取凭证引擎
        discovery_service: identityservice
        path: /api/gate/credentials
        client_id: litegate
        client_secret: secret
        negative_ttl: 60
        fail_policy: deny

微服务自注册动态标签写法 (openapi-backend 节点 Tags)

# ── 1. 动态业务路由及大一统 IDS 引擎注册 ──
# 声明业务 API 路由的匹配路径前缀
litegate.http.routers.open-orders.path=/openapi/v1/orders
# 启用大一统安全认证引擎并设置为 openapi (离线 MD5 验签) 模式
litegate.http.routers.open-orders.ids.enabled=true
litegate.http.routers.open-orders.ids.mode=openapi
# 关联绑定的动态安全认证提供者
litegate.http.routers.open-orders.ids.provider=dynamic-openapi

# ── 2. 动态身份校验提供者 (IDSProvider) 注册 ──
# 声明身份服务提供商类型为 remote_auth (支持集群动态回源发现)
litegate.http.ids_providers.dynamic-openapi.type=remote_auth
# 声明身份验证模式为 openapi
litegate.http.ids_providers.dynamic-openapi.mode=openapi
# 从请求头提取应用标识 AppId 变量
litegate.http.ids_providers.dynamic-openapi.extractor={header._appid}
# 声明身份发放提供者（由网关动态调度后端 identityservice 完成物理回源）
litegate.http.ids_providers.dynamic-openapi.discovery_service=identityservice
# 挂载局部自定义凭证校验接口路径
litegate.http.ids_providers.dynamic-openapi.path=/api/gate/credentials
# 绑定接入身份中心的客户端凭证 (ClientID / ClientSecret)
litegate.http.ids_providers.dynamic-openapi.client_id=litegate
litegate.http.ids_providers.dynamic-openapi.client_secret=secret
# 非法 AppId 404 负缓存强力阻断时间 (秒)
litegate.http.ids_providers.dynamic-openapi.negative_ttl=60

💡 选路提升防护：验签通过后，网关在网关内部安全提取 sid 并自动提升为物理选路键 KeyRouteSelector["sid"] 交互 Litemesh 进行定向分片路由，同时彻底屏蔽向下游的 sid 透传，确保边缘安全。

场景五：单机静态本地反向代理 (Single-node Local Static Proxy - 无服务发现)

单机静态传统场景
这是最纯粹的单机、微型或传统部署场景。在这种部署模式下，网关不依赖 Consul 或 Litemesh 等任何动态服务发现机制，而是完全作为前置代理，直接将所有流量硬编码转发给本地或指定内网 IP 的特定物理端口（如 127.0.0.1:8080）。

网关静态 YAML 配置 (local-static-proxy.yaml)

domain: api.example.com
port: 80

routes:
  - name: local-api-route
    priority: 100
    match:
      path_prefix: /v1
    action:
      type: proxy
      # 强制声明上游类型为静态物理地址列表
      upstream_type: static
      # 直接硬编码指向本地微服务物理监听端口
      upstream:
        - 127.0.0.1:8080

动态服务自注册标签 (Tags)

💡 纯静态编排：无动态标签写法
由于本场景完全不经过任何注册中心，网关仅依赖静态 YAML 的 upstream 进行本地分发，微服务实例无需且无法上报任何动态自注册 Tags，贯彻“理性的动静分离”边界。

💡 免注册极简架构：最适合微型业务或单机 Docker-Compose 极速部署，保持了传统反向代理（如 Nginx 极简反代）原汁原味的心智模型。

场景六：多实例静态负载均衡集群 (Multi-instance Static Load Balancing)

内网集群静态场景
适用于传统内网固定物理机或固定容器 IP 的集群部署。在同样无服务发现的情况下，网关通过在静态 YAML 中直接配置多个固定上游 IP 列表，并在网关七层直接驱动多节点负载均衡分发算法（如轮询、IP Hash），配合网络级超时重试来保障高可用。

网关静态 YAML 配置 (multi-static-lb.yaml)

domain: app.example.com
port: 80

routes:
  - name: app-lb-route
    priority: 100
    match:
      path_prefix: /
    action:
      type: proxy
      upstream_type: static
      # 轮询负载均衡算法 (支持 rr / ip_hash)
      lb_policy: rr
      # 绑定后端多实例静态 IP 与端口
      upstream:
        - 10.0.0.21:8080
        - 10.0.0.22:8080
        - 10.0.0.23:8080

动态服务自注册标签 (Tags) 与负载均衡说明

⚖️ 多节点静态高可用：
在静态负载均衡模式下，LiteGate 内部路由分配器（Dispatcher）会对静态配置的多个 upstream 实例建立主动的 TCP 探活连接。 一旦探测到某个物理节点离线，网关会在内存中自动剔除其健康路由状态，同时将请求无感重试并派发至其他健康存活的后端节点。
💡 纯静态编排：无动态标签写法
由于本场景完全不经过任何注册中心，网关仅依赖静态 YAML 的 upstream 进行本地分发，微服务实例无需且无法上报任何动态自注册 Tags，贯彻“理性的动静分离”边界。

💡 免注册集群架构：适用于不方便部署 Consul 动态服务发现的传统企业内网、隔离的专网物理集群，或者是固定的虚拟机环境，以极简的方式提供高可用保障。

场景七：WebDAV 静态托管 + 内置简易 SSO 认证 (WebDAV + SSO Auth)

静态存储防卫场景
面向团队内网文件共享或个人私有存储场景。LiteGate 提供了开箱即用的静态 WebDAV 文件托管服务，支持可视化网页浏览界面与文件读写限制。为了保证文件安全，我们直接在网关路由挂载内置的简易 SSO 鉴权组件 (Simple SSO)。用户无需申请复杂的 OIDC 服务证书，仅需 4 行配置即可保护私有文件系统，并允许渲染完全自定义的本地登录模板。

网关静态 YAML 配置 (webdav-sso.yaml)

domain: storage.example.com
port: 443
force_https: true

routes:
  - name: secure-webdav
    priority: 100
    match:
      path_prefix: /dav
    action:
      type: webdav
      root: /data/share     # 本地物理存储路径
      web_ui: true          # 开启图形化网页界面
      read_only: false      # 允许文件上传与删除修改
      
      # 挂载内置 SSO 简易安全防御
      auth:
        type: sso
        sso_enabled: true
        secret: "please_change_to_a_random_secret_at_least_32_bytes_long" # 会话 Cookie 签名密钥 (生产必须使用随机且不小于 32 字节的强密钥)
        template: /var/litegate/login.html # 自定义本地登录页面
        users:
          admin: "8c6976e5b5410415bde908bd4dee15dfb167a9c873fc4bb8a81f6f2ab448a918" # admin123 密文

动态自注册标签 (Tags) 与命令行辅助说明

🛠️ 密码哈希生成器命令行：
为了防止配置泄露和明文落盘风险，LiteGate 内置了强密码加盐哈希保护。切勿在 users 下写入明文。 在终端执行： litegate --hash "your_password" 控制台将直接输出高强度 SHA256 / Bcrypt 防御密文，直接贴入配置文件即可。
💡 静态专属防护说明：
WebDAV 的 root、web_ui、read_only 属性均属于站点级物理存储编排。根据 LiteGate 产品边界设计，动态标签不提供且禁止声明此类本地物理挂载路径，从源头上杜绝多机容器部署时文件系统状态不一致所产生的漏洞与逻辑灾难。

💡 内置 SSO 的降维优势：相比于繁琐地配置外部 OIDC 身份中心，LiteGate 的内置 SSO 仅需数行配置，便能实现企业级的拦截防卫与 Cookie 滑动续订机制，对于团队协作及内网存储项目来说极其便携。

场景八：老域名 301 永久平移 + 老系统根路径 302 重定向修正 (Legacy System Steering & Domain Migration)

老系统自愈与迁移场景
在实际生产部署中，经常需要对老系统进行新老域名无感平移（如老域名 sales.old-example.com 迁移到 a.example.com）。同时，由于老系统早期路由没有写好，用户直接访问根路径 / 时往往无法打开页面，需要在网关物理入口层强制 302 重定向到子系统 /sys 入口，其他流量则继续原样反代。LiteGate 提供了高可读性的重定向指令与模板变量替换机制，免去了繁琐晦涩的 Nginx 正则重写配置。

文件 1：新域名配置 (a.example.yaml - 路由自愈反代)

domain: a.example.com
port: 80              # 💡 可选。不配置默认也是 80/443
force_https: true

routes:
  # ── 1. 修复老系统路由未写好问题：根路径精确匹配，302 跳转至子系统 ──
  - name: root-to-sys-redirect
    priority: 100     # 💡 高优先级，确保在模糊匹配 "/" 前精准截获
    match:
      path: /         # 精确匹配 "/"
    action:
      type: redirect
      location: /sys  # 302 强制重定向到真实的子应用入口
      status: 302

  # ── 2. 将其他所有子路径流量反向代理转发至老物理机 ──
  - name: legacy-system-proxy
    priority: 10      # 较低优先级
    match:
      path_prefix: /  # LPM 前缀模糊匹配其他所有流量 (如 /sys/login)
    action:
      type: proxy
      upstream_type: static
      upstream:
        - "http://10.0.0.10:8080"

文件 2：老域名重定向 (sales.old-domain.yaml - 平滑平移)

domain: sales.old-example.com
port: 80

routes:
  # ── 3. 301 永久重定向到新域名，利用模板变量原样保留路径与参数 ──
  - name: global-domain-migration
    match:
      path_prefix: /
    action:
      type: redirect
      # 💡 完美使用 ReplaceVars 模板变量 {uri}，原样透传路径与参数！
      location: "https://a.example.com{uri}"
      status: 301

老系统自愈与平滑迁移机制说明

🔄 老域名 301 无感平移：
通过在 Location 中使用内置变量 {uri}（例如：https://a.example.com{uri}），网关会自动捕获并原样拼接老域名的后续路径和 Query 查询参数。 例如，当客户端访问 http://sales.old-example.com/sys/login?user=admin 时，会自动永久跳转（301）至 https://a.example.com/sys/login?user=admin，保护老用户历史收藏夹，对搜索引擎 SEO 极其友好。
🛠️ 老系统根路由（LPM）零侵入自愈：
由于老系统应用层路由编写不善，直接访问 / 时会报错或显示空页面。 LiteGate 通过在网关层声明 path: / 并设定 priority: 100 高优先级，率先把访问根目录的流量重定向至 /sys，而其余的流量（如 /sys/login）由于更长的前缀，会被第二条 path_prefix: / 路由精准捕获，依然照常代理给物理机 10.0.0.10:8080，免除了业务应用重构的巨大成本与风险。

💡 免除 Nginx 复杂配置的烦恼：在 Nginx 中，你需要为 301 域名平移和 302 路由修正配置多个 Server 模块并编写晦涩的 Rewrite 正则匹配（易错且难以维护）。LiteGate 的 YAML 直观清晰，所有跳转策略热加载、即刻生效，无需 reload 重启。
