打标即上线:从注册到 HTTPS 的全链路零配置

在 LiteGate 生态里,微服务只需"打标签注册"就能完成从域名解析、路由生成到 HTTPS 证书申请的全链路闭环。无论标签来自 Docker labels、Consul tags、Litemesh/LiteDeploy meta 还是 Discovery agent,网关都用同一套 litegate.* 标签生成入口。

1核心概念:两层标签,职责完全不同

注册一个服务时,你在两个不同的地方贴标签。一层定义"如何识别和路由到我",一层声明"我属于哪个分片/租户"。

层 A · 服务定义标签(贴在 Service Meta)
告诉网关:如何识别我、如何鉴权、如何路由到我。只需贴一次,对该服务所有实例生效。网关读取后生成路由。

litegate.ingress.hostlitegate.ingress.path_prefixlitegate.middleware.authlitegate.router.sidlitegate.service.timeout

↓ 网关读取并生成路由
层 B · 实例筛选标签(贴在每个实例节点的 Meta)
告诉网关:我属于哪个分片 / 租户 / 集群。每个实例按自己所属分组贴,用于运行期 RouteSelector 精准过滤,不生成路由。

sid=cluster-01env=prodtenant=corp_abc

关键区别

层 A 的 litegate.router.sid = cluster-01 是"我要找什么";层 B 的 sid = cluster-01 是"我是什么"。当服务定义写了 router.sid=cluster-01,网关只把流量路由到 sid=cluster-01 的实例,完全跳过其他分片。两类标签共存于同一个 Meta 对象,网关分别处理。详见 后端标签体系。

2五种发现源,同一套标签

后端发现源有 5 种:litemesh、litedeploy、docker、consul、discovery。它们进入 LiteGate 后都会被归一成同一种 ServiceEndpoint,再由同一套 litegate.* 标签驱动路由、上游池、鉴权、IDS、mTLS 和观测。

发现源配置值连接对象更新方式适合场景
Litemeshprovider: litemeshLitemesh AgentSSE 推送已接入 Litemesh、需要 mTLS、状态变化频繁
LiteDeployprovider: litedeployLiteDeploy Agent 或集群地址SSE 推送由 LiteDeploy 管理发布、实例与服务元数据
Dockerprovider: dockerDocker Daemon SocketDocker event stream单机 Docker 或 Swarm,容器 labels 自动接入
Consulprovider: consulConsul Agent / CatalogCatalog 同步已有 Consul 注册中心和健康检查
Discoveryprovider: discoverycamodns_discovery 或兼容 agentBlocking query异构注册中心聚合(Docker/K8s/Nacos/Etcd/Litemesh 并存)
关于 static / external / hybrid

这三个不是业务发现源:static 是固定 upstream(完全不走发现);external 和 hybrid 是内部 upstream_type 名称——discovery 源运行时对应 external,而 Consul 与 Discovery 同时启用时对应 hybrid。

discovery 的确切含义

provider: discovery(对应 upstream_type: external)特指 LiteGate 通过 HTTP blocking query 连接 camodns_discovery 这类兼容 /v1/catalog/services 与 /v1/health/service/:service_name 的外部聚合层,把异构注册中心统一成 LiteGate 能理解的服务目录。它不是第 6 种业务发现源。详见 服务发现源总览。

3Magic Ingress:Tag & Forget

传统上线一个服务要部署、配路由、买域名、配公网 DNS、申请证书、配内网 DNS——六步。Magic Ingress 下你只做一步:部署带标签的服务实例,例如 litegate.ingress.host=order.example.com,其余全自动。

唯一人工步骤

服务注册并打标

用 Litemesh SDK 启动(或 Sidecar / Docker label / Consul 注册)时,带上 litegate.ingress.host 等魔术标签。

发现

CatalogLoader 实时监听

瞬间检测到新服务,解析魔术标签,在内存中即时生成虚拟 SiteConfig,更新 Router 立即接管该域名流量。

公网连通

DDNS 自动解析

探测网关公网出口 IP,调用 DNS 服务商 API(阿里云 / Cloudflare 等)把域名 A 记录指向网关公网 IP。

证书

AutoCert 自动签发

检查 Litemesh KV 是否已有有效证书;若无则获取集群分布式锁,从 Let's Encrypt 申请,成功后保存到集群 KV,所有在线节点瞬间同步加载。

内网连通

Split-Horizon DNS

触发 dns_update Webhook 更新内网 DNS,把域名指向网关内网 IP。内网用户用同一域名、同一合法证书直接访问。

最终成果

零配置维护(不写 nginx.conf / gateway.yaml)、零运维(不手动配解析、不买/传证书、不处理续期)、内外网一致体验(同一域名、同一 HTTPS)。证书细节见 证书生命周期。

4注册即上线:预期时间线

以 payment 微服务携带 litegate.ingress.host 标签注册为例,从调用 SDK 到 HTTPS 全面就绪的真实时序:

时间点事件状态
T + 0s服务调用 SDK 完成注册🚀 开始
T + 0.5sLiteGate 通过 SSE 感知,完成内存路由构建🌐 内网 HTTP 可访
T + 1s内网 DNS(CamoDNS)解析生效📡 域名就绪
T + 60s证书申请完成并在集群内广播🔒 HTTPS 全面就绪
为什么这个架构很"硬"

解耦:业务开发不需要懂网关配置,只需知道自己的域名。高可用:某个 LiteGate 节点挂了,内网 DNS 通过心跳自动切到另一节点。零操作:证书、解析、路由全自动,真正做到 "Architecture as Operations"。

5选路与优先级

当启用物理选路(token_steering)时,网关解析 Token 后的富元数据可以在运行期动态改变路由目标,覆盖服务注册时贴的静态分片。

priority
SaaS / Session 动态返回的 RouteSelector   ← 最高优先级(运行时按身份覆盖)
    ↓ 如不存在,则使用
litegate.router.sid / router.* 静态标签   ← 服务注册时的默认分片

这意味着同一套网关路由规则可以秒级支持多租户弹性物理隔离:登录态决定路由到哪个集群,无须在注册时写死静态 sid。身份决策详见 IDS 开发指南。

6注册示例

consul json
{
  "Name": "lockd",
  "Tags": ["raft", "dist-lock", "v1"],
  "Port": 8080,
  "Meta": {
    "litegate.ingress.host": "lockd.example.com",
    "litegate.middleware.auth": "token_steering",
    "litegate.router.sid": "cluster-01",
    "litegate.service.timeout": "30s",
    "sid": "cluster-01",
    "cluster": "prod"
  }
}
litemesh kv
litegate.ingress.host        lockd.example.com
litegate.middleware.auth     token_steering
litegate.router.sid          cluster-01
litegate.service.timeout     30s
litegate.observability.log   info
sid                          cluster-01   # 实例侧
cluster                      prod         # 实例侧
注意区分

上例中 litegate.* 开头的是服务定义标签(层 A);末尾的 sid / cluster 是实例侧标签(层 B)。两者共存于同一个 Meta,网关分别处理。Docker provider 的实战配置见 Docker 服务发现部署指南。
