证书管理实战：自动申请、集群共享与自带付费证书

这篇文档面向落地配置：你只需要决定证书从哪来——让网关自动向 ACME 申请，还是用你已经买好的商业证书——剩下的存储、续期、多节点共享和热加载由 LiteGate 完成。本文把单机与集群、自动与付费四种组合逐个写清楚，并教你在内置 DNS 服务商不够用时，如何自己写插件编译进去。

0先搞清楚：证书有三种来源

LiteGate 的入口层在 TLS 握手时按 SNI 找证书。证书可以来自下面三条路径，它们可以在同一个网关里并存：

A · 自动申请

配置 auto_cert，网关运行时向 Let's Encrypt 等 ACME 机构申请、续期。适合自己掌控 DNS 或 80 端口的域名。
本文 A1 / A2

B · 自带付费证书

你从 DigiCert / 阿里云 / 沃通等买了商业证书（OV/EV、或更长链路的 DV），把 .pem + .key 交给网关加载。
本文 B1 / B2

C · 按需签发

SaaS 多租户场景：握手时域名才第一次出现，授权通过后即时签发。属于 A 的进阶玩法，详见 证书生命周期 与 on-demand 文档。

单机 vs 集群的唯一区别

配置语义完全一致，区别只在于证书存到哪里：单机存本地磁盘 certs_dir；集群把权威副本存进 Consul / Litemesh KV，所有节点共享同一份，再镜像回各自本地磁盘做高速加载。所以「集群配置」= 「单机配置」+ 「打开一个存储角色」。

1公共配置：auto_cert 基础块

自动申请相关配置都在全局 config.yaml 的顶层 auto_cert 块下。最小可用骨架：

yaml
auto_cert:
  enabled: true                 # 总开关
  email: "ops@example.com"      # ACME 账号邮箱（必填）
  certs_dir: "./certs"          # 本地证书目录（默认 ./certs）
  ca_provider: "letsencrypt"    # CA：letsencrypt(默认)/zerossl/buypass/google/custom
  renew_days: 30                # 过期前多少天触发续期
  check_interval: 12            # 周期检查间隔（小时）
  use_staging: false            # true=用测试环境，调试时避免触发频率限制
  storage_secret: ""            # 共享存储下加密私钥的对称密钥（>=32 字节，强烈建议）
字段名容易踩的坑

DNS 服务商的凭证写在 dns_providers[].config 下（注意是 config:，不是 provider_config:）。provider_config 仅用于已废弃的顶层单 Provider 兼容写法。新配置一律用 dns_providers 列表。

2选择证书颁发机构（CA）与 EAB

默认 Let's Encrypt，无需任何额外凭证。切换 CA 只改 ca_provider，网关会自动选用对应 ACME 目录地址，并为每个 CA 单独保存账号私钥，互不干扰。

ca_provider内置 ACME 地址是否需要 EAB
letsencrypt（默认）acme-v02.api.letsencrypt.org否
zerosslacme.zerossl.com/v2/DV90是
buypassapi.buypass.com/acme否
google / gtsdv.acme-v02.api.pki.goog是
custom由 acme_url 指定视服务而定

ZeroSSL 与 Google Trust Services 强制要求 EAB（外部账号绑定）。到对应控制台拿到 kid/hmac 后成对填写，否则启动时会直接报配置错误并拒绝启动：

yaml
auto_cert:
  enabled: true
  email: "ops@example.com"
  ca_provider: "zerossl"
  eab_kid: "your-key-id"
  eab_hmac: "your-hmac-key"

私有 CA / step-ca

yaml
auto_cert:
  enabled: true
  ca_provider: "custom"
  acme_url: "https://ca.internal.example.com/acme/acme/directory"   # custom 时必填
迁移 CA 会被一致性校验拦截

集群存储里记录了 _cluster_ca_provider，所有节点必须用同一个 CA + ACME URL。若要切换 CA，需先删除存储中的该元数据键（本地磁盘即 certs_dir 下的 _cluster_ca_provider.pem/.key）再重启。

3A1 · 自动申请：单机

单机不需要任何外部存储，证书直接落 certs_dir。按你能控制的校验通道二选一。

方式一：HTTP-01（最简单，需公网 80 端口）

不需要配置任何 DNS 服务商。在站点 YAML 里声明域名并开启 force_https，网关会拦截 /.well-known/acme-challenge/ 完成校验。前提是公网能访问到本机 80 端口、域名 A 记录已指向本机。

yaml · sites/app.yaml
domain: app.example.com
force_https: true               # 开启即自动激活 HTTP-01 自动证书
routes:
  - name: web
    match: { path_prefix: "/" }
    action: { type: proxy, upstream: ["127.0.0.1:8080"] }

同时在全局开启 auto_cert.enabled 与 email 即可（见 §1 骨架，不需要 dns_providers）。

方式二：DNS-01（支持泛域名 / 不开 80 端口）

申请 *.example.com 这类泛域名证书，或 80 端口不可开放时，必须走 DNS-01——网关用 DNS API 写入校验 TXT 记录。内置服务商：aliyun / cloudflare / tencent / huawei。

yaml · config.yaml
auto_cert:
  enabled: true
  email: "ops@example.com"
  certs_dir: "./certs"
  storage_secret: "a-32-bytes-or-longer-secret-key-xxxxx"
  dns_providers:
    - name: "ali-main"           # 逻辑名（被 domains 路由引用，可与 type 不同）
      type: "aliyun"             # 内置服务商类型
      enabled: true
      domains: ["*.example.com", "example.com"]
      config:                    # 整个 map 原样传给 Provider
        access_key_id: "LTAI..."
        access_key_secret: "..."
        # 可选传播控制：
        # propagation_wait: "60s"
        # recursive_nameservers: "223.5.5.5,8.8.8.8"

域名匹配某个 provider 的 domains 时，网关就用该 provider 走 DNS-01。证书申请成功后写入：

text · certs_dir 布局
certs/
├── example.com.pem            # 完整证书链
├── example.com.key            # 私钥（0600）
├── _wildcard.example.com.pem  # 通配符证书单独前缀，避免与裸域冲突
├── _wildcard.example.com.key
├── _acme_account.key          # Let's Encrypt 账号私钥（其它 CA 为 _acme_account_）
└── _cluster_ca_provider.pem   # CA 一致性元数据
单机也建议设 storage_secret

本地磁盘上的 .key 仍是明文（受 0600 保护，满足握手高速加载）。但只要配了 storage_secret，未来切到共享存储或推送 KV 时私钥会自动 AES-256-GCM 加密。提前设好，迁移零改动。

4A2 · 自动申请：集群

多个 LiteGate 节点对外服务同一批域名时，不要让每个节点各自向 CA 申请——会重复签发、触发频率限制、证书各不相同。正确做法：选一个共享存储作为权威副本。在 §3 的 DNS-01 配置基础上，只需多打开一个存储角色。

用 Litemesh KV 作为证书存储（推荐）

yaml · config.yaml
litemesh:
  enabled: true
  address: "127.0.0.1:7400"
  token: "${LITEMESH_TOKEN}"
  roles:
    cert_storage: true          # ← 打开这个角色，证书即存入 Litemesh KV
  keys:
    cert_prefix: "litegate/certs/"   # 可选，默认即此

auto_cert:
  enabled: true
  email: "ops@example.com"
  storage_secret: "a-32-bytes-or-longer-secret-key-xxxxx"   # 集群强烈必填
  dns_providers:
    - name: "ali-main"
      type: "aliyun"
      enabled: true
      domains: ["*.example.com"]
      config: { access_key_id: "LTAI...", access_key_secret: "..." }

或用 Consul KV 作为证书存储

yaml · config.yaml
consul:
  enabled: true
  address: "127.0.0.1:8500"
  roles:
    cert_storage: true          # ← 同理，证书存入 Consul KV
  keys:
    cert_prefix: "litegate/certs/"

打开角色后，集群自动获得这些保障（无需你额外配置）：

机制解决什么
分布式锁同一域名同时被多节点遇到时，只有一个节点向 CA 申请，其余等待并读取结果。
pending 标记Litemesh 存储下标记「正在申请」的域名，锁内双检，进一步压制脑裂/锁竞争导致的重复签发。
确定性延迟申请/续期前加入分散延迟，避免整个集群在同一毫秒冲击 CA 或 DNS API。
本地镜像权威副本在 KV，签发后自动镜像到各节点 certs_dir，TLSManager 仍从本地磁盘高速加载。
CA 一致性_cluster_ca_provider 强制所有节点使用同一 CA + ACME URL，配置漂移会被启动校验拦截。
集群下 storage_secret 不是可选项

证书私钥会进共享 KV。不配 storage_secret 等于把私钥明文写进注册表，任何能读 KV 的人都能拿到。务必配置（≥32 字节），并通过环境变量等方式管理，不要硬编码进 git。

5B1 · 自带付费证书：单机

你已经从商业 CA 买了证书（拿到 fullchain.pem 和 privkey.key），不走 ACME。这种证书绑定在站点或 Stream 上，在站点 YAML 里用 tls.cert_file / tls.key_file 指定文件路径即可。

HTTP 站点（七层）

yaml · sites/shop.yaml
domain: shop.example.com
tls:
  enabled: true
  cert_file: "/etc/litegate/certs/shop.fullchain.pem"   # 必须是完整链
  key_file:  "/etc/litegate/certs/shop.privkey.key"
routes:
  - name: web
    match: { path_prefix: "/" }
    action: { type: proxy, upstream: ["127.0.0.1:8080"] }
付费证书不会被自动证书顶替

站点一旦 pin 了 cert_file/key_file，网关绝不会悄悄回退到 ACME/按需证书。即使文件加载失败，也会继续用上一份已加载的好证书，而不是降级——避免一次手滑把线上 HTTPS 打挂。

TCP/UDP Stream（四层，如 MySQL/gRPC over TLS）

yaml · streams/db.yaml
name: mysql-tls
port: 3306
protocol: tcp
upstream: "10.0.0.10:3306"
enabled: true
tls:
  enabled: true
  cert_file: "/etc/litegate/certs/db.fullchain.pem"
  key_file:  "/etc/litegate/certs/db.privkey.key"
  min_version: "1.2"            # 可选：1.2 / 1.3
cert_file 与 key_file 必须成对

只配其中一个，启动/加载会直接报错（Stream 路径明确校验「成对出现」）。证书必须是完整链（服务器证书 + 中间 CA），否则部分客户端会校验失败。

6B2 · 自带付费证书：集群

集群里付费证书有两种分发思路。

思路一：每个节点放同一份文件（最朴素）

用配置管理工具（Ansible / Salt / 镜像内置）把同样的 cert_file/key_file 推到每台机器的相同路径，YAML 写法与 §5 完全一致。简单，但轮换时要逐台更新文件。

思路二：证书放进 KV，所有节点从 KV 读（推荐）

把 cert_file/key_file 的值写成 KV 引用前缀，网关启动/加载时从共享存储拉取，一处更新全集群生效。支持两种 scheme：

前缀含义
litemesh-kv:<key>从 Litemesh KV 读取该 key 的值作为 PEM 内容
consul-kv:<key>从 Consul KV 读取该 key 的值作为 PEM 内容
yaml · sites/shop.yaml（集群）
domain: shop.example.com
tls:
  enabled: true
  cert_file: "litemesh-kv:litegate/manual-certs/shop/fullchain"
  key_file:  "litemesh-kv:litegate/manual-certs/shop/privkey"
routes:
  - name: web
    match: { path_prefix: "/" }
    action: { type: proxy, upstream: ["127.0.0.1:8080"] }

把买来的证书内容写入 KV

bash · Consul 示例
# PEM 文件内容整体作为 value 写入
consul kv put litegate/manual-certs/shop/fullchain @shop.fullchain.pem
consul kv put litegate/manual-certs/shop/privkey   @shop.privkey.key
bash · Litemesh 示例
# 用 litemesh CLI / API 把同样的内容写入对应 key
litemesh kv put litegate/manual-certs/shop/fullchain --file shop.fullchain.pem
litemesh kv put litegate/manual-certs/shop/privkey   --file shop.privkey.key
cert_file 与 key_file 必须用同一种 scheme

不能一个 litemesh-kv: 一个本地路径——前缀必须成对匹配，否则回退到本地磁盘加载逻辑导致找不到文件。前缀只对 tls.cert_file/key_file（手动证书）生效；ACME 自动证书走的是 auto_cert 的存储角色，二者机制不同。

KV 里的手动证书私钥不会自动加密

与 ACME 私钥不同，手动证书走 cert_file/key_file 直读路径，写进 KV 的内容就是你放进去的明文 PEM。请用 KV 自身的 ACL / 加密能力保护这两个 key，或考虑用 §4 的自动证书路径（私钥受 storage_secret 加密）。

7混合：付费证书与自动证书并存

真实场景常常是：主域名买了 OV 证书走付费，泛域名子站点和内部域名走自动。两者按站点粒度独立配置，互不影响：

  • 付费站点：站点 YAML 配 tls.cert_file/key_file → pin 住，永不被 ACME 顶替。
  • 自动站点：站点不配 tls 块（或只 force_https），由全局 auto_cert 接管申请。
  • 优先级：握手时若站点显式 pin 了证书，直接用 pin 的；否则查内存里的自动/已加载证书；再否则走按需签发（若开启）。

8内置 DNS 不够用？写个插件编译进来

内置只有 aliyun / cloudflare / tencent / huawei。你的 DNS 服务商不在其中、或想用内部 IPAM / 私有 DNS API 完成 DNS-01 校验（以及可选的 DDNS A/AAAA 同步）时，自己写一个 DNS Provider 插件即可。契约在公共包 github.com/jamesleeon/LiteGate/pkg/plugin，第三方可在独立仓库编写并编译进去。

一个名字，两种能力（按需各自实现）

能力接口干什么
证书 DNS-01plugin.CertDNSProviderACME 质询时增删 _acme-challenge 的 TXT 记录
DDNS 同步plugin.DDNSProvider把本机 IP 写到域名 A/AAAA 记录

只签证书就只实现 CertDNSProvider；两个都要就都实现。给某 provider 配了证书用途（域名进了 dns_providers[].domains）却没实现 CertDNSProvider，核心会在用到时返回明确 error。

第 1 步：实现接口 + 注册

go · mydns/provider.go
package mydns

import (
	"context"
	"fmt"

	"github.com/jamesleeon/LiteGate/pkg/plugin"
	"github.com/rs/zerolog/log"
)

const providerName = "mydns" // 用户写在 dns_providers[].type 里的名字，不能与内置名重名

type provider struct {
	apiKey    string
	apiSecret string
}

func (p *provider) GetName() string { return providerName }

// 能力一：证书 DNS-01
func (p *provider) AddTXTRecord(ctx context.Context, domain, token, keyAuth string) error {
	// 在 _acme-challenge. 上加一条值为 keyAuth 的 TXT；注意尊重 ctx 取消
	return p.upsertTXT(ctx, "_acme-challenge."+domain, keyAuth)
}
func (p *provider) DeleteTXTRecord(ctx context.Context, domain, token, keyAuth string) error {
	return p.removeTXT(ctx, "_acme-challenge."+domain, keyAuth) // 按值精确清理，别误删并发质询
}

// 能力二（可选）：DDNS
func (p *provider) Sync(ctx context.Context, domain, ip string) error {
	return p.upsertAddr(ctx, domain, ip) // 记录类型由 ip 是 v4/v6 决定
}

func init() {
	err := plugin.RegisterDNSProvider(providerName, func(config map[string]string) (plugin.DNSProvider, error) {
		key, secret := config["api_key"], config["api_secret"]
		if key == "" || secret == "" {
			return nil, fmt.Errorf("%s: api_key and api_secret are required", providerName)
		}
		return &provider{apiKey: key, apiSecret: secret}, nil
	})
	if err != nil {
		log.Error().Err(err).Str("plugin", providerName).Msg("register DNS provider failed")
	}
}

第 3 步：编译进二进制（用户拥有 main）

在你自己的仓库建一个 main，空导入插件包再调 litegate.Run()：

go · main.go
package main

import (
	"log"

	"github.com/jamesleeon/LiteGate/pkg/litegate"
	_ "your-module/mydns" // ← init() 里 RegisterDNSProvider
)

func main() {
	if err := litegate.Run(); err != nil {
		log.Fatalf("litegate exited: %v", err)
	}
}

9轮换、续期与热加载

自动证书：全自动

网关每 check_interval 小时检查一次，过期前 renew_days 天自动续期；续期成功写入存储 + 镜像本地 + 触发 TLSManager 热加载，全程无感。集群下只有一个节点真正续期，其余读结果。

付费证书：换文件后要「碰一下」YAML

重要：换 .pem 文件本身不会触发热加载

配置文件监听器只监听 sites/、streams/ 目录下 .yaml/.yml 的写入事件，不监听证书文件本身。所以你把新的 shop.fullchain.pem 覆盖上去后，网关不会自动重载。轮换付费证书时，请在替换证书文件后重新保存（touch）对应的站点 / Stream YAML，触发一次 reload——站点路径会重新从磁盘读取证书；或者重启进程。

KV 引用方式（litemesh-kv: / consul-kv:）同理：更新 KV 里的值后，需要重新保存引用它的 YAML 触发加载，KV 变更本身不会自动 push 重载。

强制重新申请自动证书

删除存储中对应域名的证书与私钥（本地磁盘即 certs_dir 下的 example.com.pem/.key，或 KV 中对应 key）并重启，网关立即重新进入申请流程。

现象排查方向
证书一直 PendingHTTP-01 看公网 80 端口是否可达；DNS-01 看 DNS API 凭证与 A 记录；看是否撞了 CA 频率限制（调试期先 use_staging: true）。
启动报 CA provider conflict集群里 ca_provider/acme_url 漂移了。统一配置，或删除存储中 _cluster_ca_provider 后重启。
zerossl/google 启动即报错缺 eab_kid/eab_hmac，到 CA 控制台领取后成对填写。
付费证书 HTTPS 失败cert_file/key_file 是否成对、路径是否可读、是否完整链；KV 方式检查前缀 scheme 是否一致、key 是否存在。
换了证书文件没生效见 §9——需 touch 对应 YAML 或重启。
通配符证书签不下来必须用实现了 CertDNSProvider 的 DNS Provider，HTTP-01 不支持泛域名。
日志告警「私钥明文存储」共享存储下务必配 storage_secret（≥32 字节）。

auto_cert（全局）

字段说明
enabled自动证书总开关
emailACME 账号邮箱（必填）
certs_dir本地证书目录，默认 ./certs
ca_provider / acme_urlCA 选择；custom 时 acme_url 必填
eab_kid / eab_hmaczerossl/google/gts 必填
renew_days / check_interval续期提前天数 / 周期检查间隔（小时）
use_staging用 CA 测试环境，调试期避免频率限制
storage_secretAES-256 对称密钥，加密存储中的私钥（共享存储强烈必填）
dns_providers[]DNS 服务商列表：name/type/enabled/domains/config/ddns
on_demand按需签发（SaaS 多租户），详见证书生命周期文档

站点 / Stream 的 tls 块（手动证书）

字段说明
tls.enabled开启该站点/Stream 的 TLS

tls.auto_cert / tls.domains（Stream）让四层监听也走自动证书
