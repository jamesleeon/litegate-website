LiteGate mTLS 使用指南

本文说明在 Litemesh 身份体系下，南北流量（LiteGate → 后端）与 东西流量（后端 ↔ 后端）如何启用 mTLS 双向认证。两侧共用同一套 「mtls 标签 + mtls_port 元数据」约定：后端只声明一次， 南北向的 LiteGate 与东西向的 SDK 都会自动改用 mTLS 访问。

Litemesh CA SPIFFE 身份 证书自动轮换 Fail-Closed 南北流量 东西流量 明文 + mTLS 双模
一句话总览

后端可同时监听明文与 mTLS 端口。开启 mTLS 后，注册到 Litemesh 时打上 mtls 标签并带上 mtls_port。从此：南北向的 LiteGate 会用 mTLS 转发流量；东西向的 Litemesh SDK（CallService）也会按同一标签自动用 mTLS 调用。一处声明，南北 + 东西全链路生效。

适用场景：后端服务已接入 Litemesh；LiteGate 通过 upstream_type: litemesh 发现后端（南北）； 服务间通过 Litemesh SDK CallService 互相调用（东西）；后端希望只接受来自可信工作负载的 mTLS 请求。

1mTLS 工作原理

LiteGate 的 mTLS 完全围绕 Litemesh 的身份体系运转，分为两条主线：证书生命周期（LiteGate 自身的客户端身份）和对端身份校验（验证后端是不是真的可信）。

证书生命周期

当 litemesh.mtls: true 且配置了 spiffe_id 时，LiteGate 启动 CertManager：

  1. 本地生成 2048 位 RSA 密钥对（私钥永不出本进程）。
  2. 携带公钥与自身 SPIFFE ID 调用 Litemesh POST /v1/ca/sign 申请工作负载证书。
  3. 拉取 Litemesh Root CA（GET /v1/ca/root），构建带热加载回调的 tls.Config。
  4. 后台每小时检查，剩余寿命低于 1/3 时自动轮换（上限 30 天、下限 1 小时），失败重试 3 次，并上报 Prometheus 指标。
无需手动管理证书

LiteGate 不读取磁盘上的证书文件做南向 mTLS —— 证书完全由 Litemesh 动态签发与轮换。你只需保证 LiteGate 能访问 Litemesh Agent。

对端身份校验 & Fail-Closed

向后端发起连接时，LiteGate 在 TLS 握手中：

  • 出示客户端证书：通过 GetClientCertificate 提供自己的 Litemesh 证书，构成"双向"。
  • 校验证书链：用 Litemesh Root CA 验证后端证书是否可信。
  • 校验 SPIFFE 身份：逐一比对后端证书 URI SAN 与期望的 SPIFFE ID，强于普通 TLS 的域名匹配。
两条硬性安全约束

① Fail-Closed：若证书提供方不可用、客户端证书为空、Root CA 池为空，连接会立即拒绝拨号，绝不静默降级为明文。

② insecure_skip_verify 无法绕过 mTLS：开启 mTLS 后，即便配置了跳过校验，LiteGate 仍强制执行 Root CA 与 SPIFFE 身份验证（仅打印一条警告）。

2后端双模：明文 + mTLS

推荐后端同时监听两个端口：一个明文 HTTP（如 :8080）用于内网可信场景或健康检查，一个 mTLS HTTPS（如 :8443）用于零信任调用。是否启用 mTLS 由后端自己决定 —— 启用后，只需在注册时打上 mtls 标签（见第 3 节），南北向的 LiteGate 与东西向的 SDK 就会自动改走 mTLS 端口。

双模的意义

同一份代码、同一个服务实例可以平滑灰度：先以明文注册（不带 mtls 标签）跑通链路，再加上 mTLS 监听并补上标签，调用方无需改任何代码即自动升级为双向认证。回退同理。

mTLS 端口要做三件事：向 Litemesh CA 申请工作负载证书、用它启动 HTTPS、要求调用方也提交由 Litemesh CA 签发的客户端证书。下面用 Go 标准库 net/http 示例。

go 
package main

import (
	"crypto/rand"
	"crypto/rsa"
	"crypto/tls"
	"crypto/x509"
	"fmt"
	"log"
	"net/http"

	litemesh "github.com/james/litemesh/sdk"
)

func main() {
	namespace := "default"
	serviceName := "user-service"
	mtlsAddr := ":8443"

	// LiteGate 默认按此格式推导后端 SPIFFE ID:
	// spiffe://litemesh.local/ns/<namespace>/sa/<service-name>
	spiffeID := fmt.Sprintf("spiffe://litemesh.local/ns/%s/sa/%s", namespace, serviceName)

	client := litemesh.NewClient(litemesh.Config{
		AgentAddr: "http://127.0.0.1:8787",
		AuthToken: "your-litemesh-token",
	})

	privateKey, _ := rsa.GenerateKey(rand.Reader, 2048)
	certManager, err := client.WatchCertificate(spiffeID, privateKey)
	if err != nil {
		log.Fatalf("watch workload certificate failed: %v", err)
	}
	defer certManager.Close()

	rootPEM, _ := client.GetRootCA()
	rootPool := x509.NewCertPool()
	rootPool.AppendCertsFromPEM([]byte(rootPEM))

	tlsConfig := &tls.Config{
		MinVersion:           tls.VersionTLS12,
		GetCertificate:       certManager.GetCertificate,
		GetClientCertificate: certManager.GetClientCertificate,

		// 服务端 mTLS 关键: 要求调用方提交客户端证书, 并用 Litemesh Root CA 校验
		ClientAuth: tls.RequireAndVerifyClientCert,
		ClientCAs:  rootPool,
		RootCAs:    rootPool,

		// 可选: 进一步只放行 LiteGate 的 SPIFFE ID
		VerifyConnection: allowClientSPIFFE(
			"spiffe://litemesh.local/ns/gateway/sa/litegate",
		),
	}

	mux := http.NewServeMux()
	mux.HandleFunc("/api/users", func(w http.ResponseWriter, r *http.Request) {
		w.Header().Set("Content-Type", "application/json")
		w.Write([]byte(`{"service":"user-service","mtls":true}`))
	})

	server := &http.Server{Addr: mtlsAddr, Handler: mux, TLSConfig: tlsConfig}
	log.Printf("mTLS server on %s, spiffe_id=%s", mtlsAddr, spiffeID)
	log.Fatal(server.ListenAndServeTLS("", ""))
}

func allowClientSPIFFE(allowed ...string) func(tls.ConnectionState) error {
	set := make(map[string]struct{}, len(allowed))
	for _, id := range allowed {
		set[id] = struct{}{}
	}
	return func(cs tls.ConnectionState) error {
		for _, cert := range cs.PeerCertificates {
			for _, uri := range cert.URIs {
				if _, ok := set[uri.String()]; ok {
					return nil
				}
			}
		}
		return fmt.Errorf("client SPIFFE ID is not allowed")
	}
}
最小配置

若暂不想限制调用方身份，可去掉 VerifyConnection，但请保留 ClientAuth: tls.RequireAndVerifyClientCert 和 ClientCAs，至少保证调用方证书来自同一个 Litemesh CA。

3注册到 Litemesh（标签驱动）

把普通端口、mTLS 端口、标签和元数据一起注册。mtls 标签是南北与东西两侧共用的开关 —— LiteGate 和 Litemesh SDK 看到它后都会自动改走 mTLS 端口。

一个标签，两侧生效

南向：LiteGate 的 litemesh discovery 读取 mtls 标签 → 用 mTLS 转发。
东西：Litemesh SDK 的 CallService 通过 serviceRequiresMTLS() 读取同一个 mtls 标签 → 用 mTLS 调用。
所以后端只需声明一次，全链路统一。

go 
err := client.Register(&litemesh.ServiceInstance{
	ID:        "user-service-1",
	Name:      "user-service",
	Namespace: "default",
	Addr:      "10.0.0.12",
	Port:      8080,        // 普通 HTTP 端口
	Protocol:  "http",

	// 关键: 裸标签 mtls 表示该实例要求 mTLS 访问
	Tags: []string{"mtls"},

	Meta: map[string]string{
		"mtls_port":       "8443",                                            // LiteGate 访问的 mTLS 端口
		"spiffe_id":       "spiffe://litemesh.local/ns/default/sa/user-service", // 可选, 显式声明
		"tls_server_name": "user-service",                                    // 可选, 作为 TLS SNI
	},
})

最小可用注册：当服务名 + 命名空间能推导出正确 SPIFFE ID 时，spiffe_id 可省略。

go 
Tags: []string{"mtls"},
Meta: map[string]string{"mtls_port": "8443"},
// Name=user-service, Namespace=default →
// 默认期望: spiffe://litemesh.local/ns/default/sa/user-service

若 trust domain 不是 litemesh.local，加一个 trust_domain 即可改变默认推导前缀：

go 
Meta: map[string]string{
	"mtls_port":    "8443",
	"trust_domain": "example.internal",
}
// 默认期望变为: spiffe://example.internal/ns/default/sa/user-service
统一使用裸标签 mtls（两侧已对齐）

南向（LiteGate）与东西向（SDK）现在只认裸标签 mtls，判定规则已统一。请使用 Tags: []string{"mtls"}，建议小写 —— LiteGate 端大小写敏感、精确匹配，SDK 端则会容忍大小写和首尾空格，统一小写可保证两侧行为完全一致。

4南北流量：LiteGate 配置

全局 litemesh 配置（启用客户端身份）

这是南向 mTLS 的前提：LiteGate 必须先拥有自己的 Litemesh 身份，才能向后端出示客户端证书。

yaml 
litemesh:
  enabled: true
  address: "http://127.0.0.1:8787"
  token: "your-litemesh-token"
  namespace: "default"

  # 开启 mTLS: LiteGate 会向 Litemesh CA 申请自己的工作负载证书
  mtls: true
  spiffe_id: "spiffe://litemesh.local/ns/gateway/sa/litegate"
  mtls_port: 8443   # 可选: 全局兜底端口, 实例 Meta.mtls_port 优先
必填项

南向 mTLS 要求 mtls: true 且 spiffe_id 非空，两者缺一则 CertManager 不会初始化，发现的 mTLS 实例将因"证书提供方不可用"而 Fail-Closed 拒绝。

同时确保 litemesh 被作为服务发现源启用：

yaml 
service_discovery:
  catalogs:
    - provider: litemesh
      enabled: true
      url: "http://127.0.0.1:8787"
      namespace: "default"

路由级配置

大多数情况只需用 litemesh 发现，由实例自己通过 mtls 标签声明安全访问方式 —— 路由无需额外配置：

yaml 
domain: api.example.com
routes:
  - name: user-service
    match:
      path_prefix: /users
    action:
      type: proxy
      upstream_type: litemesh
      service_name: user-service
      namespace: default

若想在路由上强制启用 mTLS 或显式声明期望身份（覆盖实例声明），可使用：

yaml 
    action:
      type: proxy
      upstream_type: litemesh
      service_name: user-service
      namespace: default
      mtls: true                # 强制本路由走 mTLS
      expected_spiffe_ids:      # 显式声明期望的后端身份
        - spiffe://litemesh.local/ns/default/sa/user-service
优先级

是否启用 mTLS = action.mtls OR 实例 mtls 标签。期望 SPIFFE ID 优先取路由 expected_spiffe_ids，其次取实例 Meta 推导值。推荐让实例自声明，路由保持简洁。

5东西流量：SDK 调用

东西流量是后端服务之间的相互调用，由 Litemesh SDK（github.com/james/litemesh/sdk）直接完成，不经过 LiteGate。SDK 与 LiteGate 复用同一个 mtls 标签约定：调用方发现目标实例后，若目标带 mtls 标签，就自动切到 https + mtls_port 并用 mTLS 客户端发起请求。

服务间 mTLS 调用

调用方需要做三步：① 用自己的 SPIFFE ID 拿到 Litemesh 工作负载证书；② 构建动态 mTLS 客户端并注入 SDK；③ 直接 CallService，mTLS 判定与端口切换由 SDK 自动完成。

go 
package main

import (
	"crypto/rand"
	"crypto/rsa"
	"log"

	litemesh "github.com/james/litemesh/sdk"
)

func main() {
	client := litemesh.NewClient(litemesh.Config{
		AgentAddr: "http://127.0.0.1:8787",
		AuthToken: "your-litemesh-token",
	})

	// ① 拿到本服务(order-service)的工作负载证书, 自动轮换
	spiffeID := "spiffe://litemesh.local/ns/default/sa/order-service"
	privateKey, _ := rsa.GenerateKey(rand.Reader, 2048)
	cm, err := client.WatchCertificate(spiffeID, privateKey)
	if err != nil {
		log.Fatalf("watch cert failed: %v", err)
	}
	defer cm.Close()

	// ② 用动态证书 + Litemesh Root CA 构建 mTLS 客户端, 注入 SDK
	rootPEM, _ := client.GetRootCA()
	tlsCfg, err := litemesh.BuildDynamicTLSConfig(cm, rootPEM)
	if err != nil {
		log.Fatalf("build tls config failed: %v", err)
	}
	client.SetMTLSClient(litemesh.GetMTLSClient(tlsCfg))

	// ③ 直接调用 user-service。若其带 mtls 标签,
	//    SDK 自动用 https + mtls_port + 客户端证书发起请求。
	result, err := client.CallService("user-service", &litemesh.CallOptions{
		Namespace: "default",
		Method:    "GET",
		Path:      "/api/users",
	})
	if err != nil {
		log.Fatalf("call failed: %v", err)
	}
	log.Printf("user-service response: %#v", result)
}
SDK 如何判定是否走 mTLS

SDK 内部 serviceRequiresMTLS() 仅在实例带裸 mtls 标签时返回 true（与 LiteGate 判定规则一致）。命中后 resolveCallEndpoint() 把协议切为 https、端口切到 Meta["mtls_port"]，再用 mTLS 客户端发起请求。

注意事项

生产必须先 SetMTLSClient

CallService 若目标是 https 但你没有通过 SetMTLSClient 注入 mTLS 客户端，SDK 会临时降级为 InsecureSkipVerify=true 以兼容自签证书 —— 这只适合本地联调。生产环境务必先注入由 BuildDynamicTLSConfig 构建的真实 mTLS 客户端，否则等于跳过了对端校验。

SPIFFE 身份校验需自行扩展

SDK 的 BuildDynamicTLSConfig 默认只校验 Litemesh Root CA 证书链，其 VerifyConnection 是空实现，不像 LiteGate 那样自动比对对端 SPIFFE URI SAN。若东西向也要做身份白名单（只允许特定 SPIFFE ID 调用），需在该回调中自行实现（可参考第 2 节的 allowClientSPIFFE）。

证书轮换与南北一致

SDK 的 WatchCertificate 与 LiteGate 采用同款自动轮换策略（剩余寿命低于 1/3 时续期，检查间隔为寿命的 1/10，最短 1 分钟、最长 1 小时），无需手动续期。

7流量边界矩阵

明确 Litemesh mTLS 在各方向流量的覆盖范围：

方向链路协议由谁提供身份
北向 North 客户端 → LiteGate 单向 TLS（或明文） 公网证书 / AutoCert（Let's Encrypt）
南向 South LiteGate → 后端 mTLS 双向认证 Litemesh CA + SPIFFE（自动校验身份）
东西 East-West 后端 ↔ 后端（SDK） mTLS 双向认证 Litemesh CA（SPIFFE 校验需自行扩展）
Litemesh mTLS 覆盖南向 + 东西，不覆盖北向

LiteGate 的入站监听不强制要求客户端证书。所以严格说北向是"客户端→网关 单向 TLS"，而非端到端 mTLS。Litemesh 的零信任身份保证存在于网关↔后端（南向）和后端↔后端（东西）之间。

若北向也需要客户端证书认证（如内网服务间调用 / 设备接入），应在入站 TLS 层另行配置 ClientAuth，这不属于本文覆盖范围。

南北与东西的对称性

两个方向都以后端实例的 mtls 标签为统一开关，都用 Litemesh CA 签发并自动轮换证书。主要差异：南向（LiteGate）默认强制 SPIFFE 身份比对，东西向（SDK）默认只验 CA 链、SPIFFE 白名单需在 VerifyConnection 中自行实现。

8与 Istio Sidecar 方案对比

本方案是「网关 + SDK 库内嵌」模式：mTLS 逻辑跑在 LiteGate 进程和业务进程内部（in-process）。Istio 则是「Sidecar 代理」模式：每个 Pod 注入一个 Envoy 代理，用 iptables 透明劫持所有进出流量。两者解决的是同一类问题（零信任 mTLS + 服务发现），但权衡截然不同。

本方案的优势

✓ 适合什么

① 高性能 / 低开销：无 Sidecar 进程、无额外网络跳数，数据面是业务进程的原生 TLS 连接，延迟与资源开销最低。
② 部署简单：不强依赖 K8s，裸机、VM、容器都能跑，组件少、运维心智负担低。
③ 与网关天然融合：南北向（LiteGate）与东西向（SDK）共用同一套 mtls 标签和 Litemesh CA，身份体系统一。
④ 行为可控透明：mTLS 在你自己的代码里，握手、校验、错误都直观可调试，没有"魔法劫持"。

本方案的劣势

✗ 受限于什么

① 有代码侵入：东西向调用必须引入 SDK 并走 CallService；遗留服务或非 Go 服务接入成本高。
② 语言绑定：SDK 目前主要面向 Go，多语言生态不如 Sidecar 透明。
③ 覆盖靠自觉：绕过 SDK 的直连流量不会被纳管，无法像 iptables 那样"强制全量 mTLS"。
④ 功能聚焦：没有 Istio 那样开箱的流量治理（熔断、重试、镜像、故障注入）、声明式授权策略和统一遥测，需要时得自行补齐。
⑤ 东西向 SPIFFE 校验需自实现：SDK 的 VerifyConnection 默认只验 CA 链，身份白名单要自己写（见第 5 节）。

一句话选型

追求极致性能、轻量部署、不想引入 K8s/Sidecar 复杂度，且服务以 Go 为主、能接受引 SDK —— 本方案更优。需要多语言透明接入、强制全量纳管、丰富的流量治理与声明式策略，且已重度使用 K8s —— Istio 更合适。两者也可共存：用本方案做网关南北向 + 核心 Go 服务东西向，Istio 覆盖其余异构工作负载。

9常见问题

spiffe_id 必须写吗？

不必须。只要后端证书的 SPIFFE ID 符合默认格式 spiffe://litemesh.local/ns/<namespace>/sa/<service-name> 就能省略。

mtls_port 和 port 有什么区别？

port 是普通端口，mtls_port 是开启 mTLS 后实际访问的 TLS 端口。常见部署：8080 普通 HTTP、8443 mTLS HTTPS。

insecure_skip_verify 能跳过 mTLS 校验吗？

不能。mTLS 模式下 LiteGate 会强制执行 Litemesh Root CA 和 SPIFFE 身份校验，即便配了 insecure_skip_verify: true 也只会打印一条警告，不会跳过身份验证。

证书会过期吗？需要我手动续期吗？

不需要。LiteGate 每小时检查，剩余寿命低于 1/3 时自动向 Litemesh 重新签发并热加载（上限 30 天、下限 1 小时），失败自动重试 3 次。

后端如何只允许 LiteGate 调用？

在后端 tls.Config.VerifyConnection 中检查客户端证书 URI SAN，只放行 LiteGate 的 SPIFFE ID（见 第 2 节示例）：

text 
spiffe://litemesh.local/ns/gateway/sa/litegate

连接报"cert provider unavailable / client certificate is empty"怎么办？

这是 Fail-Closed 在生效，说明 LiteGate 未能拿到自己的工作负载证书。排查：litemesh.mtls: true 是否设置、spiffe_id 是否非空、LiteGate 能否访问 Litemesh Agent 的 /v1/ca/sign 与 /v1/ca/root。

南北和东西需要分别配置吗？

后端声明只需一次：注册时带上裸 mtls 标签 + mtls_port。南向由 LiteGate 自动识别，东西向由 SDK 自动识别。两侧的调用方各自需要持有自己的 Litemesh 证书（LiteGate 通过 litemesh.mtls/spiffe_id 配置，SDK 通过 WatchCertificate + SetMTLSClient）。

用 Meta["mtls"] 或 mtls=true 标签能触发 mTLS 吗？

不能。南北（LiteGate）与东西（SDK）现已统一为只认裸 mtls 标签，Meta["mtls"]、mtls=true 等写法都不再生效。请始终使用 Tags: []string{"mtls"}（小写）。

东西向调用如何只允许特定服务访问？

SDK 默认只验 CA 链。要做 SPIFFE 身份白名单，需在被调方的服务端 VerifyConnection 检查客户端证书 URI SAN（同第 2 节的 allowClientSPIFFE），或在调用方 BuildDynamicTLSConfig 的 VerifyConnection 中校验对端服务端身份。
