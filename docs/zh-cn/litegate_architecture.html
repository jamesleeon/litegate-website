0系统架构图景

LiteGate 采用清晰的三栏式架构设计：左侧来自互联网的流量，通过灵活的域名和协议进入中间的 LiteGate 核心路由引擎。经过规则匹配与过滤器链后，除了常规代理转发至右侧的后端基础设施外，LiteGate 特有的内置 Web Server 能够针对静态页面、Markdown 文档和 WebDAV 共享等请求进行本地闭环响应，无需再转发至外部 nginx 容器。所有服务发现源（Providers）均支持热载入，Kubernetes 正在通过通用协议适配中。

1三条核心设计哲学

控制面 / 数据面分离

解耦配置流与流量流,让数据面在"零锁(Lock-Free)、零分配(Zero-Allocation)"状态下运行,换取极高吞吐。

反转执行流水线

请求物理执行不是机械的 L1→L7 堆叠,而是把网络防御与资源限额前置、安全与身份注入中置、负载均衡与链路治理后置。

多源聚合与自研共识

既兼容 Consul 与 Discovery 协议,又原生集成极简的自研 Litemesh Gossip 协议,支持去中心化高效组网。

2控制平面与数据平面

网关内部由两个高度解耦的平面协作组成,实现高并发下的"路由无感知零开销热更新"。

控制平面 · 配置流(由 app.go 编排)
watcher.go · fsnotify 本地目录 consul_loader.go · Catalog litemesh_loader.go · 成员状态 external · API 注册中心 mcp/server.go · 大模型推送
↓ CatalogLoader 统一规范化为 ActiveSite / ActiveStream  ↓ 原子指针交换
数据平面 · 流量流(零锁只读)
router.go · atomic.Value 路由树 middleware/core.go · 生命周期 action/executor.go · 代理执行 proxy/pool.go · 端点连接池

控制平面职责

CatalogLoader 是一个开放式适配器:把静态文件路由、Consul Catalog、Litemesh 容器实例和外部 API 注册中心的端点无差别合并,甚至能同时承载同一服务的 Consul 物理实例与 Litemesh 容器实例。

数据平面职责

路由器只持有一个只读路由树指针,无需任何全局 Mutex,以只读内存副本做高并发匹配;所有上游被编译并缓存进 PoolManager,请求流经时不再反射或重分配内存。

3反转执行流水线:一个请求的完整旅程

概念上遵循 L1–L7,但物理执行顺序经过深度"反转"。安全与限流被前置,以保障网关不会因后端鉴权失效而雪崩。

L1 · 入口防御 · router.go

Router.ServeHTTP(Pre-processing)

任何配置上下文生效前优先做全局 IP 过滤(Fast-Fail);剥离客户端伪造的 X-Tenant-ID/X-SID 等敏感身份头;规范化 URL 路径;零开销拦截 ACME Challenge 和 /_litegate/ 系统接口(命中则直接返回,不进入业务层)。

L2 · 路由匹配

GetSiteMatch

在内部 Trie 树中按 Host/Path 快速定位站点 [id] 与具体 Route,把命中的 MatchedPrefix 注入 Action Context。未命中任何 Site/Route 则在此返回 404。

L3 · 业务容器 · core.go

CoreHandler(TraceID & Lifecycle)

只有命中路由后才包装进 CoreHandler:生成符合 W3C 的 X-Trace-Id(若客户端未带)、基于 recover() 挂载 Panic 隔离(单中间件 panic 也只返回 500,进程不挂)、初始化 logMeta 日志容器。

L4 · 中间件链 · BuildRouteHandler

Core → 动态插件 → CORS → IP → WAF → RateLimit → Auth → Compress

RateLimit 在 Auth 之前:限的是该路由总吞吐,既在后端崩溃前过载保护,也保护下游 IDS 鉴权服务和 Redis 不被无效请求压垮。Auth(OIDC / IDSSession / RemoteAuth)成功后,把解密出的租户元数据映射成标准 Header(X-Tenant-ID、X-SID、X-DB-Name),实现零侵入上下文透传。

L3/L5/L6 · 代理执行 · proxy.go

Proxy Executor(Routing & Rewriting)

路径改写(StripPrefix 动态读取 MatchedPrefix / PrependPrefix);灰度抽签(按 router.gray_weight 每请求掷骰子,中签走 upstream.gray 节点,灰度池空则回退全量);两阶段 FilterEndpoints(强过滤 selector + 软匹配 meta + 自动回退);LB 选节点(RoundRobin / IpHash / Weighted / P2C)并应用超时/重试/熔断。

后端

Outbound 发包 → 回写响应

物理发包到选中的健康实例,接收响应并写回客户端。

L7 · 可观测性 · core.go (defer)

Observability

请求完成后聚合整条流水线:路由 ID、选中的后端物理地址、鉴权结果、各阶段耗时,输出标准化 JSON 日志。

4零停机热重载:原子指针交换

生产环境里任何重置路由的行为(加域名、改灰度权重)都不能中断正在传输的 TCP 连接。LiteGate 用原子指针交换规避抖动。

  1. 检测变更
    Watcher 或 KV 感知到配置修改 / 服务发现变化。
  2. 离线构建
    拷贝并解析全量配置,在内存中静默构建全新 Trie 树,完成校验、端口冲突检测,不影响运行中的数据面。
  3. 原子交换
    构建就绪后执行 Val.Store(NewTree),指针在纳秒级瞬间切换。
  4. 旧树回收
    旧树没有新活跃引用,处理完遗留请求后由 GC 异步安全回收。
Fail-Safe

如果离线构建 NewTree 失败(格式错、证书冲突等),重载流程直接报错终止,指针绝不切换,旧树继续稳定服务。端口 80/443 全程保持监听,无需 fork 进程或优雅关闭端口,无任何系统级 I/O 损耗。

5Litemesh 集群共识

多节点集群下,LiteGate 摆脱 K8s / ZooKeeper 这类重型组件,用改良 Gossip 协议达成最终一致性。

Notify-Pull 模式

传统 Gossip 在大集群频繁广播成员明细,易造成带宽风暴。Litemesh 改成:节点变化时只广播一个极轻的"变更 ID",接收方本地判定后异步单播向源节点拉取明细,带宽降低约 85%。

逆熵状态恢复(Anti-Entropy)

网络丢包可能造成临时状态丢失。Litemesh 每 5 分钟后台拉起一次轻量级全量校验,与随机节点比对差异,确保极端断网恢复场景下的最终一致性。

6L4 Stream 与弹性高可用

除了 7 层 Web 服务,LiteGate 原生集成高性能 L4 Stream Server,处理非 HTTP 流量,如 MySQL 连接池、DNS 协议代理及底层 IoT TCP/UDP 数据。转发层内建闭环流量治理引擎,实现微秒级自愈。

灰度软路由

按 gray_weight 为每次请求掷骰子,中签进灰度端点池;灰度池全宕则自动退回全量健康池,确保高可用。

双阶段实例过滤

第一阶段硬匹配 router.selector 与租户标签,强隔离池为空直接返回 502,物理层绝不混杂跨租户流量;第二阶段在强隔离池中按 router.meta 软匹配,无命中则回退。

熔断与重试

每次 outbound 失败在满足重试因子时自动换物理节点重发;某实例持续报错触发熔断阈值则标记"亚健康"并移出活跃池。

7性能与连接池调优

LiteGate 能在笔记本级配置跑出 ~5.5w RPS、常驻内存 ~50MB,核心在于对 PoolManager 的极致调优。参数定义在 internal/proxy/types.go 的 DefaultProxyConfig()。

go
func DefaultProxyConfig() *ProxyConfig {
    return &ProxyConfig{
        MaxIdleConns:        4096,             // 全局最大空闲连接数
        MaxIdleConnsPerHost: 4096,             // 每个后端 Host 最大空闲连接数
        MaxConnsPerHost:     8192,             // 每个后端 Host 最大总连接数
        IdleConnTimeout:     90 * time.Second,

        DefaultDialTimeout:           10 * time.Second,
        DefaultTLSHandshakeTimeout:   10 * time.Second,
        DefaultResponseHeaderTimeout: 30 * time.Second,

        CleanupInterval:        5 * time.Minute,  // 空闲连接自动清理
        TransportIdleTimeout:   10 * time.Minute,
        EnableTransportCleanup: true,
    }
}

极高的单 Host 空闲长连接

Caddy 或标准 Go 代理单 Host 默认空闲连接常为 2 或 100,高并发下空闲池瞬间装满,被迫不停"建连→传输→关连",带来巨大握手开销并易耗尽临时端口抛 Address already in use。LiteGate 直接拉到 4096,并发请求复用长连接,把 TLS 握手开销摊薄为零。

动态传输清理与自愈

CleanupInterval: 5m 的后台回收器巡检空闲池;一旦某后端实例请求量滑坡或被缩容摘除,自动物理关闭这些空闲连接,把内存平滑归还操作系统,常驻保持 ~50MB 极低水位。
