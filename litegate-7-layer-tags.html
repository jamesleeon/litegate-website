7-Layer Tags

LiteGate 7 层 Tag vs Traefik Labels

这页不比较“谁功能更多”，只比较服务注册时到底在声明什么。 Traefik 的 labels 更像在生成 Traefik 自己的 router、service、middleware 对象。 LiteGate 的 7 层 tag 更像在声明流量如何穿过入口系统本身。

LiteGate 的关键字更接近运行时治理语义 Traefik 的关键字更接近内部对象拼装语义 差异不只是写法，而是建模视角不同
一句话判断 LiteGate 的 7 层 tag 是有用的，而且大部分已经真实进入运行时。 它最大的价值不是标签数量，而是把入口、路由、治理、上游和观测拆成了有职责边界的执行层。

Core Difference

两边都在用标签，但标签表达的不是同一件事

LiteGate 最新 tag 体系遵循 litegate.[id].<layer>.<key>， 其中 id 是逻辑路由分组，layer 是 7 层治理模型。 这意味着一个服务注册时，写的不是一堆零散开关，而是在描述完整流量执行面。

Traefik labels 在表达什么

  • traefik.http.routers.* 定义路由对象。
  • traefik.http.services.* 定义上游服务对象。
  • traefik.http.middlewares.* 定义中间件对象。
  • 本质是通过标签把 Traefik 内部配置对象拼出来，再把对象连起来。

LiteGate 7 层 tag 在表达什么

  • ingress 定义从哪里进来。
  • router 定义如何匹配、如何分流、如何灰度。
  • middleware 定义认证、限流和请求处理链。
  • service 定义网关与后端连接治理。
  • upstream 定义物理实例属性。
  • observability 定义日志和追踪行为。

At A Glance

最小例子一摆，差异就很明显

LiteGate

litegate.api.ingress.host=stock.51ecp.com
litegate.api.ingress.path=/apiv2
litegate.api.router.strip_prefix=true
litegate.api.service.timeout=30s
litegate.api.observability.log=info

Traefik

traefik.http.routers.api.rule=Host(`stock.51ecp.com`) && PathPrefix(`/apiv2`)
traefik.http.routers.api.middlewares=strip-api
traefik.http.middlewares.strip-api.stripprefix.prefixes=/apiv2
traefik.http.services.api.loadbalancer.server.port=8080
LiteGate 这边写的是“入口怎么暴露、怎么治理”。Traefik 这边写的是“创建一个 router、一个 middleware、一个 service，然后把它们连起来”。

7 Layers

逐层看，这套设计到底是不是有用

层级 LiteGate 关键字 实际作用 判断
L1 Entrypoint litegate.entrypoint.bind 可直接把动态服务挂到额外监听端口；若未显式声明 host/path，会联动推出 0.0.0.0:<port> 与默认 /<serviceName> 路径。 这是实打实的动态调试入口能力。开发者给服务打一条 bind 标签，就能临时开一个独立端口联调，不必先写站点文件。
L2 Ingress litegate.[id].ingress.host
litegate.[id].ingress.path		 直接决定服务暴露到哪个 host/path，是动态站点和路由生成的核心输入。 这是价值最高的一层，最接近“注册即上线”的本质。
L3 Router rule priority gray_weight lb_policy
router.selector.* router.meta.* methods headers.*		 既控制请求匹配，也控制灰度、分片、实例筛选、路径改写和流量策略。 非常强，已经不只是 router，而是 traffic policy layer。
L4 Middleware auth auth_provider ratelimit inject_response 进入 RemoteAuth、限流和身份注入链路。 有效，但当前更偏核心治理接口，还不是广义中间件宇宙。
L5 Service timeout retry breaker* 决定网关与后端之间的超时、重试和熔断策略。 分层很合理，语义清晰，和 router/upstream 的边界也比较好。
L6 Upstream port weight gray
health_path mtls insecure		 定义实例物理属性，参与端口覆盖、灰度池划分、健康检查和后端 TLS 行为。 很有用，尤其和 L3 的灰度、选择器配合时非常顺。
L7 Observability log trace 影响路由级日志等级和链路追踪元数据注入。 是实打实进运行时的，不只是装饰性标签。

Why It Matters

这套 7 层 tag 真正值钱的地方

1. 标签不再是平铺开关

很多系统的 labels 最终只是把零散配置项塞进 provider。LiteGate 的 7 层 tag 则把标签组织成一个分层运行时模型， 让每个关键字都归属于明确职责。

2. 一个服务可以有多条逻辑入口

litegate.[id].* 这个结构让同一服务可以同时拥有多条逻辑路由和不同治理策略， 而不是被限制在单一“服务 = 单一路由”思维里。

3. 分层之间是可联动的

例如 router.gray_weight 和 upstream.gray、 router.selector.* 和实例 metadata、observability.* 和 access log， 都不是各自孤立存在。

4. L1 对开发联调尤其友好

entrypoint.bind=0.0.0.0:9090 这种写法，本质上就是“给这个服务临时拉一个可访问端口”。 如果服务名是 api，默认路径还能直接推出 /api，这非常适合开发、灰度验证和临时旁路调试。

5. 默认就是开发者友好的透明代理

不显式配置时，LiteGate 默认尽量保留原始请求语义：Host 不乱改、路径不乱改， 但会自动补齐 X-Real-IP、X-Forwarded-For、 X-Forwarded-Proto 和 X-Forwarded-Host。 这很符合后端微服务开发者的期待：能不配就不配，但后端该拿到的信息还是要自动拿到。

Vs Traefik

和 Traefik 相比，真正强在哪里

Traefik 的长处

  • 标签生态成熟，Docker / Kubernetes / Consul 等环境接入体验顺滑。
  • 对象模型稳定，大家都熟悉 router / service / middleware 这套心智。
  • 如果用户只想在现有云原生环境里生成入口配置，Traefik 的标签已经足够强。

LiteGate 的长处

  • 标签直接对准网关执行层，不只是组装内部对象。
  • 更适合把 host、path、灰度、认证、实例选择、熔断、mTLS、观测统一收在同一模型里。
  • 更适合“服务注册就是上线意图声明”的平台化场景。
  • 在 entrypoint.bind 这件事上，更像给开发者一把随手可开的调试入口钥匙；Traefik 的 entryPoints 主要还是静态配置面，不能靠服务标签直接新开监听端口。
  • 默认行为更偏透明代理，明显更像后端开发者写给后端开发者的入口系统，而不是先逼你学一套复杂边缘配置学。

Fair Verdict

最公平的判断

不够准确的说法

“LiteGate 只是把 Traefik labels 换了一套名字。”这不对，因为两边标签背后的抽象并不一样。 Traefik labels 更偏对象拼装，LiteGate 7 层 tag 更偏运行时分层治理。

更准确的说法

LiteGate 的 7 层 tag 体系是有用的，而且核心层大部分已经真实落进运行时。 它的价值不在于“标签更多”，而在于让服务在注册时就能描述流量如何进入、如何分流、如何治理、如何观测。 它背后的产品取向也很明确：优先让后端开发者在少配置甚至不配置的情况下，把服务顺手接起来。

对比官方设计，Traefik 的 entryPoints 属于静态安装配置，路由只能引用已有 entryPoints； Nginx 当然能用 listen 开端口，但那是手写 server block 的静态配置； Caddy 也能在站点地址里直接写端口。LiteGate 这一层的特别之处在于： 它让“服务注册元数据”本身就能把调试端口和默认访问路径一起长出来。
