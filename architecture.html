Architecture

统一入口层的结构与边界

这页不再只是把模块列出来，而是解释 LiteGate 为什么能同时承接站点托管、动态代理、证书自动化、服务发现和 SaaS 入口能力， 又不会把简单场景拖成一个过重的平台。

这页想回答的不是“功能有没有” 而是 LiteGate 为什么能从网站和 API 一路长到私有云、动态服务发现和多租户 SaaS 场景，同时还保持入口层边界清晰。
Positioning

LiteGate 的核心不是多功能，而是入口层重新统一

官网首页讲“为什么值得看”，这页讲“为什么这套结构站得住”。

LiteGate 是一个把站点托管、七层代理、四层代理、动态服务发现、自动 HTTPS 和接入治理整合在一起的统一入口层。 它既能从本地 YAML 直接起站，也能接入 Litemesh、Consul 或外部 discovery agent，把动态服务暴露成统一入口，同时继续承接证书、鉴权和入口治理。

Capabilities

它在入口层里统一了哪些能力

下面这些能力不是平行堆叠，而是围绕同一个入口执行面组织起来的。

站点托管

静态文件服务、SPA 回退、Markdown 渲染、WebDAV 和简单文件承接都可以直接挂到入口层。

Static SPA Markdown WebDAV

代理能力

HTTP/HTTPS 反向代理、WebSocket / SSE / gRPC 以及 L4 TCP / UDP 转发都在同一条入口链路内。

HTTP WebSocket TCP / UDP HTTP/3

动态发现

支持静态 upstream、Consul、Litemesh 和兼容 blocking query 的外部发现层，适合从固定到动态逐步演进。

Static Consul Litemesh External

接入治理

AutoCert、DNS-01、On-Demand TLS、OIDC / OAuth2、远程鉴权、WAF、限流和熔断，都能继续留在入口层统一治理。

AutoCert DNS-01 OIDC WAF / Rate Limit
Core Structure

从入口请求到后端系统，中间其实只有一套入口执行面

这也是 LiteGate 能同时兼容简单站点和复杂 SaaS 场景的关键。

入口请求

浏览器 / App 静态站点、SPA、Markdown、WebDAV、控制台前端
API 客户端 HTTP / HTTPS / WebSocket / SSE / gRPC 请求
TCP / UDP 客户端 访问四层端口代理和非 HTTP 服务
统一接入
Unified Entry Layer

LiteGate 核心执行面

站点托管、代理、证书、服务发现和身份入口并不是分散的模块，而是同一条入口链路上的不同职责层。

路由与站点承接 静态站点、SPA 回退、Markdown、WebDAV、响应 / 重定向都在这一层统一处理。
代理与服务发现 HTTP 代理、L4 代理、负载均衡、静态 upstream、Consul / Litemesh / External Discovery。
证书与入口治理 AutoCert、DNS-01、On-Demand TLS、OIDC / OAuth2、Remote Auth、WAF、限流、熔断。
运维与观测 Dashboard、流量可视化、配置热更新、日志和状态能力继续留在入口层统一感知。
托管 / 转发 / 调度

后端与外部系统

站点资源 dist、文档目录、文件目录、静态资源和前端构建产物
业务服务 静态 upstream、微服务、L4 服务和动态实例集合
服务发现与外围依赖 Litemesh、Consul、CamoDNS Discovery、DNS Provider、OIDC Provider、Redis
Integration Path

接入时最重要的，不是功能列表，而是你需要哪些外围组件

不同复杂度的团队，接入 LiteGate 的方式并不一样。

方式 1：只用 LiteGate

适合静态站点、SPA、少量固定 upstream 场景。配置来源可以是本地 YAML，也可以是 Consul / Litemesh KV。这一层已经能覆盖站点托管、反向代理、L4 代理、证书、鉴权和基础治理。

方式 2：LiteGate + litemesh-registrator-go

适合 Docker / Swarm 动态接入。注册器会监听 Docker 事件，把带有 litemesh.register=true 的容器或服务自动注册到 Litemesh，并把 litegate.* 标签转成服务元数据，供 LiteGate 动态发现和路由使用。

方式 3：LiteGate + camodns_discovery

适合已有异构注册中心的环境。CamoDNS Discovery 暴露 /v1/catalog/services 和 /v1/health/service/:service_name 这类 blocking query 接口，把 Docker、K8s、Nacos、Etcd、Litemesh 聚合成 LiteGate 可消费的统一发现层。

Boundary Matrix

不同场景下，LiteGate 应该如何接入

这张表的重点不是“有没有能力”，而是哪些场景适合直接用，哪些场景需要配合发现层或外围组件。

场景 推荐方式 适用说明 价值
静态站点 / SPA / 文档托管 LiteGate 直接托管 适合官网、控制台前端、文档站点和内部平台入口。 少组件、少维护、上线快。
固定 upstream 的 API 接入 LiteGate 直接代理 适合后端节点较稳定、希望快速统一入口的项目。 统一路由、统一 HTTPS、统一治理。
Docker / Swarm 动态服务 LiteGate + Litemesh 适合容器实例频繁变动、需要自动注册和自动切换的场景。 服务上线更快，入口层无需手工维护实例列表。
已有 Consul / Litemesh 环境 LiteGate 动态发现 适合已经有服务注册与健康检查体系的团队。 保留现有体系，同时统一外部入口。
异构注册中心并存 LiteGate + Discovery 聚合层 适合 Docker、K8s、Nacos、Etcd 等并存的混合环境。 减少入口层对多种注册中心的适配成本。
需要统一入口安全与治理 LiteGate 统一承载 适合需要 HTTPS、鉴权、限流、WAF、熔断等接入能力的业务入口。 把安全与治理收敛到入口层，而不是分散到每个服务里。
Why It Holds

为什么这套结构更适合 LiteGate

它不是为了做一个更重的平台，而是为了让入口层既能承接简单场景，也能继续长到更复杂的现实系统里。

它真正解决的问题

把“站点入口、动态服务入口、自动 HTTPS 和接入层鉴权”放回同一个网关模型里，避免前端托管、反代、证书、发现分别靠不同组件拼装。

它最适合的人

适合想用单二进制或轻量 Docker 方案解决入口层问题的团队，尤其是私有云、VPS、Docker、Swarm 和混合注册中心环境。

它刻意不强调什么

它不是 Kubernetes Ingress Controller 文档，也不是服务注册中心本身。动态注册和异构发现能力，仍然由外围组件协同完成。

Hosted on LiteGate

这个架构页本身也可以直接由 LiteGate 托管

这不是一句装饰性文案，而是希望让用户知道：LiteGate 不只是能讲这些结构，它本身就能承接这种真实站点页面。

首页、架构页和入口模型页都可以直接由 LiteGate 提供静态托管。
如果后面接入 API、认证回跳、证书自动化和服务发现，也不需要换入口层。
这就是 LiteGate 想强调的统一入口层：先把网站接住，再把复杂度接住。
