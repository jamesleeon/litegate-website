Gateway Philosophy

LiteGate 不是另一个代理，而是在收拢真实上线动作

很多网关能转发请求，但真实团队痛的并不是“能不能反代”，而是站点上线、证书、DNS、前端发版、后端扩容、 服务发现、租户分流这些动作为什么总要拆成很多步。LiteGate 想做的是把这些动作重新收进一条更短的链路里。

看今天验证了什么 看它和普通网关差在哪
不是只讲代理

它在意的不只是流量进来之后怎么办，还在意上线之前和升级之后还有哪些重复动作。

不是只讲配置

服务启动、入口接入、前端切版本、证书复用，这些动作应该尽量自然发生，而不是靠人补流程。

不是只讲架构图

控制面可以分层，但数据面和上线动作不能为了概念优雅多走很多跳，否则最后会变成性能税和运维税。

What We Learned

今天这些细节说明了什么

不是功能“理论上能做”，而是很多关键链路已经在真实场景里跑通了。

同 host 合并前后端已经跑通

一个 host 下，前端静态页面和后端 API 自动拼接，不再默认拆成两套域名和两层入口。

服务注册已经带入口意图

服务注册不只是 IP 和端口，还带 host、path、版本和标签，服务起来就更接近“上线”而不是“待接入”。

file server 已经很顺

一行 lite.yaml 就能把目录变成可访问文件站点，而且还能继续作为前端 ZIP 的下载源。

前端自动升级闭环成立

ZIP 上传、KV 改值、LiteGate 自动拉包切版本，这已经不是目录同步，而是版本声明式发布。

泛域名体验非常强

新子域名接进来时，不一定还要回到域名服务商后台，也不一定再单独申请证书，现成的泛域名证书可以直接复用。

默认约定开始有产品感

例如服务名默认推导 path，复杂场景再显式覆盖。简单场景足够省事，复杂场景也仍然可控。

Why Different

它和普通网关真正不同的地方，不是功能数量，而是职责边界

LiteGate 不是想取代所有系统，而是在把原本散落在网站服务器、动态代理、证书脚本、静态分发和上线流程里的动作收回来。

普通网关更像“请求来了怎么转发”
  • 重点在 upstream、路由、TLS、负载均衡。
  • 静态站点、前端发布、服务发现接入往往要再拼别的组件。
  • 很多动作能做，但不是默认就顺。
LiteGate 更像“一个服务怎么从声明走到可访问”
  • 站点入口、API 接入、证书、DNS、文件承接都在同一套模型里。
  • 服务发现和前端版本切换不是额外外挂，而是继续沿着入口模型收敛。
  • 更关注少手工、少跳转、少逐台操作。
真正理解网关，不只是理解怎么转发请求。

更关键的是理解一个团队在真实世界里到底会被哪些上线动作反复折腾，然后把技术和运维收进一套更统一的执行面。LiteGate 的很多设计，都是从这条主线逼出来的。

What It Means

为什么这种产品形态在现在更有意义

现在能生成代码不难，难的是长期跑得住、升级不痛、扩容不乱、别人也能接着用。真正有价值的不是功能堆叠，而是让主路径一直顺。

对小团队

没人专职运维时，最值钱的是少手工、少控制台切换、少逐台同步，而不是再多一份复杂平台。

对中型团队

服务多起来以后，真正贵的是上线动作碎裂。统一入口、服务发现和版本切换能大幅降低持续交付摩擦。

对混合云和多租户场景

公有云、私有云、混合云并存时，最需要的不是再多一个代理，而是更少分裂的接入和升级模型。

对开源产品本身

一个项目是不是玩具，不只看功能有多少，而看它能不能在真实环境里长期承担稳定、升级、扩容和接入的责任。

LiteGate Gateway Philosophy

这页记录的是一个很核心的理解：LiteGate 不是在追求“又一个网关”，而是在尝试把真实团队会反复经历的上线和升级动作收短、收顺、收成一条链路。
