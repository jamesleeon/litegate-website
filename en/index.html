Why LiteGate

Migration shouldn't stem from "more features", but from "less fragmentation"

LiteGate is not trying to be a bloated gateway. It consolidates scattered duties in site hosting, proxying, certs, discovery, and access governance back into a unified entry layer.

What many entry architectures eventually degrade to

  • Sites hosted in one tool, dynamic proxies in another
  • Certs, DNS, service discovery, and identity gateways scattered across external scripts or third-party platforms
  • As systems grow complex, the ingress layer becomes the hardest part to maintain

LiteGate offers a simpler answer

  • A unified site model hosting both static resources and backend APIs
  • Native auto HTTPS, dynamic discovery, and entry governance built directly into the main pipeline
  • Grow smoothly from simple setups to full SaaS scenarios, without shifting architectures halfway

Automation Story

From a single label to instant go-live: see the complete pipeline

LiteGate's true strength lies not just in routing, but in consolidating service registration, domain binding, cert acquisition, and HTTPS readiness into a single continuous action.

01

Declare service or site

Write a few lines of `*.lite.yaml` or tag a container/service with `litegate.*` labels to declare your ingress intent.

02

Register & discover

Service instances register with Litemesh, or expose unified discovery interfaces for LiteGate through a bridging layer.

03

DNS & certs follow suit

New domains bypass manual DNS/cert steps; A-record sync, DNS verification, and HTTPS readiness are handled natively by the ingress plane.

04

Directly live and ready

Ultimately, you get more than just "config written" — you get a live domain, an active cert, and ready routes.

Smoother new domain launches After declaring a site, A-records and certs follow automatically, reducing the risk of forgetting DNS edits during releases.
Fewer manual prerequisites Eliminates prerequisites like "manually point DNS first, then request a certificate".
Wildcards made easy DNS validation natively fits private cloud, multi-domain, and multi-environment setups, especially for managing wildcards.
Govern certs and sites together Domains, site entries, cert policies, and backend routing no longer require maintenance in separate components.

Frontend Release

Frontend upgrades without individual node deploys: just publish a version

This is more than a static file config option. It condenses frontend releases into a single version submission: zip and upload assets to OSS, write version info to Litemesh KV, and LiteGate automatically syncs, unpacks, switches, and manages history across all nodes.

# 1. build frontend and upload ZIP to OSS
$oss://frontends/main-12.zip
# 2. write one KV key
litegate/config/front/main
{
"version": "12",
"url": "https://oss.example.com/front/main-12.zip",
"md5": "3e5a..."
}
# 3. LiteGate nodes auto sync and switch

Consolidated release operations

No need to log into each node to swap static directories or run complex distribution scripts. Just upload the ZIP, update a KV key, and your release is done.

LiteGate handles the rest

Nodes listen to KV updates, fetch the new package, extract to local version dirs, switch symlinks, and retain historical versions for quick rollback.

Better multi-node consistency

New nodes fetch the correct version automatically, while existing nodes update simultaneously. For small teams, this beats manual directory copy or rsync across nodes.

Less Ops

No more file-copy across nodes

Frontend upgrades scale from "distributing files" to "declaring versions" — perfect for small teams running without dedicated DevOps.

Multi-Node

Natively consistent multi-node

All nodes converge state around a single KV key, eliminating human errors across machines during deployment.

Rollback Ready

Rollback-ready history

Retaining previous versions ensures rollback doesn't require rebuilding; simply shifting the version pointer yields a clean rollback.

Lite vs Full

Start lightweight for simple cases, upgrade seamlessly for complexity

Lite Mode offers the shortest path to go live, while Full Mode is a complete governance model. They share the same core, exposed via different configuration entry points.

Lite Mode

Shortest path for simple setups 
example.com:
  proxy: 127.0.0.1:8080
  • Perfect for single sites, SPAs, basic proxying, redirects, and WebDAV.
  • A few lines of *.lite.yaml get your site up and running
  • An intuitive user-facing configuration, rather than a secondary configuration core

Full Mode

Retaining full governance & extensibility 
domain: example.com
routes:
  - match:
      path_prefix: /
    action:
      type: proxy
      upstream:
        - 127.0.0.1:8080
  • Perfect for complex matches, authentication, middlewares, discovery, and advanced ingress governance.
  • When your system demands richer expressions, seamlessly switch to the full model
  • Lite configs compile down to the standard model, ensuring clear migration paths
