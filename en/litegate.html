Not a Caddy replacement, but a step further forward

LiteGate inherits the low barrier-to-entry of site hosting and auto HTTPS, but scales beyond a mere web server.

Not a Traefik alternative, but completes the entry plane

Dynamic routing is just one part. LiteGate consolidates site hosting, cert automation, and identity-driven access into a single execution plane.

For Nginx users, this liberates the certificate lifecycle from manual chores

If you often forget to point DNS before requesting certificates, this tedious prerequisite is managed by LiteGate.

If you still manually apply, rotate, and reload certs, DNS-01, wildcards, and auto-renewal bring peace of mind by bringing these into the entry layer.

From a single label to instant go-live

The real benefit isn't just packet routing; once ingress intent is stated, container registry, DNS sync, cert acquisition, and HTTPS readiness deploy continuously.

Decoupled Service Discovery

If you use Consul, Litemesh, Nacos, etcd, or heterogeneous clusters, LiteGate doesn't force a full swap. It decouples the discovery layer to bridge arbitrary registries.

MCP is not a buzzword: it executes complete ingress setup

Instead of abstract "AI configuration", you describe the target, and AI uses LiteGate MCP to inspect schemas, write configs, and verify end-to-end access.

Frontend updates are now native to the ingress pipeline

Upload frontend ZIPs to OSS, update a Litemesh KV version key, and LiteGate handles multi-node sync, symlinks, and historic rollbacks.

01 Package

Once build completes, compress into a ZIP and upload to OSS; no more rsync or directory copy across nodes.

02 Commit Version

Write version, url, and checksum into KV. The release workflow is as simple as submitting a version pointer.

03 Auto Converge

Nodes converge automatically to the target version — ideal for small teams deploying Standalone configurations.

This automated pipeline is where LiteGate feels like a true unified ingress layer

It goes beyond dynamic routing, focusing on the entire go-live pipeline rather than just a segment of it.

01 Declare

Write a *.lite.yaml or label your containers with litegate.* to state your ingress intent.

02 Discover

Service instances register to Litemesh, or expose unified discovery interfaces through a bridging layer.

03 Go Live

New domains bypass manual DNS/cert steps; A-record sync, DNS verification, and HTTPS readiness are handled natively by the ingress plane.
