Positioning

Consolidating the Ingress Plane: The Core of LiteGate

While the landing page answers "why it is worth watching", this page explores "why the architecture holds stable".

LiteGate is a unified entry layer that consolidates website hosting, L7 proxying, L4 proxying, dynamic service discovery, auto HTTPS, and ingress governance. It can spin up sites directly from local YAML or connect to Litemesh, Consul, or external discovery agents to expose dynamic backends while managing certs, auth, and routing.

Integration Path

Bridges Matter: Essential External Components for Ingress

Teams with varying technical requirements can plug into LiteGate in distinct ways.

Option 1: Standalone LiteGate

Perfect for static web assets, SPAs, and small sets of static upstreams. Configs can be fed via local YAML or Consul/Litemesh KV. This tier handles site serving, reverse proxying, L4 TCP/UDP, TLS certs, basic auth, and routing.

Option 2: LiteGate + litemesh-registrator-go

Optimized for Docker/Swarm dynamic onboarding. The registrator watches Docker engine events, registering containers with the litemesh.register=true tag into Litemesh, converting litegate.* labels into service metadata for dynamic discovery and routing.

Option 3: LiteGate + camodns_discovery

Ideal for heterogeneous registry environments. CamoDNS Discovery bridges Docker, K8s, Nacos, etcd, and Litemesh, pushing catalog changes to LiteGate.

Boundary Matrix

Boundary Matrix: Direct vs Collaborative Ingress

This matrix focuses not on raw capability, but on which scenarios work out-of-the-box versus requiring external discovery bridges.

Scenario Ingress Strategy Application Notes Core Value
Static site / SPA / Docs hosting Native LiteGate hosting Marketing sites, admin consoles, documentation hubs, and internal dashboard portals. Fewer components, zero maintenance overhead, and rapid deployment.
Static upstream API routing Direct LiteGate proxying Environments with stable backend nodes wanting a consolidated entry point. Unified routing, unified HTTPS, and unified access governance.
Docker / Swarm dynamic services LiteGate + Litemesh Ideal for container environments where instances churn rapidly and require automated onboarding and balancing. Faster service launches; no manual server list maintenance in the entry plane.
Existing Consul / Litemesh environments LiteGate Dynamic Discovery Teams with mature service registries and health-check workflows. Consolidate external ingress while keeping your existing catalog.
Heterogeneous registries coexisting LiteGate + Discovery bridging Hybrid deployments running Docker, Kubernetes, Nacos, or etcd simultaneously. Reduces the cost of adapting the ingress layer to multiple registries.
Unified ingress security & governance Consolidated LiteGate governance Ingresses requiring HTTPS, auth, rate limiting, WAF, and circuit breakers. Consolidates security and governance at ingress, rather than scattering them across backends.
Why It Holds

Why This Structure Fits LiteGate

It is not designed to be an over-engineered platform, but rather to ensure the entry plane accommodates simple scenarios while growing naturally alongside complex production architectures.

The Problem It Solves

Consolidates sites, dynamic proxies, auto HTTPS, and auth into one model, preventing fragmented setups of separate web servers, proxies, and cert managers.

Who It Is For

Teams seeking single-binary or lightweight Docker ingress solutions, especially in VPS, private clouds, Swarm, and hybrid registries.

What It Deliberately Omits

It is not a Kubernetes Ingress Controller specification, nor does it serve as the raw service registry itself. Dynamic catalog registry and heterogeneous discovery bridging are managed in collaboration with peripheral layers.
