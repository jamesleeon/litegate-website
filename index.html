Is It For You

你不用先看功能，先看自己是不是现在该用 LiteGate 的人

官网先帮用户判断适配度，比先铺一页功能清单更重要。

很适合你

入口层已经开始分裂

你已经在用 Caddy、Nginx 或类似工具托管站点，但又需要另一套入口处理动态服务、注册中心、反向代理或认证链路。

可能适合你

现在是网站和 API，后面会继续长

今天只是在接静态站点和后端服务，但你已经能预见 Docker、服务发现、身份入口和多租户分流会很快进入系统。

暂时不用换

你只需要一个最简单的站点服务器

如果只有单静态站点，没有动态后端、注册中心或 SaaS 入口要求，LiteGate 现在不一定值得你迁移。

Why LiteGate

迁移动机不该来自“功能更多”，而应该来自“分裂更少”

LiteGate 不是想变成一个什么都做的巨型网关，而是把原本散在站点托管、代理、证书、发现和接入治理上的职责重新收回到一个入口层里。

很多团队现在的入口层会慢慢变成这样

  • 站点托管在一套工具里，动态代理在另一套工具里
  • 证书、DNS、服务发现和身份入口散落在外围脚本或平台里
  • 系统一复杂，入口层先变成最难维护的地方

LiteGate 的答案更简单

  • 用同一个站点模型承接静态资源和后端 API
  • 把自动 HTTPS、动态发现和入口治理放回主链路
  • 让系统从简单模式一路长到 SaaS 场景，而不是中途换轨

Automation Story

从一个 label 到直接上线，这条链路应该被完整看见

LiteGate 更强的地方，不只是能做动态路由，而是它可以把“注册服务、接入域名、配证书、完成 HTTPS”这条上线链路收成一个连续动作。

01

声明服务或站点

写一个 `*.lite.yaml`，或者给容器 / 服务打上 `litegate.*` 相关 label，先把入口意图表达出来。

02

注册与发现接上

服务实例被注册到 Litemesh，或者通过 discovery 桥接层暴露成 LiteGate 能消费的统一发现接口。

03

DNS 与证书自动跟上

新域名不必再先走一轮手工 DNS 和证书流程。A 记录同步、DNS 验证和 HTTPS 就绪可以继续由入口层收敛处理。

04

服务直接对外可用

最后拿到的不是“配置写完了”，而是域名可访问、证书已生效、入口路由已就绪的可用结果。

新域名上线更顺手 配置站点后，A 记录和证书可以自动跟上，减少发布时忘改 DNS 的概率。
手工前置动作更少 不用再依赖“先把域名指过来，再开始申请证书”这一套前置步骤。
泛域名更顺手 DNS 验证天然更适合私有云、多域名和多环境部署，尤其是需要统一管理泛域名时。
证书和站点一起治理 域名、站点入口、证书策略和后端路由不用再拆到不同组件里分别维护。
这也是它和很多只负责“路由到哪台机器”的动态网关不一样的地方：LiteGate 更关注一条完整的上线链路，而不是只接住其中一段。

Frontend Release

前端升级不再逐台发包，提交一个版本就够了

这不是多一个静态文件配置项，而是把前端发布动作收敛成一次版本提交： 前端构建产物打成 ZIP 上传到 OSS，把版本信息写进 Litemesh KV， LiteGate 会在所有节点自动完成下载、切换和历史版本保留。

# 1. build frontend and upload ZIP to OSS
$oss://frontends/main-12.zip
# 2. write one KV key
litegate/config/front/main
{
"version": "12",
"url": "https://oss.example.com/front/main-12.zip",
"md5": "3e5a..."
}
# 3. LiteGate nodes auto sync and switch

发布动作变得很短

你不需要登录每台节点替换静态目录，也不需要额外跑一套前端分发脚本。上传 ZIP，再更新一个 KV 键，发布动作基本就结束了。

LiteGate 接住剩下的事情

节点监听 KV 变化后，会自动拉取新版本、解压到本地版本目录、切换当前服务路径，并按策略保留最近几个历史版本，回滚也更轻。

多节点会更舒服

新节点加入后可以自己收敛到正确前端版本，已有节点也能一起跟上。对小团队来说，这比逐台 rsync 或手工覆盖目录要顺手得多。

Less Ops

不用逐台发静态资源

前端升级从“分发文件”变成“声明版本”，很适合没有专职运维、主要靠开发自己上线的小团队。

Multi-Node

多节点天然更一致

所有节点围绕同一个 KV 键收敛状态，发布不再依赖每台机器都被正确操作过一次。

Rollback Ready

历史版本保留下来

保留最近几个版本目录后，前端回滚不需要重新构建一遍，改回版本指针会更干净。

Discovery Architecture

服务发现是分离的，你不需要为了 LiteGate 重做注册中心

如果你已经有 Consul、Litemesh，甚至是 Nacos、etcd 或多套异构环境，LiteGate 的接法并不是“全换掉”，而是把发现层解耦出来按场景对接。

Built-in providers

适合从简单开始。先用 `static`，后面再平滑升级到 `Consul`、`Litemesh`、`external` 或 `hybrid`。

static Consul Litemesh hybrid

litemesh-registrator-go

适合 Docker / Swarm。给服务打 label，registrator 自动把运行中的容器注册进发现层，LiteGate 直接消费结果。

Docker labels auto register Litemesh

camodns_discovery

适合已有异构注册中心。它把 `Nacos`、`etcd`、`k8s`、`Docker` 等来源桥接成 LiteGate 能消费的统一发现接口，并通过 blocking query 风格把变化按事件感方式推给入口层，而不是让网关自己轮询每一种来源。

Growth Path

不是一上来就做复杂网关，而是先把简单事情做顺

LiteGate 更有说服力的地方，是它允许团队按真实复杂度往前走，而不是一开始就强迫你接受完整平台负担。

Stage 01

网站和 API

先从静态站点、SPA、简单反代和自动 HTTPS 开始，把入口最基础的事情做顺。

Stage 02

动态服务

当实例开始频繁变化，再接入 Consul、Litemesh 或其他发现源，让路由随着服务拓扑演进。

Stage 03

SaaS 入口

当系统进入多租户、集群分流、身份驱动路由、灰度和专属实例阶段，入口层不需要推翻重来。

LiteGate 的价值不只是能处理复杂场景，而是系统长大之后，你不用中途换整套入口体系。

Lite vs Full

简单场景先轻量，复杂场景再升级

Lite 模式是最短上线路径，Full 模式是完整治理模型。两者不是两套运行时，而是同一个内核的不同输入层。

Lite Mode

给简单场景的最短路径 
example.com:
  proxy: 127.0.0.1:8080
  • 适合单站点、SPA、简单反代、重定向和 WebDAV
  • 几行 *.lite.yaml 就能把站点先跑起来
  • 更像面向用户的简洁入口，而不是第二套配置内核

Full Mode

保留完整的治理与扩展能力 
domain: example.com
routes:
  - match:
      path_prefix: /
    action:
      type: proxy
      upstream:
        - 127.0.0.1:8080
  • 适合复杂匹配、认证、中间件、发现和高级入口治理
  • 当系统需要更多表达能力时，直接切回完整模型即可
  • Lite 配置最终会编译成标准模型，迁移路径清晰

Scenarios

这些场景下，LiteGate 会比普通反代更顺手

不是所有项目都需要 LiteGate，但下面这些场景会更容易感受到它的价值。

官网、文档站和管理后台

静态站点和后端 API 放在同一个入口里，减少额外拼装和重复配置。

私有云或 VPS 上的小型产品

单二进制、Docker、自动 HTTPS 和 DNS 证书流程，更符合轻量部署和多域名现实。

轻量微服务入口

当实例开始动态变化，服务发现和标签路由能让入口继续承接复杂度。

多租户 SaaS

身份驱动路由、租户上下文注入和 sid 分流，更接近 SaaS 接入面真实需要的能力。

新域名上线不想再先折腾 DNS

声明站点后，让入口层自动处理 A 记录同步、证书申请和 HTTPS 就绪，比“先配外围再测网站”更省心。

MCP Example

MCP 不是一句“AI 能配”，而是它真的能替你走完一轮入口配置

这件事如果没有具体画面，就很容易显得空。更准确的说法应该是：你把场景说清楚，AI 通过 LiteGate MCP 去读说明、生成配置、保存并验证。

你对 AI 说

给我挂一个官网和一个 /api
域名是 demo.example.com
前端目录是 ./dist
后端在 127.0.0.1:8080
顺便把证书和 HTTPS 路径也检查一下

LiteGate MCP 可以做

  • 读取站点 schema 和配置说明
  • 生成 `*.lite.yaml` 或完整 YAML
  • 保存配置并执行校验
  • 继续检查入口是否已经可访问

用户真正得到的是

不是一句抽象的“AI 辅助”，而是一轮完整的入口配置动作被压缩成自然语言任务，特别适合官网、管理后台、文档站和简单 API 接入场景。
