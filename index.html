它是什么

不只是反代，而是统一入口层

互联网流量从左侧进来，经过中间的 LiteGate 核心引擎做匹配与治理，再到右侧的后端设施。 站点托管、动态代理、自动证书、服务发现和身份入口都收在同一个执行面里。

想看请求在网关内部的完整流转（反转流水线、控制面/数据面分离、零锁数据面与原子热重载），可继续阅读 架构与请求生命周期 →

能力一览

一个网关，十件本来要好几套组件的事

下面每一项都在主链路里原生支持，不靠外围脚本拼装。

站点与内容托管

静态站点、SPA、Markdown 和 WebDAV，一个站点文件就能托管前端资源，不用再单挂一层。

自动 HTTPS 与证书

DNS-01、泛域名、自动续期和 On-Demand TLS，把手工申请、替换和 reload 收回入口层。

自动 DNS（DDNS）

声明站点后自动同步 A 记录，内置阿里云、Cloudflare、华为云、腾讯云，新域名不必先手工指过来。

服务发现

litemesh、litedeploy、consul、docker、discovery 五种发现源，打个 label 即上线；多源混用时自动合并归一。

gRPC 一等公民

一行 proto: grpc：h2c/grpcs 自动二分、mTLS 自动升级、grpc.health.v1 健康检查、gRPC-Web 转码与服务反射，全部原生，无需插件。

集群配置中心

多节点集群里，站点 YAML 和证书统一存进 Litemesh KV：写一次自动同步到所有节点并热重载，证书还由分布式锁单点签发、全集群共享，无需逐台维护配置。

前端版本发布

构建产物打 ZIP 上传 OSS、写一个 KV 键，所有节点自动下载、原子切换并保留历史版本回滚。

MCP 自动驾驶

内建 MCP 服务器、20+ 工具，AI 助手直接建站、改路由、读 schema、热重载和排障。

身份与 SaaS 入口（IDS）

OIDC 登录、JWT(JWKS) 与 API-Key 配额、remote auth、租户上下文注入和 sid 分流，人与机器流量都收进入口层。

安全与可观测

WAF、动态限流、IP 限制、CORS、Litemesh mTLS，以及访问日志、Dashboard 和 Prometheus 指标。

可观测

内建状态看板：写段配置，省掉一个 Uptime Kuma

给路由配上 status 动作，网关就把它正在转发的后端渲染成一个实时状态页。指标全部来自真实转发流量——第一手、白盒、零额外探测开销，还能下钻到动态发现的物理实例，不必再单独部署一套监控组件。

场景

这些场景下，它会比普通反代更顺手

不是所有项目都需要 LiteGate，但下面这些场景会更容易感受到它的价值。

官网、文档站和管理后台

静态站点和后端 API 放在同一个入口里，减少额外拼装和重复配置。

私有云或 VPS 上的小型产品

单二进制、Docker、自动 HTTPS 和 DNS 证书流程，更符合轻量部署和多域名现实。

微服务

服务注册到 Litemesh 后被 LiteGate 自动消费，东西向调用走 mTLS 零信任，实例动态变化时路由自动跟上。

多租户 SaaS

身份驱动路由、租户上下文注入和 sid 分流，更接近 SaaS 接入面真实需要的能力。

Quick Start

先拿到第一个响应，再决定要不要继续

最值得先试的，是用 Lite 模式起一个前后端同域的站点。

# sites/hello.lite.yaml
localhost:
  respond: "Hello, LiteGate!"
$./litegate
$curl http://localhost/
# Hello, LiteGate!
$litegate -t
# config check passed

几行 YAML 先把站点跑起来

只想快速确认 LiteGate 的使用体验，这就是最短的一条路径。前后端同域、自动 HTTPS 都默认就绪。

也可以让 AI 替你配

开启内建 MCP 后，把场景说给 AI，它通过工具读 schema、生成配置、保存并验证入口是否真的可访问。
