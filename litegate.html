LiteGate 不只是代理层，而是统一入口层

这张图最适合说明 LiteGate 的产品位置。它不是单独站在 Traefik、Caddy、Nginx 的某一个点上，而是把网站托管、动态代理、自动证书、服务发现和身份驱动入口收进一个更完整的执行面里。

一个特别值得强调的迁移动机 对很多从 Nginx 过来的团队来说，更实际的价值是把手工申请、续期、替换和 reload 证书这一类重复流程收回到入口层里统一处理。

不是 Caddy 替代品，而是继续往前走一步

LiteGate 继承了站点接入、静态资源和自动 HTTPS 的低门槛体验，但没有把入口层停留在单纯的网站服务器。

不是 Traefik 同级替代，而是把入口面补完整

动态服务发现和路由只是其中一层。LiteGate 继续把站点托管、证书自动化和身份驱动入口收回到同一个执行面里。

对 Nginx 用户，这能把证书流程从手工里解放出来

如果你以前经常忘记先配 DNS，再回头申请证书，这一类发布前置动作现在也可以被 LiteGate 自动收回。

如果你现在还在手工申请、续期、替换和 reload 证书，那么 DNS-01、泛域名和自动续期的价值，就在于把这些重复流程收回到入口层里统一处理。

从一个 label 到直接上线

更完整的故事不是“服务被路由到了”，而是入口意图一旦被声明，服务注册、域名接入、证书申请和 HTTPS 就绪可以沿着同一条链路继续完成。

服务发现是分离接入的

如果你已经有 Consul、Litemesh，甚至是 Nacos、etcd 或多套异构环境，LiteGate 的接法并不是全换掉，而是把 discovery 层解耦出来按场景对接。

MCP 要有具体画面才成立

更准确的说法不是“AI 能配”，而是你把场景说清楚，AI 通过 LiteGate MCP 去读 schema、生成配置、保存变更并验证入口是否真的可访问。

前端发布现在也能被收进这条入口链路里

把前端 ZIP 放到对象存储，更新一个 Litemesh KV 键，LiteGate 会在所有节点自动同步、切换并保留历史版本。

01 Package

前端构建完成后，打成 ZIP 上传到 OSS 或其他可访问对象存储，不再逐台复制静态目录。

02 Commit Version

把 version、url、校验信息写进 KV。对发布动作来说，这一步就像提交了一个前端版本指针。

03 Auto Converge

所有节点会自己收敛到目标版本，适合多节点、小团队、开发直接上线的实际场景。

这条自动化链路，才是 LiteGate 更像完整入口层的地方

它比单纯的动态路由更进一步，因为它关心的是一条完整的上线动作，而不是只接住其中一段。

01 Declare

写一个 *.lite.yaml，或者给容器 / 服务打上 litegate.* 相关 label，先把入口意图表达出来。

02 Discover

服务实例被注册到 Litemesh，或者通过 discovery 桥接层暴露成 LiteGate 能消费的统一发现接口。

03 Go Live

新域名不必再先走一轮手工 DNS 和证书流程。A 记录同步、DNS 验证和 HTTPS 就绪可以继续由入口层收敛处理。

服务发现是解耦的，你可以按场景接入

LiteGate 不要求所有用户都用同一套注册中心。你可以先静态，再动态，也可以用桥接层承接异构发现来源。

Built-in providers

从最简单的静态 upstream 开始，再逐步升级到内建发现源。

static Consul litemesh

litemesh-registrator-go

适合 Docker / Swarm，通过 labels 自动把运行中的服务注册进发现层。

Docker labels auto register litemesh

camodns_discovery

适合异构环境，把 Nacos、etcd、k8s、Docker 等来源桥接成 LiteGate 可消费的统一发现接口，并通过 blocking query 风格把变化按事件感方式推给入口层，而不是让网关自己轮询每一种来源。
