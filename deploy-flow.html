Deploy Flow

LiteGate 不只是帮你第一次上线，也在帮你后续升级

很多方案都能把站点挂起来，但真正麻烦的是之后的扩容、前端发版、后端实例变化、DNS 和证书前置动作。 LiteGate 的价值，不只是做反向代理，而是把上线和升级这两条链路一起收顺。

看初次上线 看后续升级
第一次上线更短 声明站点之后，前端、API、HTTPS 和入口不再分开拼装。
后续扩容更自然 新实例注册后自动进流量，新节点加入后自己收敛。
升级动作更统一 前端切版本，后端加实例，入口层继续自动接住变化。
First Go-Live

同样是把一个 SaaS 功能上线，两条路的心智并不一样

以一个“报销发票”服务为例。传统方式当然能做，但上线前后会散落很多重复动作。LiteGate 想做的是把这些动作收成连续流程。

Traditional

传统组合方式

功能写完之后，通常还要自己把外围动作一个个补齐。

  • 准备域名、VPS、Web Server、反向代理。
  • 先去域名平台处理 DNS，再回来申请 HTTPS 证书。
  • 前端静态资源和后端 API 往往由不同工具承接。
  • 后端扩容后，还要再改 upstream、reload 或手工同步配置。
  • 前端发新版时，容易又回到逐台发包和替换目录。
LiteGate

LiteGate 的方式

先声明“我要上线一个站点”，再让入口层继续接住剩下的动作。

  • 准备域名和机器，把站点入口意图写清楚。
  • / 走前端，/api 走后端，入口层统一承接。
  • DNS 和证书尽量跟着站点声明一起收敛，而不是先去手工前置。
  • 前端可以走 ZIP + OSS + KV 版本同步。
  • 后端可以走服务发现，让实例变化自己被入口层吸收。
LiteGate Flow

一个更像真实交付现场的上线流程

重点不是配置字段本身，而是用户完成的动作会不会更短、更连贯。

01

准备站点入口

拿到域名和机器之后，先把站点 host、前端入口和后端 API 路径声明出来。

02

前端与后端接入

前端可以直接托管静态目录，也可以走 ZIP + OSS；后端从固定 upstream 开始，后面再平滑接服务发现。

03

DNS 与证书收住

把访问入口、DNS 记录同步和 HTTPS 申请尽量收进同一条站点链路里。

04

开始持续升级

之后的前端发版、后端扩容、新节点加入，都沿着同一套入口模型继续演进。

Upgrade Flow

真正让团队舒服的，不是第一次上线，而是之后每一次升级

LiteGate 的差异不只在“先上线”，还在于后面扩容、切版本和回滚时，不用重新拼一遍基础设施动作。

Without LiteGate

服务压力大了之后，扩容和升级动作容易越来越碎。

new backend instance
+ update upstream
+ reload gateway
+ sync static files
+ re-check https / domain
+ re-check new node state
With LiteGate

入口层继续吸收变化，团队更像是在“提交新状态”，而不是“逐台修环境”。

frontend:
+ upload zip to OSS
+ update one KV version key

backend:
+ start new instance
+ register to discovery

LiteGate:
+ auto sync
+ auto route
+ auto converge

前端升级

传统方式下，前端常常要逐台同步目录。LiteGate 这边更像是“上传 ZIP，然后改版本指针”，节点自己完成同步和切换。

后端扩容

服务发现接进来之后，新实例起来并注册，入口层自动感知，不再把高频扩容动作绑定在 reload 上。

新节点加入

新节点不需要人工补很多动作，前端版本会自己追平，后端流量也会通过发现层自动接入。

回滚

前端保留最近几个版本目录后，回滚可以变成改回版本；后端则更接近新旧实例并存再切流的模型。

LiteGate 想解决的不是“能不能代理”，而是“一个服务从第一次上线，到后续扩容和升级，能不能一直保持顺手”。
